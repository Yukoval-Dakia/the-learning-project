## 2. OPEN-Q1 / OPEN-Q2 裁决提案（plan 阶段定，待 owner 批）

### OPEN-Q1 · Copilot SKILL.md per-subject vs 单份共享 → **裁决：单份共享**

**根据**：CopilotTask 的对话方法论（mutation-vs-edge 决策树、suggestion_kind 判据、proposal_feedback 解读、生命周期触发判据、history/ambient 用法）**全部是 cross-subject**——它操作的是知识图谱结构与提案语义，与具体学科无关（不像 note/quiz-gen 那样依赖「文言文例句」「数学公式」的学科内容规范）。讲解/解题节奏的 per-subject 差异由 **TeachingTurnTask 自己的 profile builder** 承载（`buildTeachingTurnPrompt`），不在 Copilot 对话方法论里。

**机制可行性（已实测）**：`populateIsolatedSkills`（runner.ts:277-317）对 `src/subjects/` 下**每个目录**找 `skills/` 子目录镜像，**不校验该目录是否注册过 SubjectProfile**（SubjectRegistry 是 profile.ts 的显式 import 列表，与 dir 扫描解耦）。因此一份共享包可落位 `src/subjects/_shared/skills/copilot/SKILL.md`，零新基建被自动镜像。

**落位**：`src/subjects/_shared/skills/copilot/SKILL.md`。
- 目录名 `copilot`（SKILL.md frontmatter `name: copilot`）跨科唯一，不与 `note-*`/`quiz-gen-*` 命名空间冲突（runner 扁平 mirror 的 collision 风险只在同名时，`copilot` 全仓唯一）。
- resolver `resolveCopilotSkills()` 不接受 subjectId 参数（cross-subject，恒返回 `['copilot']` 或 undefined），与 `resolveNoteSkill(subjectId)` 的签名差异即体现「这是共享包」。
- **降级**：缺 `_shared/skills/copilot/SKILL.md` → 返回 undefined → 不传 skills → 现状散文回退，零差异。

**反对理由（诚实保留）**：`_shared` 伪 subject 目录是新约定，未来若有人误以为它是个 subject 会困惑。缓解：目录名用下划线前缀 `_shared` 明示「非学科」，并在 `resolveCopilotSkills` 顶部注释说明这是 cross-subject 共享包的落位约定 + 指向本 plan。

### OPEN-Q2 · mutation-vs-edge 知识是否抽共享 knowledge-ops skill（与 KnowledgeReviewTask 共用）→ **裁决：本期不抽，单份 copilot SKILL.md + 注释指向后续共享**

**根据**（owner SPEC 已指明可降级）：
- KnowledgeReviewTask 的 mutation 分类知识（registry.ts:531-532）确与 CopilotTask 重叠，但 **KnowledgeReviewTask 走 `mcp__loom__write_proposal` 单工具 + 仍焊在 registry TS 串**——它本身没进 SKILL.md 体系。抽一份 `knowledge-ops` 共享包要求**同时改造 KnowledgeReviewTask 的加载链**（它当前不传 ctx.skills），这是 C2 范围外的第二处重构，成本 = 额外一个 task 的 SKILL.md 接线 + 测试 + 回归面，违反「债务未兑现前不过度重构」。
- 第二真相源风险**本期可接受**：copilot SKILL.md 与 KnowledgeReviewTask TS 串各表述一次 mutation-vs-edge，两者都是「proposal-only 的知识图谱编辑方法论」，但**受众不同**（Copilot 是对话式按用户意图提议；KnowledgeReview 是夜间扫全树批量提议），措辞本就该有差异，不是纯复制。

**落地**：copilot SKILL.md 内 mutation-vs-edge 章节顶部加注释 `<!-- 与 KnowledgeReviewTask(registry.ts:531) 的 mutation 分类知识同源；本期各表一份，后续若第三处需要可抽 _shared/skills/knowledge-ops 共享包，见 YUK-284 plan §2 OPEN-Q2 -->`，并在 Linear 开 follow-up issue 记录该解除条件（见 §9）。

**反对理由（诚实保留）**：两处真相源未来漂移（一处改了另一处忘改）。缓解：注释交叉引用 + follow-up issue 标注「出现第三处 mutation 分类需求时抽共享包」的可检查触发条件。

## 3. C1 / Step 0 — 命名去歧义 + else-if 收敛为查表

### 3.1 目标
- AP-4：`COPILOT_SKILL_KINDS` → `COPILOT_BEHAVIOR_PACK_KINDS`（行为包语义，停止用「skill」指代 TS 服务编排）。

- **纯重构，零行为变化，类型兜底。**

### 3.2 文件级改动

**修改 `src/server/copilot/chat.ts`：**

1. **重命名常量 + 类型**（:105-115）：
   - `COPILOT_SKILL_KINDS` → `COPILOT_BEHAVIOR_PACK_KINDS`
   - `CopilotSkillKind` → `CopilotBehaviorPackKind`
   - `CopilotSkillContext` z.object 内 `skill: z.enum(COPILOT_SKILL_KINDS)` → `z.enum(COPILOT_BEHAVIOR_PACK_KINDS)`
   - ⚠️ **保留 `skill` 字段名本身**（z.object 的 key 是 wire 协定，前端 `skill_context.skill` + replay 都依赖它；改 wire key 会破坏 #348/#354 持久化 replay）。只改**常量/类型标识符名**，不改 JSON 字段名。`CopilotSkillContext` / `CopilotSkillContextT` 类型名是否改：**保留**（它是 wire envelope 的类型，跨 Dock/replay/route 引用面大，改名是 C1 范围蔓延；只改枚举常量名即可达成「停止用 skill 指代 kinds 集合」的目标）。

2. **分支链 → 查表**（:686-820）：当前 `if (skill_context.skill === 'teaching') {…} else if (…'solve') {…} else if (…'quiz') {…} else throw`。
   - 收敛为 `Record<CopilotBehaviorPackKind, () => Promise<CopilotChatResult>>` 查表。每个 handler 是一个闭包（捕获 `req`/`db`/`deps`/`sessionId`/`causedByEventId`/`now`/`streaming` 等已在 scope 的局部），返回 terminal `CopilotChatResult`。
   - teaching handler 因含 `db.transaction` + ask_check 物化 + skillTurn 组装，逻辑最重 → 抽为命名局部 async fn `runTeachingBehaviorPack()`；solve/quiz 同理抽 `runSolveBehaviorPack()`（quiz 已是 `emitQuizReply` 闭包，直接复用）。
   - 派发：`const handler = BEHAVIOR_PACK_HANDLERS[req.skill_context.skill]; return handler();`。
   - ⚠️ **每个 handler 必须在自己内部 `return` 终态 `CopilotChatResult`（含 streaming.onDelta + 完整 return 对象），派发处直接 `return handler()`。NO 块尾共享 return**。现状代码 teaching/solve 分支**不**自己 return、而是穿到 `if (req.skill_context)` 块尾的共享 return（chat.ts:828，读 `replyMd`/`realTaskRunId`/`skillTurn`）——C1 必须把这个共享 return 拆进每个 handler 内部，消灭「块尾读半成品局部」的形态。这是 C3 能干净降级（solve 缺 handler → 不进任何 handler → 落 free-form）的结构前提：若 C1 留着块尾共享 return，C3 的 solve 降级会撞上未赋值局部（见 §5.4 P1 控制流红线）。
   - **消除 `else throw`**：`Record<CopilotBehaviorPackKind, …>` 编译期强制三键齐全（漏一键 TS 报错），加第 4 种 kind 时 z.enum 加值 → Record 立即缺键编译失败，**比运行时 throw 更早、更安全**。保留一行注释说明「穷尽性由 Record 类型保证，无需运行时兜底」。
   - ⚠️ **行为等价性红线**：三个 handler 的内部逻辑（事务边界、reply 事件 payload 每个字段、streaming.onDelta 时机、return 形状、realTaskRunId 来源、skill_turn/skill_context 持久化）**逐字保持**。这是「搬移到闭包」而非「重写」——diff 应是结构性移动，不是语义改动。

3. **更新注释**：:91-104 的 `skill_context` block comment 把「skill」措辞校正为「behavior pack」（保留 R5「skill ≠ surface」的论述，但澄清这里的 kinds 是服务层行为包不是知识层 SKILL.md）。

**连带更新引用点（grep `COPILOT_SKILL_KINDS` / `CopilotSkillKind` 全仓）：**
- 实测引用：仅 `chat.ts` 内部 + 可能的 route/test。C1 必须 grep 确认无遗漏：
  ```

  逐个改名。前端 `CopilotSkillContextT`（类型）不改名故不受影响。

# Copilot 对话方法论

你是唯一面向用户的对话式学习助手。所有 mutation 仅 propose 不直接写（这条在 systemPrompt 已硬约束，此处不重复判据，只给「怎么做」的方法论）。

## knowledge_mutation vs knowledge_edge 怎么选

<!-- 与 KnowledgeReviewTask(registry.ts:531) 的 mutation 分类知识同源；本期各表一份，
     后续若出现第三处 mutation 分类需求可抽 src/subjects/_shared/skills/knowledge-ops 共享包。
     见 docs/superpowers/plans/2026-06-08-yuk284-debt-wave.md §2 OPEN-Q2。 -->

要整理知识树的「形状」时用 `propose_knowledge_mutation`：

| 场景 | mutation |
|---|---|
| 挪到别的 parent 下 | reparent |
| 合并冗余节点 | merge |
| 拆开过粗的节点 | split |
| 归档没用的 | archive |
| 加新子节点 | propose_new |

只是想在两个**已存在**节点间「连一条关系边」时用 `propose_knowledge_edge`（prerequisite / related_to / contrasts_with / applied_in / derived_from 等）。

一句话判据：**动层级 / 增删节点 → mutation；连已有节点的关系 → edge。**

## learning_item 生命周期提案判据

只在用户**明确表达意图**时提议，且都只 propose：

| 用户说 | 提议 |
|---|---|
| 「这个我已经学完 / 掌握了」 | propose_learning_item_completion |
| 「想重新学一遍 / 忘了想再练」 | propose_learning_item_relearn |
| 「现在先放一放 / 以后再说」 | propose_learning_item_defer |
| 「这个不要了 / 删掉 / 归档」 | propose_learning_item_archive |

提议前先用 `get_learning_item_context` 确认目标 item，不要凭名字猜。用户没明确表达生命周期意图时，不要主动提议这四类。

## suggestion_kind 怎么标

每次调 propose_* 工具时设置可选的 `suggestion_kind`：

- **proactive**（默认，可省略）：基于一次成功读取，提议下一步动作。
- **corrective**：**仅**当这条提议是在修正你自己刚在本条消息内观察到的一次失败时才用。

判据红线：读取返回 0 条结果属于**正常成功**（你查了但没找到），**不是失败**——不要因为上游读取为空就把提议标成 corrective。只有真正修复观察到的失败才是 corrective。

## proposal_feedback 怎么解读

输入里若有 proposal_feedback，每条是一个 (kind, relation) 单元，带 top_dismiss_reasons（用户为何 dismiss）和 top_rubric_gates（rubric 为何拒绝）。把它当作**该 relation 的具体失败模式**：提议 knowledge_edge 时避免重蹈这些模式。

纯加性原则（ND-5）：proposal_feedback **绝不**压制信号驱动的提议——它只让你「换个不踩雷的提法」，不让你「因为怕被拒就不提」。为空时按原行为。

## conversation_history 怎么用

输入里若有 conversation_history，它是本次会话最近若干轮的对话记录（每条只有 role 和 text，即用户原话与你的回复正文）。

优先复用其中已有的信息：能从历史直接回答就别再重复调 DomainTool 去读同样的东西（history-preference）。历史里没有的才去查。

## ambient_context 怎么用

输入里若有 ambient_context，它告诉你用户当前所在的页面 route 以及可选的 focused_entity（当前聚焦的实体）。用它把回答**收拢**到用户此刻的上下文——例如用户在某个知识节点页面问「这个怎么学」，focused_entity 就是那个节点。

## 禁止

- 直接写 mutation（永远 propose）。
- 用户没明确表达意图就提议 learning_item 生命周期变更。
- 把 0 结果读取标成 corrective。
- 因 proposal_feedback 而压制本该提的信号驱动提议。
```

> 草稿是 §4.2 切割表「迁SKILL」行（S5/S6/S8/S9/S11/S13）的扩写，措辞与原 systemPrompt 一致以保证行为等价。C2 实施时可微调，但**判据的语义不得变**（行为等价红线）。

- **不能简单缩为 `['teaching']`**：因为 chip quiz 仍 seed `skill_context:{skill:'quiz'}`（wire 保留），z.enum 必须仍接受 `'quiz'`（否则 route 层 `CopilotChatRequest.parse` 拒绝 chip quiz 请求 → 破坏 #348）。solve 已无 seed，但**持久化的旧 solve reply**（若历史有）replay 时 `skill_context.skill==='solve'` 仍需 parse 通过。

- **裁决**：拆成两个常量，语义分层：
  ```typescript
  // wire 层：route 仍可能收到的 skill_context.skill 值（含 service-action quiz + 历史 solve）。
  // 保留三值以兼容 chip quiz wire（#348）+ 旧 solve reply replay。

  // （quiz 早拦截走 service action；solve 已摘出 chat.ts 不再接）。
  export const COPILOT_BEHAVIOR_PACK_KINDS = ['teaching'] as const;
  ```
  - `CopilotSkillContext` z.object 的 `skill` 字段用 `COPILOT_SKILL_CONTEXT_KINDS`（wire 宽，向后兼容）。
  - chat.ts 派发：quiz 早拦截（§5.2）→ solve 不再有分支（删）→ 剩余 `skill_context.skill` 必为 `'teaching'`，进 `BEHAVIOR_PACK_HANDLERS`（现仅 teaching handler）。若收到 `'solve'`（历史/异常）：因无 handler，**显式降级**（log + 回落 free-form，不 throw——R5 单用户自托管下宁可降级不崩）。

  - ⚠️ **P1 控制流红线：降级必须真正退出整个 `if (req.skill_context)` 块，不能只是 `if (!handler)` 不 return**。现状 `if (req.skill_context) { …局部声明 replyEventId/replyMd/realTaskRunId/skillTurn…; 分支; …共享 return（chat.ts:828）}`——teaching 分支**不在自己内部 return**，而是落到块尾共享 return（828），那处 return 读 `replyMd`/`realTaskRunId`/`skillTurn`。若仅写「`if (!handler) { warn; /* 不 return */ }`」，控制流**仍会执行块尾共享 return（828）**，而此时 `replyMd`/`realTaskRunId` **从未赋值** → TS「used before assigned」编译错 + 运行期返回空壳 reply。「落到下方 free-form」这句要求**绕过**块尾共享 return，而非穿过它。

  - **正确改形（二选一，实施时择一，diff 最小者优先）**：
    - **(a) 把整个 `if (req.skill_context)` 收敛为 teaching-only 早返回块**：在块**最前**加 `if (req.skill_context && req.skill_context.skill === 'teaching') { …teaching 全链路…; return {…}; }`（teaching 自己 return，不再有块尾共享 return）。quiz 已在 §5.2 更早早拦截 return；solve/未知 kind **根本不进这个 teaching 块**，自然继续往下走到 free-form——无悬空变量、无穿透 return。这是结构最干净的方案（块内只剩 teaching 一条路径，所有 `replyMd` 等局部都在该路径内必定赋值）。
    - **(b) 保留查表但让降级 `return` 一个「转交 free-form」哨兵**：不可行——free-form 路径在块**之后**才构建 budgetTracker/mcpServer/runInput（chat.ts:854+），无法从块内 return 进去。故只能用 (a) 的「不进块」语义，不能用「进块再 return 出去」。
    ```typescript
    // 方案 (a) 骨架：
    if (req.skill_context?.skill === 'teaching') {
      // …现 teaching 分支全链路（db.transaction + materializeAskCheck + skillTurn 组装）…
      if (streaming) streaming.onDelta(replyMd);
      return { task_run_id: realTaskRunId, reply: replyMd, surface: 'copilot', /* …§828 字段逐字保留… */ };
    }
    // solve / 未知 kind：不进上面的 teaching 块 → 显式降级 log 后继续往下到 free-form CopilotTask。
    if (req.skill_context && req.skill_context.skill !== 'quiz') {
      console.warn('[copilot] no behavior pack handler; falling through to free-form', {
        skill: req.skill_context.skill, task_run_id: taskRunId,
      });
    }
    // …（free-form CopilotTask 路径，chat.ts:854+，逐字不变）…
    ```
    ⚠️ 这把 C1 的「Record 穷尽 + 无 throw」演进为「teaching 早返回 + 其余降级 free-form」——比 throw 更鲁棒（solve 历史请求不崩），且**消除了块尾共享 return 读未赋值局部的悬空风险**。C1 阶段若已把三分支抽成 `BEHAVIOR_PACK_HANDLERS` 查表 + 块尾共享 return，C3 须连这个**块尾共享 return 一并删除**，teaching handler 自带 return（即 C1 的查表方案要预留这个演进，不要把「共享 return」做成无法局部 return 的形态——见 §3.2.2 行为等价红线补注）。
