## 0. Background

AF S4 is the **last** and **XL** slice of the agent-framework sequence. Its job is not to build new infrastructure — it is to make **Copilot the single user-facing conversational agent** (AF §1.1) by absorbing the *turn behaviors* of two other chat surfaces (Active Teaching + SolveTutor) as **Copilot skills** (AF §1.3 — behavior packs, not tool switches), moving the user-facing entry to Copilot, while three protective constraints keep the KPI accounting, the raw `ask_check` INSERT, and the legacy routes intact (AF S4 :527/:529/:531).

The merge target is **three** chat surfaces, not two (AF S4 :518-522, D10 :52):

1. **Active Teaching** — `TeachingTurnTask` single-turn structured JSON (`explain`/`ask_check`/`end` + `structured_question` payload), per-page `TeachingDrawer`, sole entry at `learning-items/[id]/page.tsx:331` 「对话教学」.

3. **Copilot** — `CopilotTask`, the only agentic tool-loop chat; global `CopilotDock` (S0 composer, shipped U3); `type='conversation', entrypoint='copilot'` session.

## 3. Design forks — orchestrator rulings + this plan's adjudications

### 3.1 Session model (OQ1 = C) — orchestrator-fixed

The merged teaching/solve behavior runs **inside Copilot's own `conversation` session** (`entrypoint='copilot'`, `goal_id=null`), carrying the ref **per-turn** via `skill_context` rather than repointing the session goal. **Single-session model (Cross-统合 revised from the original two-session write):** a teaching turn inside Copilot does **not** create or resolve a second teaching `learning_session` — the ask_check question, accept-chip lineage, and active-question state all key off the **Copilot** session id. Legacy teaching writes its own parallel `entrypoint=null, goal_id=learning_item_id` session only when its own `TeachingDrawer` entry is used (R3). **The reuse predicate (`findReusableCopilotConversation`, `conversation.ts:130-157`) gets zero change** — its `entrypoint='copilot'` filter (`:148`) + `goal_id=null` invariant continue to isolate Copilot reuse from teaching sessions, and because teaching-inside-Copilot *is* the Copilot session, there is no cross-session capture risk at all (HIGH risk #3 dodged). Solve's `type='tutor'` state machine is untouched (it is the heaviest, and migration-window parallelism (R3) requires it live anyway; the solve-skill does not touch it — §4.3).

**Why single-session, not two (critic CRITICAL, Cross-统合 confirmed):** the original §4.2 wrote a teaching turn to *two* sessions — the Copilot `entrypoint='copilot'` session (chat envelope) AND a teaching `entrypoint=null` session (ask_check + accept-chip). That is unbuildable as specced and fatal at runtime:
> 1. `conversation.ts` has **no find-or-create for a teaching session** — `startConversation` (`:59-92`) always INSERTs a new row; `findReusableCopilotConversation` (`:130-157`) *excludes* teaching sessions by the `entrypoint='copilot'` filter (`:148`, with an explicit comment :144-147 that this is to keep teaching's `entrypoint=null` rows out). So the skill would have to **create** a fresh teaching session every teaching turn.
> 2. That teaching session receives **no `actor_kind='user'` events** (the user's ask is written to the Copilot session as `experimental:copilot_user_ask`, `chat.ts:239`). The idle clock (`promote_conversation_idle.ts:29-47`) is `COALESCE(last_user.at, started_at) < cutoff` and **does not filter `entrypoint`** — so the userless teaching session idles after 5 min (`IDLE_MS`, `:21`).
> 3. `accept-chip/route.ts:135` calls `Conversation.assertActive`, which (`conversation.ts:436-458`) throws 409 on any `status !== 'active'`. An idled teaching session → accept-chip 409 → **corrective-chip KPI silently lost**.
> The single-session model eliminates all three: the user's `copilot_user_ask` keeps the *one* session's idle clock fresh, `assertActive` holds, and no userless session is ever created.

### 3.2 Task kind (OQ2 = B2) — orchestrator-fixed

`TeachingTurnTask` is **retained as a narrow internal task**. The teaching-skill **composes** a call to it at the **service layer** (preserving the single structured `ask_check`/`explain`/`end` contract + `structured_question` carrier + the solve hint reuse), and renders the structured output as a Copilot reply + chips. It does **not** enter the `CopilotTask` LLM tool loop and does **not** enter the tool surface (R2/R5). This dodges CRIT risk #2 (the task-kind-convergence trap: `needsToolCall:false/maxIterations:1` structured JSON has no natural home in the `maxIterations:6` free-form tool loop).

### 3.3 Skill discrimination (OQ3 = explicit minimal ref) — orchestrator-fixed

No full S2b `CurrentUserContext`. `CopilotChatRequest` gains optional `skill_context: { skill: 'teaching'|'solve', ref: { kind, id } }`. The entry button opens the Dock with this parameter. **This is the forward-compatible seed of S2b's `active_ref`** (the future `CurrentUserContext.active_ref` will subsume it) — documented inline with that note. This plan **names the field** `skill_context` (orchestrator left naming to the planner); the `ref` shape `{ kind, id }` matches the existing `leave_agent_note` ref shape (`AF §4` `refs: Array<{kind, id}>`) for vocabulary consistency.

### 3.4 UI absorption (OQ4 = entry re-point) — orchestrator-fixed

`learning-items/[id]/page.tsx:331` button re-points to open `CopilotDock` + `teaching` `skill_context` (learning_item id as ref). `TeachingDrawer` + its routes stay in parallel (R3, untouched this phase). No drawer-in-drawer component merge.

### 3.5 Solve range (OQ6 = backend-only) — orchestrator-fixed (Cross-统合 revised)

solve-skill provides hint capability (**question face + reference solution** as context; **prior-attempt summary is OUT — see OQ12 below**, the current `planSolveHint` does not seed it). **It does NOT reuse `planSolveHint`** — that fn is tutor-session-bound (`solve.ts:152-171`: it takes a `sessionId`, calls `Tutor.getTutorQuestionId`, and gates on `status==='active'`), so reusing it would require a live tutor session inside the Copilot path (an R3/R7 violation and a lifecycle trap). Instead the solve-skill calls `TeachingTurnTask` **directly** via a shared seed helper extracted from the *pure* seed body `solve.ts:172-209` (question load → subjectProfile → TeachingTurnTask input → `runTaskFn`), which is **independent of the tutor session**. No `SolveTutorPanel`, no `/questions` UI, no tutor state-machine change. Orphan panel left for cut-over.

### 3.6 Budget (OQ5 = untouched) — orchestrator-fixed

`COPILOT_CONTEXT_BUDGET` unchanged; the teaching-skill's `TeachingTurnTask` call is a **service-layer call, not a tool call**, so it does not consume the tool-call budget. **Lightweight-cap flag for critic (§10):** the teaching-skill composes a *second* model call per Copilot turn (the `TeachingTurnTask` invocation) on top of the copilot turn's own budget. This is a *cost/latency* concern, not a tool-budget concern — see §4.4 for the proposed soft guard.

## 10. Weakest two spots (for critic focus) — both RESOLVED by Cross-统合

1. **[RESOLVED → re-architected] The two-session write was the load-bearing structural risk, and the critic + Cross-统合 confirmed it is unbuildable + fatal.** Original concern: can a teaching session be cleanly resolved from inside a Copilot turn? **Answer: no — `conversation.ts` has no teaching-session find-or-create (`startConversation:59-92` always INSERTs; `findReusableCopilotConversation:130-157` excludes teaching by the `entrypoint='copilot'` filter), and a lazily-created userless teaching session idles in 5 min (`promote_conversation_idle.ts:29-47`, no entrypoint filter) → accept-chip `assertActive` (`conversation.ts:436-458`, active-only) 409s → corrective KPI lost.** **Resolution**: the plan is revised to a **single-session** model (§3.1/§4.2) — the teaching turn lives entirely on the Copilot session, whose idle clock the user's `copilot_user_ask` keeps fresh, so `assertActive` holds and no userless session exists. The remaining residual risk is now narrow and named: the `copilot_reply` payload **must** carry `turn_kind` for the corrective-chip resolver to anchor (§4.2 — covered by the §7 resolver assertion).

2. **[RESOLVED] The `materializeAskCheckQuestion` extraction is a clean just-the-INSERT extraction — Cross-统合 verified the full transaction body.** The INSERT `values({...})` runs `turn/route.ts:99-128`; the corrective-failure-counting (`getActiveQuestionState`, `:172`) is a **post-commit read OUTSIDE the transaction**, so the INSERT is cleanly separable — **no failure-counting is dragged in**. The shared fn is exactly the `question` INSERT + its `learning_item.knowledge_ids` lookup (`:102-128`), parameterized by `sessionId` (Copilot session id when called from the skill). OQ7's "service fn" is the clean extraction, not a risky transaction refactor.

**New weakest spot after revision (for the implement/review lanes):** the **accept-chip resolver extension + `copilot_reply` `turn_kind` payload** (§4.2) is the one place the single-session model adds new coupling. If the impl lane widens the resolver's `action` filter but forgets to write `turn_kind` onto the `copilot_reply` payload (or vice-versa), corrective-chip anchoring silently falls back to the wrong agent event. The §7 resolver+payload assertion + the §7 R3 legacy regression (proving `teach_message` anchoring still works) together fence this — both must be present before merge.
