```markdown
# ADR-0015: `learning_record` + `memory_brief_note` 为一等领域实体

**Status:** Accepted
**Date:** 2026-05-22
**Supersedes:** —
**Superseded by:** —
**Related:** ADR-0005（single-owner write paths）/ ADR-0006 v2（event substrate）/ ADR-0008（learning_session）/ Plan 2026-05-18 `learning-data-loop`

## Context

2026-05-18 `learning-data-loop` plan 引入了两张持久化表：

- `learning_record`（`src/db/schema.ts:L210`，migration `drizzle/0007_learning_record_loop.sql`）：用户可见的活动上下文标记，例如 "做了第 N 题"、"读了 atomic note X"，由前端 / API 调用直接产生。
- `memory_brief_note`（`src/db/schema.ts:L244`，同 migration）：Dreaming agent 维护的学习记忆摘要 — 跨多个 `learning_record` / `event` 聚合出的"用户最近在学什么 / 卡在哪"短文。

`event`（ADR-0006 v2）覆盖不可变 action log；`learning_session`（ADR-0008）覆盖会话信封。这两张新表与 event/session 并列但未有 ADR 追认设计决策。

未来新人 / Agent 读 ADR 时无法区分：
- 为什么不直接用 `event` 表达 "做了第 N 题"？
- 为什么不复用 `learning_session` 作为 activity context？
- `memory_brief_note` 与 Dreaming 的关系归谁所有？

## Decision

追认 `learning_record` + `memory_brief_note` 为一等领域实体，并明文以下约束：

### 1. `learning_record` 语义边界

- **定位**：用户可见的"活动 tag" — 描述一次具体的学习触点（做题 / 阅读 note / Active Teaching 一个 turn 完成），有 user-facing 含义。
- **与 `event` 的关系**：`event` 是不可变 action log（系统视角，所有持久化都走 event）；`learning_record` 是从 event 派生的 user-visible 视图（用户视角，"我今天做了什么"）。两者一对多：一个 learning_record 通常对应 1+ 个 event（attempt + judge + grade）。
- **写路径**：`src/server/records/queries.ts:L74`（insert）/ `L149`（update）— 单一所有者（ADR-0005 对齐）。其他模块只读，不直接插。
- **不变量**：`learning_record.event_id` 必填，指向产生该 record 的 leading event；其他相关 event 通过 `event.session_id` / `event.subject_id` 关联。

### 2. `memory_brief_note` 语义边界

- **定位**：Dreaming agent 维护的"用户学习状态摘要" — ≤200 字的短文，描述用户最近的学习焦点 / 卡点 / 进度。供 ReviewIntentTask / SessionSummaryTask / TeachingTurnTask 等 task 读取作为上下文。
- **与 `event` 的关系**：纯派生物。memory_brief_note 由 Dreaming cron 周期性扫 event 表 + learning_record 表后写入，**不接收 user input**。任何用户操作只产 event；event 触发 Dreaming 重算 brief。
- **写路径**：Dreaming agent 独占（`src/server/dreaming/*.ts`，TBD — Phase 2C 落地后定位）。其他模块只读。
- **不变量**：每个 (`subject_id`, `domain`) 至多 1 条最新 brief；旧 brief 通过 `superseded_at` 软删。

### 3. 单一所有者原则（ADR-0005 对齐）

- `learning_record`：`src/server/records/queries.ts` 拥有所有 insert / update / soft-delete 路径。
- `memory_brief_note`：Dreaming agent 模块拥有所有写路径。
- 违反单一所有者：禁止任何 route handler / orchestrator 绕过这两个模块直接写表（schema 层应在未来 phase 加 RLS 或 trigger 强制）。

### 4. 与 `learning_session` 的关系

- `learning_session` 是会话信封（ADR-0008）：一个时间窗 + 1+ learning_record（顺序排列）。
- `learning_session` 不直接持有题 / note 内容，只引用 `learning_record.id`。
- `memory_brief_note` 不挂在 `learning_session` 下 — brief 是跨 session 聚合，session 是单次窗口。

## Consequences

- ✅ 新 ADR 提供 `learning_record` / `memory_brief_note` 设计 rationale 文档化锚点，对齐 ADR-0005 单一所有者原则
- ✅ Future Agent / 新人读 ADR 序列即可理解四张主表（event / learning_session / learning_record / memory_brief_note）的边界
- ⚠️ `memory_brief_note` 写路径仍 TBD（Phase 2C Dreaming agent 落地后定位）— 本 ADR 标"Dreaming agent 独占"是 forward-locking decision；如果 Dreaming 设计改变，本 ADR 需 revise
- ⚠️ 现有 `src/server/records/queries.ts` 没有显式 schema-level enforcement（暂靠代码 review）；未来可在 ADR-0015 follow-up 评估 RLS / trigger

## Notes

- 本 ADR 是对 Plan 2026-05-18 已落地决策的追认（drift audit 2026-05-20 §[Plan 2026-05-18] 推动）。
- `event` 仍是 substrate（ADR-0006 v2）— `learning_record` / `memory_brief_note` 是其上层 view，不是替代。
- 单一所有者细节执行交给后续 ADR / phase（Phase 2C Dreaming 落地 / Phase N+1 schema-level enforcement）。
```
