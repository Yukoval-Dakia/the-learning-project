## Fixes — minimal changes only

### Fix A: `src/server/proposals/signals.ts` — lock holds full rebuild (P1-4 + P1-5)

- Wrap `ensureProposalDecisionSignal` body in `db.transaction(async (tx) => …)` when caller passed a plain `Db` handle. Take advisory lock + run rebuild inside the same tx so the lock holds through the upsert.
- Mirror the lock in `recordProposalDecisionSignal` so incremental writes also serialize on `(kind, cooldown_key)`. Same `lockKey` derivation (`${kind}:${cooldownKey}` → `hashtextextended`). Wrap in tx when caller passed `Db`.
- Extract a small helper `withProposalSignalLock(db, kind, cooldownKey, fn)` that resolves the txn nesting (if `db` is already a `Tx`, reuse it; otherwise open a new txn).
- This serializes rebuild and incremental updates on the same key so neither can clobber the other.

### Fix B: `app/api/review/due/route.ts` — bounded + correction-aware failure lookup (P2-8 + P2-9)

- `loadLatestFailureQuestionIds(candidateLimit)` already uses a window function. Widen the partition slice from `rn = 1` to `rn <= per_question_limit` (use the same `4` cap) and apply the active-correction filter inside SQL by `LEFT JOIN` against the same correction lookup the queries layer uses. **Lighter alternative** that avoids reshaping the queries layer: keep `loadLatestFailureQuestionIds` as `rn = 1` but switch it to pick **any active question** by also reading more rows (rn ≤ 4) and joining against `getCorrectionStatuses`. The simplest minimal fix is: pull the candidate set with `rn ≤ per_question_limit` and dedupe in TS — keeps the round-trips identical.
- `getFailureAttemptsPerQuestion` should pass `limit: questionIds.length * perQuestionLimit` (not `null`) to `getFailureAttempts`, so the inner scan stays bounded. Combined with the deterministic SQL order below, the per-question cap remains stable.

### Fix C: `src/server/events/queries.ts` — deterministic failure ordering (P2-10)

- Add `desc(event.id)` as secondary key to the SQL in `getFailureAttempts` so same-timestamp ties resolve deterministically. This is a pure ordering change; no semantics shift.
