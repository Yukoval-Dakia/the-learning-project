## 2. seam 基建详细设计（Commit A）

### 2.1 `buildQueryOptions` 条件注入（零回归核心）

`runner.ts:357-391` return 的 Options 对象，**条件性**追加 `outputFormat`。当前是直接 `return { … }` 字面量（末字段 `skills: ctx.skills ?? []`，:390），需改为先建 `const options`、条件 mutate、再 return：

```ts
function buildQueryOptions(kind, ctx, abortController): Options {
  const def = tasks[kind];
  const resolved = resolveTaskProvider(kind, ctx.override);
  const allowedTools = ctx.allowedTools ?? def.allowedTools;
  const options: Options = {
    model: resolved.model,
    /* …existing 字段完全不动，含 skills: ctx.skills ?? []… */
  };
  // YUK-299 seam: outputFormat 透传。默认 undefined → 不写 key →
  // Options 对象与 seam 前字节级一致（约束①）。仅当 handler 显式给定
  // ctx.outputFormat 时才注入，已迁移的 task 才走 structured-output 路径。
  if (ctx.outputFormat !== undefined) {
    options.outputFormat = ctx.outputFormat;
  }
  return options;
}
```

**零回归保证**：`ctx.outputFormat` 默认 `undefined`（`RunTaskCtx.outputFormat?` optional，无 default 赋值）。所有未迁移 task（通用对话 / teaching / quiz / judges / 全部 stream）从不设置它 → `if` 不进入 → 返回的 Options **键集与值与 seam 前完全相同**。`OutputFormat` 类型 import from `@anthropic-ai/claude-agent-sdk`。

> **改动风险提示**：从「return 字面量」改为「const options + 条件 mutate」是纯重构（同字段同值）；§5.2 的「键集逐键相等」快照断言专门兜住这步可能引入的字段漂移。

### 2.2 `runTask` 读取 structured_output（三态 — 约束④）

`runner.ts:443-451` success 分支内新增读取（**FACTUAL D 修：不 cast**，`msg` 已被 `msg.subtype === 'success'` 收窄为 `SDKResultSuccess`，与既有 `msg.result`/`msg.usage` un-cast 读取一致）：

```ts
if (msg.subtype === 'success') {
  resultText = msg.result ?? '';
  // YUK-299: 端点支持且 SDK 填充时有值；不支持 / 未启用 outputFormat 时为 undefined。
  // msg 已被 union 收窄为 SDKResultSuccess（sdk.d.ts:3579 structured_output?: unknown），无需 cast。
  structuredOutput = msg.structured_output;
  /* …usage / cost_usd / stopReason 不动… */
}
```

挂到 result（`runner.ts:501-507`）：

```ts
const result: RunTaskResult = {
  task_run_id: taskRunId,
  text: resultText,
  finishReason: stopReason,
  usage,
  cost_usd,
  structured_output: structuredOutput, // YUK-299: undefined 时对未迁移 caller 透明
};
```

**三态在 runner 层的落点**：
1. **成功有值**：端点支持 + outputFormat 启用 + 模型产出合 schema → `structured_output` 非 undefined，挂 result，handler 取用。
2. **`error_max_structured_output_retries`**：SDK 自动重试耗尽 → `msg.subtype` ≠ `'success'` → 落 `runner.ts:452` 既有 `else` → `throw`。**可观测增强**：在 throw 前对该 subtype 单独 `console.warn`（不改 throw 行为，仅区分监控）：
   ```ts
   } else {
     if (msg.subtype === 'error_max_structured_output_retries') {
       console.warn(`[${kind}] structured-output retries exhausted`, { task_run_id: taskRunId });
     }
     const apiStatus = /* …既有… */;
     throw new Error(`[${kind}] Agent SDK errored: subtype=${msg.subtype}${apiStatus}`);
   }
   ```
   既有失败留痕（`writeAiTaskRunFinished` status='failure'，runner.ts:464）**不变**——约束②满足。
3. **端点不支持回退**：mimo 端点若不识别 `outputFormat`，SDK 仍返回 `subtype:'success'` 但 `structured_output` 为 `undefined`（端点忽略该参数照常返回文本）→ result.structured_output=undefined → **handler 走文本 fallback**（§4）。runner 只透传 undefined。

> **四落库点约束②核对**：以上改动**只**新增「读 result 字段 + 挂 result 对象 + 一行 warn」。`writeAiTaskRunStarted` / `writeAiTaskRunFinished` / `writeCostLedger` 的调用参数**一字未改**。`structured_output` 数据**不**新写库——审计 :142 要求四点语义不动，故 seam 阶段**不**把 structured_output 落库；若后续要审计留痕，走独立 issue（不在本计划）。

### 2.3 `streamTask` 显式不动（范围纪律）

`streamTask`（runner.ts:556）是文本流 Response，无 `RunTaskResult` 返回值可挂 structured_output，且 stream 语义与 json_schema 一次性产出冲突。**本计划不给 streamTask 接 outputFormat**。`buildQueryOptions` 被 streamTask 共用，但因 stream caller 从不设 `ctx.outputFormat`，§2.1 的 `if` 不进入 → stream 路径 Options 字节不变。注释标注 streamTask 是 seam 的非目标（phase-deferred，留 YUK 跟踪）。

### 2.4 `RunTaskCtx` / `RunTaskResult` 类型扩展（runner.ts）

`RunTaskCtx`（runner.ts:79-121）末尾加（带注释，遵循 phase-deferred 注释规范）：

```ts
/**
 * YUK-299 seam: Agent SDK `outputFormat` 透传。OMITTED（默认）⇒ buildQueryOptions
 * 不写该 key ⇒ Options 字节级与 seam 前一致（零回归）。仅已迁移到 structured-output
 * 的 handler 显式设置（值由 zodToJsonSchemaOutputFormat() 产出）。streamTask 不读它。
 */
outputFormat?: OutputFormat;
```

`RunTaskResult`（runner.ts:56-64）加：

```ts
/**
 * YUK-299 seam: SDK 在 outputFormat 生效且端点支持时填充的结构化产物。
 * undefined ⇒ 未启用 / 端点不支持 / 模型回退文本 —— caller 须走文本 fallback 解析。
 */
structured_output?: unknown;
```

### 2.5 `TaskTextResult` 同步扩展（provenance.ts — VETO-1 修，**load-bearing**）

**问题（critic VETO-1）**：pilot 注入的 fn 是 `RunTaskFn = TaskTextRunFn`（variant_verify.ts:34），其返回类型是 `TaskTextResult`（provenance.ts:1-5），**当前无 `structured_output`**。pilot 调用点（variant_verify.ts:220）拿到的 `result` 静态类型是 `TaskTextResult`，§4 的 `result.structured_output` 会编译报错 `Property 'structured_output' does not exist on type 'TaskTextResult'`，`pnpm typecheck` gate 直接挂。

**修**：把 `structured_output?: unknown` 加到 `TaskTextResult`（provenance.ts:1-5）：

```ts
export interface TaskTextResult {
  text: string;
  task_run_id?: string;
  cost_usd?: number;
  /** YUK-299 seam: SDK structured-output 透传。runTask 的完整 RunTaskResult 结构
   *  可赋值给本接口（RunTaskResult 是 TaskTextResult 的超集）；handler 注入的
   *  TaskTextRunFn 据此可读 result.structured_output 走三态分发（约束④）。 */
  structured_output?: unknown;
}
```

**为何这一处足够**：`defaultRunTaskFn`（variant_verify.ts:63-71）已经 `return result`（完整 `RunTaskResult`），而 `RunTaskResult`（加了 §2.4 字段后）**结构上是 `TaskTextResult` 的超集** —— 多出来的 `finishReason`/`usage` 不影响向 `TaskTextResult` 的赋值。所以只需 widen `TaskTextResult` 携带 `structured_output`，`defaultRunTaskFn` 的返回值即自动透传该字段，**无需改 `defaultRunTaskFn` 实现**，也无需把 pilot 的注入 fn 类型改成完整 `RunTaskResult`。这是最小改动面。

> 三处类型互相印证：`RunTaskResult`（runner.ts，runTask 实产）⊇ `TaskTextResult`（provenance.ts，handler 注入契约）。两边都加 `structured_output?: unknown`，赋值兼容、读取合法、typecheck 绿。

## 3. 核心决策：Zod→JSON Schema 怎么做（zod_conversion_decision）

**决策：引 `zod-to-json-schema@^3.25.2` 作直接依赖；薄封装在 `src/server/ai/output-format.ts`（含方言收敛），不手写转换器。**

**理由（四维）**：

1. **兼容性已验证、零新增解析风险**：项目 zod `^3.25.0`（resolved 3.25.76）；`zod-to-json-schema@3.25.2` peerDep `zod: ^3.25.28 || ^4`，3.25.76 满足。该版本**已在 `pnpm-lock.yaml:4625` 定义、被 `@mistralai/mistralai@1.15.1`（:5611）与 `@modelcontextprotocol/sdk@1.29.0`（:5634）作间接依赖解析**（**FACTUAL C 订正：不是 drizzle-zod**）。`pnpm add` 只是把它从间接提升为直接，**lock 不产生新版本节点**——无 `drizzle-zod 0.8.x 类型破裂` 那类升级风险（owner dep 约束记忆点不适用此包）。

2. **长尾需求 > 试点形状**：试点 schema 浅（VariantVerify 1 层 object + array<string>）。但 seam 是**通用基建**——后续要迁移的 judges（StepsLlmOutput / SemanticJudgeOutput / MultimodalDirectLlmOutput）、SolutionGenerateOutput 含 nested object / union / refine / optional 链，手写转换器会持续追加分支，沦为半成品 OSS 重造。owner 哲学「成熟问题用 OSS，不自建」（CLAUDE.md design principles）直接命中。

### 3.4 封装层方言收敛契约（SHOULD-FIX B —— 基于真跑输出，**load-bearing**）

**实测**（本轮真跑 `zodToJsonSchema(VariantVerificationResult)`，两种调用形态）：

- **命名形态** `zodToJsonSchema(schema, 'Name')` → 产 `{ "$ref": "#/definitions/Name", "definitions": { "Name": {…} }, "$schema": "http://json-schema.org/draft-07/schema#" }`。**含 `$ref` + `definitions` 包裹**——不能直接塞进 `JsonSchemaOutputFormat.schema`（那应是 schema 本体而非 $ref 信封）。
- **inline 形态** `zodToJsonSchema(schema, { $refStrategy: 'none' })`（无 name）→ 产**扁平 schema 本体**（无 `$ref`/`definitions`），但**仍含** `"additionalProperties": false` 与 `"$schema": "http://json-schema.org/draft-07/schema#"`。

**`.default([])` 的实测行为**（试点 schema 唯一的非平凡特性，原 plan 漏测）：`failure_reasons: z.array(...).max(10).default([])` 产出
```json
"failure_reasons": { "type":"array", "items":{...,"minLength":1,"maxLength":500}, "maxItems":10, "default": [] }
```
**且 `failure_reasons` 从 `required` 数组中被移除**（实测 `required: ["verdict","cause_targeting","summary_md","confidence"]`，无 `failure_reasons`）。

**封装层契约（据此定）**：

`zodToJsonSchemaOutputFormat(schema)` 内部用 **inline 形态**（`$refStrategy:'none'`，不传 name）以避免 `$ref`，然后**显式剥离 `$schema` 键**（mimo 端点不需要 meta-schema 指针，留着是无意义噪声/潜在方言风险），并**保留 `additionalProperties:false`**（收紧约束，对结构化抽取有益；但若 pilot 实测 mimo 拒绝该键，则在封装层一并剥离——封装层是唯一改点）。最终：

```ts
export function zodToJsonSchemaOutputFormat(schema: ZodTypeAny): JsonSchemaOutputFormat {
  // inline（$refStrategy:'none'）→ 无 $ref；浅 schema 无嵌套引用。
  const raw = zodToJsonSchema(schema, { $refStrategy: 'none' }) as Record<string, unknown>;
  // 方言收敛（收敛到此一处）：剥 $schema meta 指针（mimo 不需要）。
  // additionalProperties / default / required-omission 是 SDK→端点的语义，保留。
  const { $schema: _drop, ...clean } = raw;
  return { type: 'json_schema', schema: clean };
}
```

**`.default()` / `required`-省略 的语义说明**：`failure_reasons` 不在 `required` 是**正确**的——它在 Zod 侧有 `.default([])`，模型可不产出该字段，SDK/端点据 schema 知道它 optional。**runner 不关心**（runner 只透传 schema）；pilot 的 Zod 二次校验（§4）会用同一个 `VariantVerificationResult.safeParse` 把 `.default([])` 补齐——所以即便端点回的 structured_output 缺 `failure_reasons`，`safeParse` 也会填 `[]`，行为与文本 fallback 路径一致。

> 单测据此固定上述全部输出形状（§5.1），把「封装层产什么」钉成契约，OSS 升级 / 方言调整都能被单测捕获。

```ts
export function parseVariantVerifyResult(result: TaskTextResult): VariantVerificationResultT {
  // 三态：
  // (A) structured_output 有值（端点支持 + 模型合 schema）→ 直接 Zod 二次校验业务约束
  // (B) structured_output 缺失（端点不支持 / 回退文本）→ char-scan fallback（保留旧防御层）
  if (result.structured_output !== undefined) {
    const parsed = VariantVerificationResult.safeParse(result.structured_output);
    if (!parsed.success) {
      // 形状对了但业务约束（max length 500 / maxItems 10 / enum）不过 → 同 fallback 一样 throw，
      // pg-boss 重投。Zod 二次校验不可省（约束⑥）：outputFormat 只保证 JSON 形状，
      // 不保证 failure_reasons[].length<=500 / maxItems<=10 等 app 语义。
      throw new Error(
        `parseVariantVerifyResult: structured_output schema invalid: ${parsed.error.issues.map((i) => i.message).join('; ')}`,
      );
    }
    return parsed.data; // safeParse 在此补齐 failure_reasons 的 .default([])
  }
  // fallback：保留既有 char-scan + safeParse（约束③ 防御层不删）
  return parseVariantVerifyOutput(result.text);
}
```

**保留项明确**：
- `parseVariantVerifyOutput`（char-scan，variant_verify.ts:73-92）**不删**——作 (B) 态 fallback。
- `VariantVerificationResult.safeParse` 二次校验在 (A) 态**仍执行**——outputFormat 给形状，Zod 校业务约束（约束⑥）。
- 下游 `writeEvent`（variant_verify.ts:244）输入**不变**——`parsed.verdict` / `parsed.failure_reasons` 字段同源，约束②（event 落库语义）满足。`runVariantVerify`（:221）现在的 `parseVariantVerifyOutput(result.text)` 调用改为 `parseVariantVerifyResult(result)`，下游全不动。

**`error_max_structured_output_retries` 在 pilot 的体现**：该 subtype 在 runner 层已 throw（§2.2 态2），`runTaskFn` 抛出 → variant_verify handler 的既有 try/catch（:294-304）+ pg-boss retry 接住，**无需 pilot 额外处理**。pilot 只处理 runner 成功返回后的 (A)/(B) 两态。
