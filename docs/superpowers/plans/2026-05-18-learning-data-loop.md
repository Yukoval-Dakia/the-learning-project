# Learning Data Loop Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Build the durable data loop where stored questions have lifecycle evidence, every confirmed answer becomes an attempt event, mistakes and selected non-mistake context become `learning_record`, Dreaming maintains a brief learner-memory note, and agents can read that context through bounded server readers.

**Architecture:** Keep `question` as the canonical material table and `event` as the immutable action log. Add `learning_record` as the user-visible activity-grounded context table, then derive question lifecycle summaries from `question + event + material_fsrs_state + learning_record` rather than duplicating counters on `question`. Add `memory_brief_note` as a derived Dreaming snapshot with three prose windows: recent week, recent months, and long-term memory.

**Tech Stack:** Next.js App Router route handlers, Postgres + Drizzle, Zod/drizzle-zod, Vitest + Testcontainers, ts-fsrs, TanStack Query frontend calls, existing `writeEvent()` and `upsertFsrsState()` owners.

---

## Scope

This plan implements the backend data loop plus minimal UI wiring needed to stop using `study_log` and stop treating correct homework answers as ignored noise.

Included:

- Replace `study_log` with `learning_record`.
- Keep `event(action='attempt')` as the answer signal for success, partial, and failure.
- Create `learning_record(kind='mistake')` for failure attempts.
- Create optional `learning_record(kind='worked_example' | 'reflection' | 'observation' | ...)` for reusable context.
- Add question lifecycle reader API for `recorded_at`, attempts, reviews, FSRS state, linked records, variants, and source assets.
- Add Dreaming-maintained `memory_brief_note` with evidence ids and a read-only API/tool surface.
- Update manual mistake and ingestion import write paths.
- Update `/record`, remove `/study-log` primary surface, and keep old routes returning `410 Gone`.

Not included:

- A full MCP server.
- A full vector/profile memory layer beyond the brief note.
- Replacing all quiz/JudgeRouter roadmap code.
- Polished timeline UI. This plan only ensures the data can support it.

Behavior:

1. Collect evidence for `scope_key='global'` first.
2. If there is no evidence in any window, return `{ status: 'skipped' }` and do not call the model.
3. Call `MemoryBriefRefreshTask`.
4. Validate output evidence ids against collected ids.
5. Write `event(action='experimental:memory_brief_refresh', actor_ref='dreaming', subject_kind='memory_brief')`.
6. Upsert `memory_brief_note.source_event_id` to the refresh event id.
7. On model or validation failure, write a failure event with `payload.failure_reason`, keep the old note unchanged, and return `{ status: 'failed' }`.
