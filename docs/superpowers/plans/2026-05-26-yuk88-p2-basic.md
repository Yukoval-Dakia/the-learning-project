# YUK-88 P2-basic — TipTap Editor + BlockTreeRenderer

> Phase-level launch plan for the P2-basic sub-wave of [YUK-92](https://linear.app/yukoval-studios/issue/YUK-92). This is the must-ship editor path before P3/P4 can rely on real `body_blocks`.

**Date**: 2026-05-27  
**Depends on**: YUK-88 P1 complete and ADR-0020 updated with P0 split-id-preserve rule.  
**Spec source**:
- `docs/planning/2026-05-26-note-rich-doc.md` §0.2 / §0.5 P2
- `docs/adr/0020-block-tree-note-rebuild.md` §11
- Official TipTap docs for custom nodes, React NodeViews, and Static Renderer

## Phase goal

Ship the basic editable block-tree note experience without polish UI: TipTap + StarterKit, custom block nodes, SSR-only read renderer, paste markdown, undo/redo, inline marks, and ADR-0021 draft.

## Lane split

### Lane A — Editor shell + lazy bundle + SSR renderer boundary

**Scope**:
- Add TipTap dependencies in root package (`@tiptap/react`, `@tiptap/starter-kit`, `@tiptap/core`, and static renderer package if chosen by implementation).
- Create editor-only client component boundary for artifact edit mode.
- Create `<BlockTreeRenderer>` that renders JSON without loading the editor bundle.
- Add fixture-based tests for rendering a minimal PM doc.

**Acceptance**:
- Viewing a note read page does not import the editor client bundle.
- Editor shell can load a fixture `body_blocks` doc and persist `editor.getJSON()` shape through a save callback.
- `CODEX_FULL_GATE=1 pnpm typecheck` passes after dependency and boundary changes.

### Lane B — Custom nodes + minimal NodeViews

**Depends on**: Lane A  
**Scope**:
- Implement custom node extensions for `SemanticBlock`, `CrossLinkBlock`, `ArtifactRefBlock`, `CalloutBlock`, and `AutoLinksContainer`.
- Use React NodeViews only where in-editor interaction is needed; keep read renderer mappings separate.
- Add split/merge commands or transaction wrappers that preserve the left block id and mint a new right block id.

**Acceptance**:
- Fixture covers `definition`, `mechanism`, `example`, `pitfall`, `check`, `cross_link`, and `artifact_ref`.
- Split keeps the original block id on the left block; merge keeps the previous block id and discards the merged-away id.
- `mark_wrong` projection fixture remains attached to the original id after split.

### Lane D1 — Paste markdown + undo/redo + inline marks

**Depends on**: Lane B  
**Scope**:
- Add markdown paste path into `body_blocks` while preserving allowed node schema.
- Confirm StarterKit history/undo-redo behavior does not violate block id policy.
- Add inline marks needed for basic editing (`bold`, `italic`, `code`, link if already in StarterKit/extension plan).

**Acceptance**:
- Pasting markdown creates valid PM JSON and rejects/strips unsupported nodes.
- Undo/redo preserves stable block ids for ordinary text edits.
- `CODEX_FULL_GATE=1 pnpm test:unit` targeted editor tests pass.

### Lane E1 — ADR-0021 draft + integration audit

**Depends on**: Lane A + Lane B + Lane D1  
**Scope**:
- Create `docs/adr/0021-*.md` for PM node schema, allowed attrs, split/merge semantics, renderer mapping, and bundle boundary.
- Update P2 status in roadmap/status docs only after implementation evidence exists.
- Run a small integration audit for P3/P4 consumers: `NoteGenerateTask`, `NoteVerifyTask`, `NoteRefineTask` future contracts must reference the same node names.

**Acceptance**:
- ADR-0021 names every shipped custom node and marks which parts are basic vs polish.
- `rg -n "semanticBlock|crossLink|artifactRef|body_blocks|block_id" docs src app` does not reveal contradictory naming.
- P3/P4 can start from the ADR without guessing the PM node schema.

## Out of scope

- Slash command, drag-drop, mention picker, cross_link picker polish.
- P3 AI pipeline rewrite.
- P4 Living Note mutator / idle coordination production logic.
- P5 backlinks and hub auto-sync.

## Chain-merge order

Lane A -> Lane B -> Lane D1 -> Lane E1.
