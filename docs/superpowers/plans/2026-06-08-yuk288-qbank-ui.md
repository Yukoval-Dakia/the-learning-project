## 0. TL;DR — 与 SPEC 的关键差异（先读这条）

SPEC 假设「列表页接 #352 真 API、API 增量补 `?subject=` 派生轴、composite 父题展开」。勘察后多处与现实有出入，须按本计划调整（四处缺口 A/B/C/D 见 §3）：

1. **API 比 SPEC 想的更全**：`#352`（YUK-280 P4）已 ship 完整 reader——`src/server/questions/list.ts`（多轴 + grounding tier + variant family 聚合/展开）+ `src/server/questions/detail.ts`（单题聚合：source_tier / labels / variant family / per-knowledge FSRS+decay / backlinks / timeline）+ `app/api/questions/route.ts` + `app/api/questions/[id]/route.ts`。**列表/详情主体直接消费现有 JSON，无需新建 reader。**
2. **字段名与设计 mock 完全不同**——UI 必须做映射层（§4 字段映射表）。设计 mock 用 `stem/options[{key,text}]/answer/knowledge/source`；真 API 用 `prompt_md / choices_md:string[] / reference_md / labels:[{id,name}] / 派生 source_tier`。
3. **composite 在真模型有两套表示，二者 reader 都不暴露**：
   - (a) `parent_question_id`+`part_index`（`kind='question_part'`，schema.ts:180-181，写路径 `parts.ts`）—— 顶层父题 + 独立 part 行；
   - (b) `question.structured`（`StructuredQuestionT` 递归树：`role:'stem'` 带 `prompt_text`(=passage)+`sub_questions[]`，schema.ts:193 / structured_question.ts:94-125）—— 单行内联 passage+小题，正是设计 mock `passage`/`children` 的直接同构。
   `list.ts`/`detail.ts` 都不返回这三者中任何一个（grep 确认 `QuestionDetail` 接口无 `parent_question_id`/`part_index`/`parts`/`structured`；`prompt_md` 经 `structuredToPromptMarkdown` 已把 passage 折进全文，但拆不出 passage / 小题边界）。**一个题用哪套表示取决于其录入路径**（VLM/Tencent 抽取走 `structured`；YUK-165 part 轴走 parent_question_id）。→ 这是 §3 缺口 A：补 reader 时**必须先判定该题走哪套**，不能只补 parts（先验证 phase-1 数据实际用哪套，见 §3-A）。**有现成读路径可仿**：`src/server/review/paper-detail.ts:247-263` 已 select `parent_question_id`+`part_index`，是「就近补 select」的直接先例。
4. **`subject` 轴未实现**——SPEC 要求新增 `?subject=` 派生轴：`question → knowledge_ids → knowledge.domain/effective_domain`。这是 SPEC 点名的 API 增量（§3 缺口 B），**派生 join，绝不给 question 加 subject 列**（subject 模型终版第 2 条 / YUK-249 钉板）。
5. **`source_tier` ≠ 设计的「来源」语义**。设计的来源筛选（种子/小测/试卷/AI 变体）对应真模型的 `source` text 列（enum 值 `manual/daily/final/dreaming/vision_*/mistake_variant/quiz_gen/...`），而非派生的 grounding `source_tier`。列表的「来源」筛选轴 UI 上走真 `source`（list reader 已支持 `source` 精确轴），grounding tier 作为次要的微指示/可选筛选（reader 已支持 `source_tier[]`）。

> 总原则（SPEC ④）：**缺字段就近补进 #352 reader（缺口 A/B），或 UI 降级显示；禁发明后端大改、禁动 schema。**

## 3. API 形状缺口的处理决定

> 四处缺口（A composite reader / B subject 派生轴 / C 题面全文搜索 / D 草稿-only 筛选）。处理原则（SPEC ④）：就近补 reader 或 UI 降级，禁后端大改/禁动 schema。仅 A/B 需补 reader（commit 2），C/D 走客户端降级。

### 缺口 A — composite reader 未暴露（两套表示）⇒ 先验数据再就近补 reader（commit 2）

- **现状（校正：两套表示，二者都不暴露）**：
  - (a) `parent_question_id`+`part_index`（`kind='question_part'`，schema.ts:180-181）：`parts.ts` 是 INSERT 写路径，read 端 `list.ts`/`detail.ts` 不返回。**已有读先例** `paper-detail.ts:247-263`（select parent_question_id+part_index）。
  - (b) `question.structured`（递归 stem/sub 树，schema.ts:193 / structured_question.ts:94-125）：`role:'stem'` 的 `prompt_text` = passage、`sub_questions[]` = 小题，**与设计 mock `passage`/`children` 直接同构**；detail.ts 也不返回 `structured`（`prompt_md` 把 passage 折进全文但拆不出边界）。
- **⚠️ 落地前必做（commit 2 第一步）**：跑一次数据勘察 SQL（`SELECT count(*) ... WHERE kind='question_part'` 与 `WHERE structured IS NOT NULL` 与 `WHERE parent_question_id IS NOT NULL`），**确认 phase-1 真实数据走哪套**——不要先验假设是 parts。设计 mock 的廉颇阅读题/数学导数题都是 passage+children 形态，更可能命中 `structured`。
- **决定（依勘察结果二选一，executor 在 commit 写明）**：
  - 若数据走 `structured` → `detail.ts` 直接把 `structured`（或抽 `{passage, sub_questions[]}` 的瘦投影）加进 `QuestionDetail`，UI 渲染 stem.prompt_text 为 passage、map sub_questions 为小题列表。**零新 join，select 既有 jsonb 列即可。**
  - 若数据走 part 行 → `detail.ts` 加 `loadParts(db, parentId)`（仿 `loadFamilyMembers` + paper-detail.ts 先例），返回有序 parts；`list.ts` 默认排除 part 行（`parent_question_id IS NULL`）。
- **降级备选**（若 Cross-统合 判定 composite 展开超 S1 范围，或 phase-1 数据根本没 composite 题）：UI 只渲染 variant family（reader 已全支持）+ stem 全文（`prompt_md` 已含），composite 小题边界打 `phase-deferred` 注释。**就近补字段，不改 schema、不改 INSERT 路径。** 三路径交统合拍板。

### 缺口 B — `subject` 派生轴未实现 ⇒ 补 list reader（commit 2，SPEC 点名增量）

- **决定**：`list.ts` 加 `subject?` 参数，派生 `subject → knowledge(domain/effective_domain) → knowledge_ids → @>[id]` 容器筛选。**复用现有 effective_domain 继承逻辑（node-page.ts），绝不给 question 加 subject 列（YUK-249 钉板 / subject 模型终版第 2 条）。**
- detail 页不需要 subject 轴（单题），但属性右栏可**降级展示**派生 subject（由 labels 的 domain 推出）——或省略，视统合。

### 缺口 C — 题面文本搜索 API 缺口 ⇒ S1 客户端降级 + 留 Linear follow-up

- **现状**：`list.ts` 无 `prompt_md` 全文搜索轴（只有精确多轴 + tier + family）。设计要求「题面文本搜索」。
- **决定（S1）**：**客户端降级**——列表 `useQuery` 取当前筛选页（百到千级，SPEC 量级假设内），前端对 `prompt_md` 做 `includes` 子串过滤（复刻 screen-questions.jsx 的 `matchQuery` 行为：题面/知识点 label/题号）。这是设计 mock 本来就用的前端过滤路径，且符合「禁发明后端大改」。
- **follow-up**：服务端全文搜索（pg `ILIKE`/`tsvector`）作为 YUK 跟进项（百级 OK、近千级前端过滤可接受，过千需服务端）。→ §7 Linear gate 记一条。

### 缺口 D — 「草稿-only」筛选 API 轴缺口 ⇒ 客户端降级（无新增 API）

- **现状**：list reader 只有 `includeDrafts: boolean`——`false`(默认) = 排除草稿（`draft_status IS NULL OR <> 'draft'`，list.ts:179-183），`true` = 全收。**没有「只看草稿」的精确轴**（route.ts 的 `ListQuerySchema` 无 `draft_status`）。设计的三 tab「全部 / 正式 / 草稿」中，「草稿-only」无直接 API 支持。
- **决定（S1，禁后端大改）**：全部 tab → `include_drafts=true`；正式 tab → `include_drafts=false`（默认）；**草稿 tab → 取 `include_drafts=true` 后客户端按 `draft_status==='draft'` 过滤**（与缺口 C 的客户端子串过滤同路径，量级内 OK）。**不给 list reader 加 `draft_status` 精确轴**（最小改动原则；若日后草稿量大再服务端化，并入缺口 C 的 follow-up issue）。
- 注意：list `QuestionListItem` 已含 `draft_status` 字段（list.ts:96），客户端过滤有据可依，无需补 reader。

### 其它：字段名映射（非缺口，是映射层，§4）

设计 mock 的 `stem/options/answer/knowledge/source/composite/children/passage/attempts/inReview/papers/mistakes/origin/depth/variants` 全部经 UI 映射层翻到真 API 字段，**不改 API 去迁就 mock 命名**。
