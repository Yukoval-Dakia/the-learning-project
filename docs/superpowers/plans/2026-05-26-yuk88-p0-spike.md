## Phase goal

Throwaway spike，验证 P1+ 范围的 3 个核心 invariant：
1. **TipTap PM doc 形态可行**：`body_blocks` = `doc.toJSON()` 能渲染 + 编辑 + 序列化往返不丢失
2. **Notion 位置规则 mark_wrong 不漂移**：mark block b1 → split b1 为 b1' + b_new → mark_wrong **仍命中 b1'**（不命中 b_new）
3. **Idle 协调机制可落地**：client presence heartbeat + server idle detection + patch queue flush 三段能联通

**Spike 性质**：**不进 main，不 merge，不 chain-merge**。Spike branch 跑通后扔。Spike 结论写在 PR 描述里，feed back 给 P1 的 ADR-0020 草稿（如需微调）。

## Lane 切分

**单 lane**（P0 范围太小不值得多 lane）：

### Lane S — TipTap fixture spike

**Owner**：lane subagent in isolated worktree（spike branch）
**Branch**：`spike/yuk-90-tiptap-block-tree`（**不**用 Linear 自动生成的 branch name；spike 性质显式标 `spike/` prefix）
**Worktree**：`worktrees/yuk88-p0-spike/` （`superpowers:using-git-worktrees`）

**Scope（按 [YUK-90](https://linear.app/yukoval-studios/issue/YUK-90) Tasks）**：

1. **Fixture 准备**：手写一份 `body_blocks` JSON（PM `doc.toJSON()` 形态，含 `{type, content[], attrs, marks[]}`）
   - 覆盖 5 个 `semantic_kind` block（definition / mechanism / example / pitfall / check）
   - 1 个 `cross_link` inline node（指向某个虚构 artifact_id + block_id）
   - 1 个 `artifact_ref` block（指向虚构 tool_quiz artifact）

2. **TipTap 集成**：
   - 安装 TipTap StarterKit + 必要扩展（`@tiptap/react`、`@tiptap/starter-kit`）
   - 写 2 个自定义 node：
     - `SemanticBlock`（顶层 block，attrs 含 `semantic_kind`、`id`）
     - `ArtifactRefBlock`（inline 渲染 placeholder，attrs 含 `target.artifact_id`、`target.kind`）
   - 写最小 NodeView 让 fixture 能渲染出来（不做 production UI，只验证 schema 可往返）

3. **Split / merge invariant 验证**：
   - 在 SemanticBlock b1 末尾按 Enter → 应得 b1（保留 id）+ b_new（新 id）
   - 在 b_new 开头按 Backspace → 应合并回 b1（b_new id 丢弃）
   - 输出：将 split/merge 前后的 `doc.toJSON()` snapshot 截下来，明示 id 变化

4. **mark_wrong 命中验证**：
   - mock 一次 mark_wrong on b1（外部 SQLite-like store：`{ block_id: b1, status: 'wrong' }`）
   - split b1 → b1' + b_new
   - 重新读 mark_wrong projection：**断言** b1' 命中 wrong（=原 id 保留）、b_new 不命中
   - 这是 ADR-0020 §2 anchor invariant 的可执行 reproduction

5. **Idle 协调 mock**：
   - 写一个最小 mock：
     - Client side：`presence: 'editing' | 'idle'` state + 5s idle 转 `idle` 的 mock heartbeat
     - Server side：一个待处理 patch queue（手写 op `{block_id, op: 'replace_content', new_content}`）
     - 行为：`editing` 时 patch defer；`idle` flush + apply（直接 mutate fixture doc）
     - 超时（如 10min）强行 apply（spike 跑 10s 验证机制）

6. **Spike 结论文档化**：
   - PR description 写：
     - Fixture 形态最终决定（PM 模型 vs Codex 早期建议的 `{id, type, children, attrs}` —— 拍板 PM 胜出）
     - Split/merge 实际行为是否与 ADR-0020 §2 描述一致（如有 gap，列出 ADR 该改的字段）
     - Mark_wrong 命中是否符合 Notion 位置规则
     - Idle 协调机制可否落地（如有阻塞，说明 ADR-0020 §8 idle 部分该怎么修）

**P0 不 chain-merge**。流程：

1. Spike worktree 跑通 → push spike branch 到 remote → 开 PR（**不**合并）
2. PR description 作为 P0 deliverable
