# 架构重设计 Design Spec — 功能逻辑先行

> 2026-06-10 · brainstorming 阶段产物 · 状态：**待用户审阅**
> 调研底稿：`docs/superpowers/specs/2026-06-10-functional-goal-map.md`（功能目标地图 + 可维护性证据，13+ agent 调研产出）
> 本 spec 的裁决次序：**应然功能逻辑（§1）是最高准绳**，架构（§2）、菜单（§3）、迁移（§4）全部服从它。

## 0. 为什么重设计（诊断结论摘要）

用户判断「现有架构难以继续维护」。诊断证据显示病灶**不是**模块边界混乱或逻辑复制（层间依赖 100% type-only、共享逻辑真实复用、code-vs-ADR 漂移小——骨架健康），而是：

1. **中心文件收费站**：`src/server/proposals/actions.ts` 2151 行（`acceptAiProposal` 单函数 1441 行、switch 22 种 kind）；90 天提交热点 `registry.ts` 61 次、`task-prompts.ts` 45 次、`schema.ts` 34 次、`globals.css` 86 次——每个新功能都过同一批文件。
2. **巨型页面**：`knowledge/page.tsx` 1313 行、`review/page.tsx` 1059 行（15 useState）、`record/page.tsx` 18 useState，取数/状态机/渲染缠绕。
3. **工程脚手架税**：vitest 三 config + 112 条手写 allowlist（失败模式「静默绿色不执行」）+ audit 脚本群看守。
4. **认知负荷**：30+ AI 任务、9 种 proposal、7 个 cron，无一张装得进脑子的图。
5. **实现-目标差距大**（用户裁定四种全占）：半成品多、已 ship 深度不够、概念错位、推进速度慢。

结论：**值得以功能目标为锚做架构重组，不需要也不应该推倒重写**（90 天 935 commit 中 24% 是 fix/revert——战伤逻辑是资产）。方案选型已定：**能动内核 + 插件式 capability，同仓渐进重建，迁移即补深**。

## 1. 应然功能逻辑

### 1.1 主轴

**AI 是学习的编排者（orchestrator），且只有一个 AI。**

- 用户只做四件事：**录题目、声明想学什么、做 AI 递来的练习、审批 AI 的提议**。
- 其余一切——归因、排期、组流、维护知识图谱与笔记、生成变式与卷、夜间自审与提议——都是 AI 的工作。
- 所有「机制」（FSRS、到期队列、任务状态机）退到幕后成为 AI 的内部工具。**用户永远不直接伺候机制。**

### 1.2 一个 AI（人格与记忆）

后台大 agent（dreaming / coach / maintenance）与 Copilot 对用户而言是**同一个「人」**：

- 它晚上「做梦」（dreaming = 它的夜间思考），早上留交班条（工作台），白天本人在场（Copilot）。
- **统一记忆与知情权**：后台一切动作对 Copilot 完全可见；用户问「为什么今天排这个」，AI 第一人称回答带因果链（「因为我昨晚看到你虚词那块衰减得厉害」）。这是「留痕可追溯」不变量升格为对话能力。
- **署名口径统一为「我」**：工作台、提议、观察全部第一人称；dreaming/coach/maintenance 降为内部分工名，不对用户出现。
- agent-notes 的语义从「各 agent 互相留言」变为「这一个 AI 的内心活动」；它本质是**小 task 与三个定时大 agent 之间的内部协调信道**（幕后基础设施）。

### 1.3 五个用户面

界面收敛为五个，其余一律幕后：

| # | 用户面 | 内容 |
|---|---|---|
| 1 | **中央工作台**（today 页重生） | AI 的每日交班台：今日练习流入口、待审提议、值得注意的观察、计划摘要 |
| 2 | **练习面 = 流 + 卷架** | 日常唯一练习入口（见 §1.6） |
| 3 | **知识图谱与笔记** | 认知结构的浏览与阅读（树骨架+网边、节点页、AI 维护的笔记）；纯阅读与手动修订，**不再承担练习** |
| 4 | **录入** | **任何题目**进系统的通道（试卷/题库/练习册；拍照/PDF/DOCX/手输）；「错题」是题目的标记/来源属性，不是录入前提；原图留存等不变量照旧 |
| 5 | **Copilot 抽屉** | 跨页随手呼出的对话（答疑/教学/操作代理） |

界面关系：**分离理顺为基底 + 中央工作台**——对话归对话、审批归审批，互相链接；工作台是每日默认入口。

### 1.4 数据环与传感器层

数据流是一个单向环：

```
意图与题目流入 → AI 编排 → 练习流流出 → 作答产生事件
   → 传感器层更新 → AI 修正路径与明日的流 → 循环
```

**传感器层**（全部幕后，唯一消费者是 AI）：

- **FSRS**：挂在知识点上，追踪记忆衰减。**纯数据源**，不再是面向用户的复习机制。
- 错题与错因（归因结果）
- 掌握度投影（事件派生，同时保留用户侧展示）
- 目标与路径进度
- 作答事件（流内散题与卷内题同构回流）

AI 对用户只有两条信道：**提议收件箱**（动用户数据须审批）+ **工作台观察摘要**。

### 1.5 规划链

单向链，所有权明确：

```
用户声明意图（goal /「我想学 X」）
   → AI 分解为中期路径并全权维护学习任务（learning_item）
   → 每天把路径 + 传感器数据翻译成练习流
   → 作答信号回流修正路径
```

- **learning_item 全权归 AI 维护**：创建/推进/养护/归档由 AI 直接做（软判断，留痕可回滚）或提议（低置信/重大转换）；用户只在工作台看到和纠偏。直接做与提议的精确分界在菜单表 ⚖️ 行裁决。
- goal 保持「只加方向不抑制」（ND-5 沿用）。

### 1.6 卷与流（练习的概念模型）

**流是「日程」，卷是「节目」。**

- **练习流** = 调度与递送层：无边界、不固化，AI 持续维护「接下来练什么」的序列（衰减复习、错题变式、新学自测、打包小卷），可随时插入/撤换/加减量。基本单位是 item。
- **卷** = 内容打包形态：组卷那一刻冻结的有界题目集合，固定清单、有头有尾、可整卷评价。
- **关系**：卷是流里的一种 item 类型。散题即时反馈；卷进「卷模式」——逐题作答、**反馈缓冲**（judge-now-show-later 语义保留于此）、整卷小结。
- **卷的三种出生**：AI 主动打包（塞流）、用户点播（「来份虚词专项卷」当场组）、外部导入（录入的整张试卷天然是卷）。出生后同构。
- **卷架**：卷的持久归宿。待做/在做/已完成，按来源与科目筛；完成的卷永久可复盘（当时作答、判分、小结、错题去向）。流里的 item 只是卷被排上日程的影子。
- **散题历史**不进卷架：挂在题目与知识点身上（题目详情、知识点活动时间线）。
- **解题会话**（苏格拉底分级提示）：流内每道题随时可呼出的能力，非独立入口。
- 复习评级回写：由流作答驱动（评级建议照常），到期队列 UI 整个删除。

### 1.7 Copilot 设计

- **定位**：绝大多数 AI 任务在后台；Copilot 是用户始终直接交流的 agent，是「那一个 AI」的在场形态。
- **权限直接继承能动性分层**，不另设规则：
  - **AI 职权内的东西，对话里当场改**：流的排序加减、今日计划、打包卷、教学节奏（软判断层）；
  - **用户数据，只 propose**：知识图谱、题目、笔记、任务重大转换——对话里生成提议卡，用户点了才落地。
- 教学（explain / ask_check）、点播组卷、答疑、操作代理均为 Copilot 内能力。

### 1.8 删除清单与连锁后果

**已裁决删除**（菜单第一批刀）：

1. **笔记内嵌自测**（embedded-check 全链路：attempt API、error-rate 触发、check 块交互、生成链）；
2. **学习记录**（疑问/顿悟/反思/资料类 capture 及其下游）；
3. **用户直面的 FSRS 到期队列**（UI 与直接交互；引擎与状态保留为传感器）。

**连锁后果**（设计内消化）：

- 「新学知识的第一次检验」唯一去处 = 练习流（AI 把新学知识点自测排进流）——一切检验都在流里发生；
- 复习页/练习页按「流 + 卷架」重塑（大 UI 重设计，实施时走 design-doc pre-flight / claude design 流程）；
- Living Note 触发器重构：error_rate 信号源消失，新信号源为流内作答；dwell 遥测去留连带裁决（⚖️）；
- 五段式笔记模板的 check 段处置：删除或改为指向流的非交互占位（菜单裁决落地形态）。

## 2. 架构设计

> 形状已经用户逐节认可（内核+包、数据/外壳/工程化、迁移三节）；此处按应然功能逻辑校准后定稿。

### 2.1 内核：六契约，小而硬

内核只承载七条产品不变量（§6），契约数**封顶 6**：

| 契约 | 承载不变量 | 形状 |
|---|---|---|
| **事件存储** | 事件唯一真相源；correction 唯一撤回 | `event` 表设计原样保留；`emitEvent` + correction 语义收进内核；任何 capability 不经事件不得写状态 |
| **投影引擎** | FSRS/掌握度是派生不是存储 | 投影 = 纯 reducer + 存储切片 + rebuild 命令；FSRS state、mastery、acceptance signals 均声明为投影 |
| **提议生命周期** | AI 写入仅 propose；可回滚 | proposal kind 注册制：每 kind 声明 schema + 落地器 + 幂等键 + 过期规则；收件箱查询、accept/dismiss、行锁、幂等内核统一做。`acceptAiProposal` 22 case 解体为各包落地器 |
| **能动性策略层** | 四层授权强制执行 | 每个工具/动作声明 effect 等级（read/judge/propose/consume-budget）；内核拒绝 AI 直写用户数据、统一预算与限流闸门；Copilot「当场改 vs propose」边界由此层裁决 |
| **AI 运行时** | 全程留痕、成本可观测 | 任务契约统一（input/output schema + prompt builder + 模型策略 + skill 白名单）；运行时管 provider 调用、结构化输出、重试、SSE、三表审计；**任务住在包里，运行时只是引擎**。统一记忆（§1.2）的读取面也由此层提供给 Copilot |
| **作业调度** | 异步优先、夜间错峰 | pg-boss 保留；包在 manifest 声明 handler + cron；内核管 worker 引导、job 事件流、清理 |

**反框架护栏**（红线）：契约数封顶 6；组合是一个**静态类型检查的组合根文件**（列出启用的包）；无动态加载、无通用插件总线；任何内核钩子只有一个使用方时降级回那个包（第二实例原则）；错误整形沿用 `http/errors` 模式收进内核 http 工具。

### 2.2 capability 包与组织三规则

每包自包含：

```
src/capabilities/<name>/
  manifest.ts    # 类型化声明：实体、AI 任务、proposal kinds、工具、cron、路由、UI 贡献
  schema.ts      # 本包独有 Drizzle 表切片
  server/        # 领域服务
  tasks/         # AI 任务（prompt + schema + handler 同居）
  proposals/     # kind 声明 + 落地器
  jobs/          # pg-boss handlers
  api/           # 薄 route handlers（不依赖 Next，外壳挂载）
  ui/            # 本包页面/feature 组件
  *.unit.test.ts | *.db.test.ts
  CONTEXT.md     # 一页纸
```

**三规则**：

1. **依赖无环**：包只依赖内核 + 共享词汇表（纯类型 ref）；包间调用只走 manifest 导出的公共接口，禁深层 import。
2. **事实经事件，查询经接口**：跨包写影响走事件；跨包读走公共服务；只有「事实」上事件，不过度事件化。
3. **横切面贡献制**：工作台块、Copilot 工具、proposal kind 由各包 manifest 贡献，shell/copilot 只组装。

### 2.3 包菜单（按应然逻辑校准，~11 个）

| 包 | 职责（应然口径） | 备注 |
|---|---|---|
| `ingestion` | 任何题目进系统：上传/OCR/VLM/切块/标注/入库 | 学习记录 capture 不迁（D11） |
| `practice` | **流编排器**（消费传感器产出今日流）+ **卷架**（卷生命周期）+ 判分路由 + 评级 + FSRS 传感器 + 变式轮换 + 解题会话 | 原 review/papers/due-list 域按 §1.6 重塑 |
| `knowledge` | 树+网+节点页+科目派生+mesh+掌握度投影 | 不变 |
| `notes` | artifact/五段笔记/Living Note（新触发器） | embedded-check 不迁（D6） |
| `quiz` | 组卷/题源搜索/单题起草/题解生成（点播与 AI 打包共用） | 可并入 practice，P1 后视体量定 ⚖️ |
| `agency` | 夜间编排：dreaming/coach/maintenance + agent-notes 内部信道 + 路径维护（learning_item AI 全权） | 「一个 AI」的夜间分工 |
| `copilot` | 对话在场形态：人格统一、教学、操作代理、hero 呈现 | 工具来自各包贡献 |
| `subjects` | SubjectProfile + capability registry + 技能解析 | 资产，原样收编 |
| `memory` | 记忆简报 + facts（统一记忆的存储侧） | 读取面经内核 AI 运行时 |
| `observability` | admin 四页（runs/cost/failures/subjects） | |
| `shell` | 工作台（交班台组装）+ 导航 + 主题 + 备份恢复入口 | today 重生为工作台 |

### 2.4 数据层：表不动，归属动

- **Postgres / Drizzle / 数据全保留**；表名、数据、迁移历史不动。
- `src/db/schema.ts` 解体为各包 `schema.ts` 切片，根部组合文件拼给 drizzle-kit。
- **表归属即审计**：每表必须被且只被一个包认领（manifest 对账脚本），替代 `audit:schema` + allowlist。
- 被砍功能的表**不删**：墓碑标记，终局阶段统一归档导出后才清理。

### 2.5 交付外壳与 UI

- **Next.js 保留，降为交付外壳**：`app/api/*` 一行挂载包 api handler；页面 = 组装（目标 ≤100 行）。包 api 不依赖 Next——将来拆独立 server 是搬接线不是搬逻辑（结构买期权，现在不付钱）。
- 其余栈全保留（pg-boss、R2+字节代理、TanStack/Zustand、Tailwind v4、Biome、Vitest）——逐项论证过「换它治不了任何在场的痛」。
- **巨页拆三层**：feature 组件（一个界面区域一个文件）+ 显式状态机（per-feature store）+ 组装页。
- `globals.css`：design tokens（@theme）留全局；组件级样式随包同居。
- 流 UI、工作台 UI 为新建面，实施时走 design-doc pre-flight + claude design 分工。

### 2.6 工程化与测试

- **vitest 单 config 双 project**，按文件名约定分区（`*.unit.test.ts` / `*.db.test.ts`）；**删除** 112 条 allowlist、`vitest.shared.ts`、`audit:partition`。
- DB 测试基建（testcontainer + 模板克隆 + per-worker fork）原样保留。
- `audit:schema` → 表归属对账替代；`audit:profile` → 保留（subjects 包内）。
- Postman：`api-endpoints.json` 拆为各包 manifest 路由声明，生成器聚合。
- 文档：每包 `CONTEXT.md` + 根部 `ARCHITECTURE.md`（内核契约 + 包清单 = 装进脑子的那张图）；111 个 AGENTS.md 冻结，迁移时逐包收编。

## 3. capability 菜单决策表（预打分）

> 图例：**keep** 原样迁入新形状 · **redesign** 按应然逻辑重塑后迁入 · **drop** 不迁（墓碑）· ⚖️ 争议行需用户裁决。
> 预打分人：AI；裁决权：用户。非 ⚖️ 行若用户不提出异议即按预打分执行。

### 3.1 录入与资产

| capability | 判定 | 理由 |
|---|---|---|
| 多模式录入入口 | redesign | 删学习记录模式；重整为「任何题目」通道（D11） |
| 图片上传 + 内容寻址 R2 | keep | 不变量基础 |
| PDF 渲染 / DOCX 双线 / 三层 OCR-VLM / 块组装 / 自动标注 / 录入会话 | keep | 战伤资产，应然角色不变 |
| 渐进式自动入库 | keep ⚖️ | 观察模式保留；**自动模式继续投入还是冻结**待裁 |
| R2 字节代理（读侧） | keep | 「永不直出 R2」不变量 |
| 人工纠错簇：vision rescue / auto-enroll revert / backfill | keep | 人工纠偏通道 |
| 复习申诉（appeal stub） | ⚖️ | 现为 stub 不重判；**补完重判链 or 砍掉**待裁 |

### 3.2 练习（流 + 卷架）

| capability | 判定 | 理由 |

## 5. 风险与对策

| 风险 | 对策 |
|---|---|
| 内核长成自建框架（违反反过度工程原则） | 契约封顶 6、静态组合根、第二实例原则写死本 spec；P1 打样专门验「契约是否过度」 |

## 6. 不变量与新形状映射

七条产品红线在新形状中的承载点：

| 不变量 | 承载点 |
|---|---|
| 单用户无多租户 | TokenGate 照旧（外壳层） |
| AI 行为可追溯可回滚 | 内核 AI 运行时三表审计 + 提议生命周期；升格为 Copilot 第一人称 provenance（D14） |
| 事件是唯一真实来源 | 内核事件存储 + 投影引擎；correction 唯一撤回 |

| 原图必须同步留存 | ingestion 包 + R2 字节代理 |
| 树是骨架网是肌肉 | knowledge 包建模不变 |
| 写入仅 propose | 内核能动性策略层强制；Copilot 权限继承之（D13） |
