> Status: **design approved 2026-05-31** (feature-level decision). Implements roadmap **P5.4**
> ("Proposal Quality Rubric enforce 化"); a **Layer-8 Copilot prerequisite** (every Layer-8 agent
> feeds the same proposal write path). Refs
> [YUK-143](https://linear.app/yukoval-studios/issue/YUK-143) +
> [YUK-174](https://linear.app/yukoval-studios/issue/YUK-174) (Layer-2 adaptive bias — **planned next**, YUK-174 Todo).

## 1. Problem

The proposal-quality rubric is **documented in `docs/modules/knowledge.md` §4 but only partially
enforced in code**. §4 defines three gate classes: §4.1 seven universal structural gates, §4.2 an
evidence-level model (strong / medium / weak with relation-specific thresholds), and §4.3
relation-specific gates for five relations. The two agent proposal write paths each enforce a
different, incomplete subset, and there is **no shared validator** between LLM output and the
propose-event write. Concretely:

- **DomainTool path** — `proposeKnowledgeEdgeExecute` (`src/server/ai/tools/proposal-tools.ts:230`,
  the `propose_knowledge_edge` tool the Copilot / Teaching / Dreaming tool-loops call) runs **6
  structural gates**: `self_edge` (`:235`), `unknown_node` (`:243`), `cross_subject` (`:251`, via
  `getSharedKnowledgeDomain` `:121`), `parent_semantic_duplicate` (`:258`, `related_to`-only),
  `duplicate_live_edge` (`:268`), `duplicate_pending` (`:305`, via `pendingProposalWithAnyCooldown`
  `:61`). It then writes the proposal at `:309` with `evidence_refs` built from the optional
  `evidence_event_ids` input (`:209`, `:316`) — **no evidence-level check, no relation-specific
  gate, no reasoning-depth gate**. A medium/weak or evidence-free agent edge sails through.
- **Legacy MCP path** — `runWriteProposal` (`src/server/knowledge/review.ts:304`), the
  `write_proposal` in-process MCP tool of `KnowledgeReviewTask` (`:371`). Its only gate is
  `checkProposalGate` (`:160`): dup-pending (`findPendingProposalForGate` `:251`) + cooldown
  (`lockProposalSignal` `:189`, reading `proposal_signals.cooldown_until`). It does **not even run
  the structural gates** the DomainTool runs — and it writes edge proposals with
  `evidence_refs: []` hardcoded (`:345`).
- **Nightly batch path** — `runEdgeProposeAndWrite` (`src/server/knowledge/propose_edge.ts:69`,
  `KnowledgeEdgeProposeTask`) runs 4 inline gates (self-loop `:120`, unknown-node `:124`,
  duplicate-edge `:129`, duplicate-pending `:133`) and attaches the run's recent-failure event ids
  as `evidence_refs` (`:145`). Also no evidence-level / relation gate. *(In scope as a third call
  site only if the design chooses to route it through the validator — see §5 Q5; P5.4's locked two
  call sites are the DomainTool and legacy-MCP paths.)*

Net impact (readiness brief §P5.4): Dreaming / Coach / Copilot can propose low-quality edges
(same-name siblings as `contrasts_with` with zero confusion evidence; `related_to` as a dumping
ground), the inbox shows them, and dismissal feeds back nothing — there is no structured signal a
later layer could learn from. P5.4 closes the deterministic floor and emits a structured verdict so
the adaptive layer (YUK-174) can later consume it.

Two facts about the current state are load-bearing for this design:

1. **Proposals are event-derived; there is no `proposals` table with a `status` column.** A
   proposal is a propose event (`action='propose'` for knowledge / knowledge_edge, or
   `experimental:proposal` / `experimental:knowledge_*` for the rest — `writer.ts:43`). Its
   lifecycle status is **computed** from the latest chained `rate` event:
   `rateStatus()` (`inbox.ts:55`) maps `rating: accept|reverse|change_type → accepted`,
   `dismiss → dismissed`, `rollback → stale`, **absent rate → pending** (overlaid with
   correction-status, `inbox.ts:276`). "Live-pending" for dedup/cooldown is exactly this: a propose
   event on a `(kind, cooldown_key)` with **no** chained rate (`findPendingProposalForGate`
   `review.ts:251`; `pendingProposalWithAnyCooldown` `proposal-tools.ts:61`). The whole
   reject-folding design (§3.4) depends on this event-derived model; see the §8 contradiction note —
   it holds.
2. **There is no user-edited proposal write path today.** Both validator call sites are agent-only
   in practice: the DomainTool runs under `ctx.callerActor.ref` (copilot / teaching / dreaming) and
   the legacy MCP path hardcodes `actor_ref: 'dreaming'`. A user manually creating an edge goes
   straight to `createKnowledgeEdge` (`edges.ts:120`, `created_by` defaults `'user'`), **not**
   through a proposal. So the "user-edited proposal = structural-only" branch (§2 RB-3) is a
   forward-looking provision keyed on `ctx.callerActor.kind !== 'agent'`, correct the day a
   user-edit-then-propose path exists, and a no-op until then. Stated so the impl doesn't assume a
   user path it has to find.

## 2. Locked decision

**Scope = Layer 1 only (Option A, two-layer split).** P5.4 ships **only** the deterministic rubric
floor — a pure-ish service that runs §4.1 / §4.2 / §4.3 gates and returns a structured verdict.
The adaptive accept-learned bias (reading `proposal_signals` to tune thresholds and feed agent
prompts) is **out of scope for this slice**, tracked as **YUK-174 (Layer 2)**. The Layer-1 validator
returns a structured `{ ok:false; gate; reason }` precisely so YUK-174 can later consume it. Layer 2
is **sequenced as the immediate follow-on** to Layer 1 (roadmap P5.4-L2; YUK-174 is in **Todo**, not
backlog) — the two-layer split is a build-order decision, not a deprioritization.

**Architecture (readiness-brief Option B):** a new
`src/server/knowledge/rubric-validator.ts` exporting

```ts
validateProposalQuality(
  payload: AiProposalPayloadT,            // the proposal about to be written
  db: Db,                                 // for evidence-event lookups (recency + judge-backed)
  ctx: { isAgent: boolean; actorRef: string },
): Promise<{ ok: true } | { ok: false; gate: string; reason: string }>
```

called by **both** write paths so neither bypasses it.

| # | Decision | Rationale |
|---|---|---|
| RB-1 | **Reusable SERVICE, not inline.** The rubric lives in one `rubric-validator.ts` called by both `proposeKnowledgeEdgeExecute` (`proposal-tools.ts:230`) and the legacy MCP `writeProposalAfterGate` (`review.ts:320`). Pure-ish: depends only on `(payload, db, ctx)`, testable by mocking `db` + `ctx`. | Option A (inline in the DomainTool) would leave the legacy MCP path permissive (readiness brief §P5.4) and bury the agent/user distinction in dispatch. One service = one source of truth for "is this proposal good enough", reachable by every Layer-8 agent. |
| RB-2 | **Gate coverage = all three §4 classes.** §4.1 **all 7 universal structural gates** (enumerated §3.1); §4.3 relation-specific gates for **all four documented mesh relations under enforcement** — `prerequisite`, `contrasts_with`, `applied_in`, `related_to` (§4.3 also lists `derived_from`; see §8 — the relation set under §4.3 is 5, but `derived_from` overlaps tree-parentage and folds into the §4.1 `parent_semantic_duplicate` gate, so RB-2 enforces a relation-specific predicate for the 4 above and treats `derived_from` under the structural class); §4.2 evidence-level. | §4 is THE source of truth; enforcing a subset is the status quo this spec ends. |
| RB-3 | **Agents = strict; user-edited proposals = structural-only.** `ctx.isAgent === true` runs all gates incl. the §4.2 evidence floor and §4.3 relation gates. `ctx.isAgent === false` (a user-edited proposal) runs **only** the §4.1 structural guards and skips evidence/relation strictness. Driven by `ctx.isAgent`, derived from `ctx.callerActor.kind === 'agent'` at the DomainTool site (`types.ts:32`) / `true` at the legacy MCP site (always `'dreaming'`). | §4 intro: "Manual user-created edges can be looser, but should still pass structural guards." Human judgment overrides the evidence heuristics; the structural guards (self-edge, unknown node, cross-subject, duplicate) are still correctness invariants for anyone. |
| RB-4 | **Evidence floor (agents only).** An agent edge proposal must carry `evidence_refs` (and/or `evidence_event_ids`) that is **non-empty**, with **≥1 referenced event inside the evidence window** (RB-5) that is **judge-backed** (§3.2). The §4.2 level is computed strong/medium/weak; **agents are rejected at medium/weak** — only `strong` writes. Downweighting medium (the §4.2 "yes, lower weight" row) is **Layer-2 / YUK-174**, NOT here. | §4.1 gate "agent proposal has no `evidence_event_ids`" + §4.2 "weak: no for agent proposal". Today `evidence_refs` defaults `[]` (`proposal.ts:50`) and nothing enforces it. Rejecting medium at Layer 1 keeps Layer 1 deterministic (no confidence-weighted writes); the nuance moves to YUK-174. |
| RB-5 | **Evidence window = 30 days, a single configurable constant** named `RUBRIC_EVIDENCE_WINDOW_DAYS` (single-source, in `rubric-validator.ts` or a shared rubric-config module). | §4.2 "recent window: 30 days". Tunable in one place; consumed by the recency check (RB-4) and by the §4.2 level computation. |
| RB-6 | **Reject behavior = fold in inbox, do NOT silently drop.** On an agent reject, the validator's verdict is returned to the call site, and the call site **writes the propose event anyway**, MARKED as rubric-rejected — carrying `{ rubric_verdict: { ok:false, gate, reason } }` (flag-on-payload vs distinct event action decided at impl, §5 Q3). The reason is **also returned to the agent** as tool output (primitive feedback) and logged to the ai run log (`writeToolCallLog`, `log.ts:23`, evidence-first / traceable). | User decision "被 reject 折叠在 inbox": rejected proposals become a user audit trail + a Layer-2 (YUK-174) signal source, not lost work. Logging keeps the evidence-first invariant (CLAUDE.md / project memory). |
| RB-7 | **LOAD-BEARING: a rubric-rejected proposal MUST NOT count as live-pending.** The dup-pending / cooldown / `variants_max`-style in-flight counters MUST exclude `rubric_rejected` events. Otherwise a rejected proposal on `(kind, cooldown_key)=K` would make `findPendingProposalForGate` (`review.ts:251`) / `pendingProposalWithAnyCooldown` (`proposal-tools.ts:61`) treat K as occupied and **block a later, valid proposal for the same edge**. The rubric-rejected marker must be filtered out of every "pending propose with no rate on this cooldown_key" query. | A rejected proposal is *terminal*, not in-flight. If it counted as pending, the rubric would permanently lock out the very edge it rejected — the opposite of "fold + allow a better attempt later". Acceptance §7 pins this with a regression test. |
| RB-8 | **UI is out of scope.** The inbox FOLDED rendering of rubric-rejected proposals is a frontend concern, deferred to the claude-design redraw (note it in the redraw brief). P5.4 backend only: (a) write the marked event (RB-6); (b) make the inbox-derive query **expose** rubric-rejected proposals as a separate folded / low-visibility bucket or flag in its returned data (a derived field on `ProposalInboxRow`, `inbox.ts:17`, or a status discriminator). No UI / component code. | Project policy: UI redraw is pending; P5.4 ships backend substrate only. |
| RB-9 | **Layer-2 hook.** The structured `{ gate, reason }` verdict (RB-6) + the folded `rubric_rejected` event bucket (RB-8) are exactly what YUK-174 consumes: it reads rejected-vs-accepted-vs-dismissed per `(kind, relation, cooldown_key)` off `proposal_signals` (`signals.ts:113` `getProposalAcceptanceRates`) + the rejected bucket to tune thresholds and feed agent prompts. P5.4 must keep the verdict shape stable. | A stable verdict contract is the only coupling Layer 2 needs; designing it now avoids a breaking change when YUK-174 lands. |

### Rationale vs the readiness-brief options

The 2026-05-30 readiness brief recommended **Option B** (new `rubric-validator.ts` service called by
both paths) over **A** (inline in the DomainTool — leaves the legacy MCP path permissive) and **C**
(Zod schema-level — needs conditional-required which Zod can't express cleanly, schema churn). P5.4
adopts **B** verbatim, and the user added the two-layer split: ship the deterministic floor (Layer 1)
now, defer the adaptive accept-learned bias (Layer 2 / YUK-174). The reject-folding decision (RB-6 /
RB-7) is the user's refinement of the readiness brief's open question "on reject: silent skip or
write error log" — neither; write a folded event that is both audit trail and Layer-2 signal.

## 3. Design

### 3.1 The §4.1 universal structural gates (the 7)

From `knowledge.md` §4.1 (`knowledge.md:277`, "Reject before writing `event(action='propose')` when"
`:279`):

| # | §4.1 gate | Today enforced by | New validator responsibility |
|---|---|---|---|
| G1 | `from_knowledge_id === to_knowledge_id` (self-edge) | DomainTool `:235`; batch `:120` | re-assert (idempotent) — structural class, runs for user too |
| G2 | either node does not exist or is archived | DomainTool `:243` (`getKnowledgeNode` `:97` filters `archived_at`); `assertKnowledgeIdsExist` (`validate.ts:5`) | re-assert via `assertKnowledgeIdsExist` |
| G3 | nodes are in different subjects (cross-subject) | DomainTool `:251` (`getSharedKnowledgeDomain` `:121`) | re-assert |
| G4 | exact live edge `(from,to,relation_type)` already exists | DomainTool `:268` (symmetric-aware) | re-assert |
| G5 | exact pending proposal already exists (dup-pending) | DomainTool `:305`; legacy `:251`; batch `:133` | re-assert — **must exclude `rubric_rejected`** (RB-7) |
| G6 | proposal only repeats an existing tree parent/child relation | DomainTool `:258` (`related_to`-only `parent_semantic_duplicate`) | generalize: any relation that merely restates tree parentage (covers §4.3 `derived_from` "reject if already tree ancestor/descendant") |
| G7a | reasoning is generic, e.g. "二者相关" with no concrete signal | **NOT enforced anywhere** | NEW — reasoning-depth gate (§3.3) |
| G7b | agent proposal has no `evidence_event_ids` | **NOT enforced** (`evidence_refs` defaults `[]`, `proposal.ts:50`) | NEW, agent-only — folds into the RB-4 evidence floor |

G1–G6 are the **structural class** (run for agents AND users, RB-3). G7a (reasoning depth) and G7b
(evidence presence) are the gap the readiness brief calls out; G7b is agent-only and is the entry
point of the §4.2 evidence floor.

> **Coverage note (the "6 vs 7" gap):** the DomainTool implements 6 *named* structural skips
> (self_edge, unknown_node, cross_subject, parent_semantic_duplicate, duplicate_live_edge,
> duplicate_pending). Mapping to §4.1: those cover G1–G6. The 7th §4.1 item ("agent proposal has no
> `evidence_event_ids`") is G7b — un-enforced today. Plus §4.1's reasoning-quality item (G7a) is also
> un-enforced. So "5–6 structural gates implemented" (readiness brief) = G1–G6; the rubric's
> evidence + reasoning gates are entirely new.

### 3.2 The §4.2 evidence level (agents only, RB-4)

`knowledge.md` §4.2 (`knowledge.md:290`):

| level | meaning (§4.2) | agent write? |
|---|---|---|
| strong | 2+ recent failure events show same pattern, OR 1 failure + explicit user note | yes |
| medium | 1 recent failure event with clear judge analysis | yes, lower weight (§4.2) — **but Layer 1 REJECTS** (RB-4); downweight = YUK-174 |
| weak | name similarity / same parent / model intuition only | no for agent proposal |

Plus §4.2 "Default thresholds": recent window 30d (RB-5); minimum agent evidence **1 event for a
non-destructive edge, 2 events for `prerequisite` / `contrasts_with` unless judge analysis is
explicit**. The validator computes the level from the proposal's `evidence_refs` (kind `'event'`,
`proposal.ts:29`) by:

1. Resolving each referenced event id and checking it is **within `RUBRIC_EVIDENCE_WINDOW_DAYS`**
   (RB-5) of now (`event.created_at`).
2. Checking it is **judge-backed** — i.e. there is a `judge` event (`action='judge'`,
   `known.ts:52`) chained to / referencing that attempt with a cause analysis (the §4.2 "clear judge
   analysis"). The exact "judge-backed" predicate is §5 Q4.
3. Counting in-window judge-backed failure events → strong (≥2, or ≥1 + user note) / medium (1) /
   weak (0 or non-judge-backed). For `prerequisite` / `contrasts_with`, require **2** unless one has
   explicit judge analysis (§4.2 + §4.3).

Agents writing at medium/weak → `{ ok:false, gate:'evidence_level', reason }`.

### 3.3 The §4.3 relation-specific gates (agents only, RB-2)

`knowledge.md` §4.3 (`knowledge.md:304`) + the wenyan good/bad-evidence hints (§2.7,
`knowledge.md:198`):

| relation | §4.3 predicate the validator enforces | reject example (the overuse §4.3 warns of) |
|---|---|---|
| `prerequisite` | require learning-order evidence; prefer from = prerequisite concept/method, to = application/narrower task; **do not propose on mere co-occurrence** | two nodes co-occur in one broad prompt → reject (`gate:'prerequisite_no_order_evidence'`) |
| `contrasts_with` | require **confusion** evidence (same answer confuses both nodes / repeated failures referencing both); weight must reflect repeated confusion, not similarity | same-name siblings (e.g. all `之` children) with zero confusion evidence → reject (`gate:'contrasts_with_no_confusion'`) — the wenyan overuse risk (§5 Q5) |
| `applied_in` | require role compatibility: from = concept/method, to = application/assessment | both ends concept-families, no application direction → reject (`gate:'applied_in_role_mismatch'`) |
| `related_to` | conservative weak edge only; **more than co-occurrence**; penalize as a dumping ground | uncertain edge with no navigation/grouping value → reject (`gate:'related_to_dumping_ground'`) |

`derived_from` (the §4.3 5th relation) is handled by the structural G6 generalization (§3.1): reject
if from/to are already direct tree ancestor/descendant with no new semantics. Reasoning-depth (G7a):
reject when `reason_md` is generic with no concrete signal — operationalized as a cheap heuristic
(no referenced event id / node name / cause category named, or matches a generic-phrase denylist),
not an LLM call (Layer 1 stays deterministic). Tune at impl.

### 3.4 Verdict shape, write-anyway-folded, and the live-pending exclusion

**Verdict (stable for YUK-174, RB-9):**

```ts
type RubricVerdict =
  | { ok: true }
  | { ok: false; gate: string; reason: string };
```

`gate` is a stable enum-ish string (`'self_edge' | 'unknown_node' | 'cross_subject' |
'parent_semantic_duplicate' | 'duplicate_live_edge' | 'duplicate_pending' | 'reasoning_generic' |
'evidence_missing' | 'evidence_level' | 'prerequisite_no_order_evidence' |
'contrasts_with_no_confusion' | 'applied_in_role_mismatch' | 'related_to_dumping_ground'`). `reason`
is a human/agent-readable string.

**Flow at each call site (agent path):**

1. Run the existing pre-write structural skips (`proposeKnowledgeEdgeExecute` `:235`–`:307` /
   `checkProposalGate` `review.ts:160`). For the DomainTool these still short-circuit with their
   existing `skipped:*` statuses for the *pure* structural duplicates (G2–G5) — those are not
   rubric rejections, they are dedup no-ops, and they keep their current behavior.
2. Call `validateProposalQuality(payload, db, { isAgent, actorRef })` immediately **before**
   `writeAiProposal` (`proposal-tools.ts:309`) / `writeAiProposal` in `writeProposalAfterGate`
   (`review.ts:337`).
3. If `{ ok:true }` → write the proposal as today.
4. If `{ ok:false }` (agent) → **write the propose event MARKED rubric-rejected** (RB-6), carrying
   `rubric_verdict`, return `{ status:'skipped:rubric_rejected', reason }` (DomainTool) /
   the verdict (legacy MCP) to the agent, and log via `writeToolCallLog`.
5. Marking option (§5 Q3): either a `rubric_verdict` key in the propose event payload alongside
   `ai_proposal`, or a distinct event action (e.g. `experimental:proposal_rubric_rejected`). The
   flag-on-payload option is lighter and keeps the row inside the existing `proposalWhere()`
   (`inbox.ts:94`); a distinct action needs the inbox filter widened. Either way the marker is
   readable by both the inbox derive (RB-8) and the live-pending exclusion (RB-7).

**Live-pending exclusion (RB-7) — the load-bearing detail.** Every query that counts "a pending
propose on this `(kind, cooldown_key)` with no chained rate" must add `AND NOT <rubric_rejected>`:

- `findPendingProposalForGate` (`review.ts:251`) and `findProposalRowsForGate` (`review.ts:287`) —
  the legacy MCP dup-pending source.
- `pendingProposalWithAnyCooldown` / `pendingProposalWithCooldown` (`proposal-tools.ts:61` / `:52`)
  — they read `listProposalInboxRows(..., { status:'pending' })` (`inbox.ts:401`). If the marker is
  flag-on-payload, a rubric-rejected event would still derive `status:'pending'` (no rate chained,
  `rateStatus()` `inbox.ts:55`) — so the inbox derive (or these callers) MUST treat a `rubric_verdict`
  marker as a terminal, non-pending status. **This is exactly why RB-7 is load-bearing: under the
  event-derived model, "no rate event" otherwise means "pending", and a rejected proposal has no
  rate event.** The cleanest impl is: the inbox derive computes `status:'rubric_rejected'` (a new
  terminal `ProposalStatus`) when the marker is present, so dup-pending callers reading
  `status:'pending'` naturally exclude it, and RB-8's folded bucket falls out for free.

### 3.5 How `isAgent` is determined at each call site

- **DomainTool** (`proposal-tools.ts:230`): `isAgent = ctx.callerActor.kind === 'agent'`
  (`types.ts:32`), `actorRef = ctx.callerActor.ref`. Copilot / Teaching / Dreaming are all
  `kind:'agent'`; a future `kind:'user'` caller flips to structural-only (RB-3). (See §8 — no
  user caller exists today, so this branch is currently unreachable but correct.)
- **Legacy MCP** (`review.ts:320` `writeProposalAfterGate`): the path always runs as
  `actor_ref:'dreaming'` (`:340`), so `isAgent = true`, `actorRef = 'dreaming'`. If a user-facing
  caller is ever routed here, thread the real actor through (§5 Q2).

## 4. Migration

Additive; **no schema change, no DB migration, no UI** (except the inbox-derive status addition,
which is a derived field, not a column).

1. **Create `src/server/knowledge/rubric-validator.ts`** exporting `validateProposalQuality` (RB-1),
   `RUBRIC_EVIDENCE_WINDOW_DAYS` (RB-5), and the `RubricVerdict` type (§3.4). Reuse
   `assertKnowledgeIdsExist` (`validate.ts:5`) and `getEffectiveDomain` (`domain.ts`) for the
   structural class; add the §4.2 evidence resolver (event recency + judge-backed) and the §4.3
   relation predicates.
2. **Wire the DomainTool call site** — in `proposeKnowledgeEdgeExecute` (`proposal-tools.ts`), after
   the existing structural skips and before `writeAiProposal` (`:309`), call the validator with
   `{ isAgent: ctx.callerActor.kind === 'agent', actorRef: ctx.callerActor.ref }`. On reject, write
   the marked event + return `status:'skipped:rubric_rejected'` (add to the output enum `:213`) +
   `reason`.
3. **Wire the legacy MCP call site** — in `writeProposalAfterGate` (`review.ts:320`), after
   `checkProposalGate` and before `writeAiProposal` (`:337`), call the validator with
   `{ isAgent: true, actorRef: 'dreaming' }`. On reject, write the marked event + return a
   `kind:'rubric_rejected'` result (add to `WriteProposalResult`, `:104`).
4. **Mark rubric-rejected events** (RB-6 / §3.4 step 5) — add the `rubric_verdict` marker via
   `writeAiProposal`'s payload (flag-on-payload) or a distinct action (§5 Q3).
5. **Exclude rubric-rejected from live-pending** (RB-7 / §3.4) — add a terminal
   `'rubric_rejected'` `ProposalStatus` to the inbox derive (`inbox.ts:15`, `rateStatus` `:55`) so
   it is not `'pending'`; verify `findPendingProposalForGate` (`review.ts:251`) and the
   `pendingProposalWith*` helpers (`proposal-tools.ts:52`/`:61`) consequently skip it.
6. **Expose the folded bucket** (RB-8) — the new `'rubric_rejected'` status is the folded /
   low-visibility bucket the inbox returns; no UI.
7. **Log rejects** (RB-6) — call `writeToolCallLog` (`log.ts:23`) with the verdict in `output_json`
   (and/or `error_reason`) at each call site.
8. Update `docs/modules/knowledge.md` §4 to mark the rubric **enforced** (Layer 1) and link this
   spec + YUK-174 for the deferred adaptive layer.

No change to `AiProposalPayload` (`proposal.ts`), `writeAiProposal`'s signature (`writer.ts:88`),
the `proposal_signals` table (`schema.ts:585`), the FSRS / review / due path, or the user-edit
`createKnowledgeEdge` path (`edges.ts:120`).

## 5. Open implementation questions (for the writing-plans stage)

1. **Exact §4.3 relation-gate predicates.** §3.3 fixes the *intent* per relation; the writing-plans
   step pins the concrete predicate (e.g. `contrasts_with` "confusion evidence" = ≥1 in-window
   judge-backed failure whose answer references both endpoint nodes; `related_to` "more than
   co-occurrence" heuristic). Start strict-but-cheap; no LLM call in Layer 1.
2. **`isAgent` threading on the legacy MCP path.** Today `writeProposalAfterGate` is always
   `'dreaming'` → `isAgent:true` is safe. If KnowledgeReviewTask ever surfaces to a user-facing
   caller, the real actor must be threaded (§3.5). Confirm no current caller needs it.
3. **Marker = payload flag vs distinct event action** (§3.4 step 5). Flag-on-payload keeps the row
   inside `proposalWhere()` (`inbox.ts:94`) and is lighter; a distinct action
   (`experimental:proposal_rubric_rejected`) is more explicit but needs the inbox filter + the
   dup-pending queries widened. Decide and apply consistently to the inbox derive **and** the
   live-pending exclusion (RB-7).
4. **"Judge-backed" evidence predicate** (RB-4 / §3.2). What metadata proves an `evidence_ref`
   (kind `'event'`) is judge-validated and ≤ `RUBRIC_EVIDENCE_WINDOW_DAYS`: a chained `judge` event
   (`known.ts:52`) on the same attempt with a non-null cause analysis, vs the
   `effectiveCauseForFailureAttempt` helper (`cause-policy.ts`) the batch path already uses. Reuse
   the existing cause resolver rather than a second path.

## 6. Non-goals (YAGNI)

- **No adaptive / learned thresholds, no feeding signals back to prompts.** Reading
  `proposal_signals` (`signals.ts`) to tune the rubric or inject accept-rate hints into agent
  prompts is **Layer 2 / YUK-174**. P5.4 emits the structured verdict; it does not consume signals.
- **No medium-evidence downweighting at Layer 1.** §4.2's "medium = yes, lower weight" is a Layer-2
  confidence-weighted write; Layer 1 rejects medium for agents (RB-4). The downweight lives in

  claude-design redraw's job (RB-8). P5.4 backend only.
- **No new `proposals` table / status column.** Lifecycle stays event-derived (§1 fact 1); the new
  `'rubric_rejected'` status is a *derived* value in the inbox projection, not a stored column.
- **No change to FSRS / review / due.** No change to `src/server/review/**`.
- **No change to the user-edit path beyond structural guards.** `createKnowledgeEdge`
  (`edges.ts:120`) is untouched; the structural-only branch (RB-3) is a forward-looking provision
  (§1 fact 2).
- **No schema-level (Option C) validation.** No `evidence_event_ids.min(1)` Zod change; the floor is
  runtime, agent-conditional.

## 7. Acceptance criteria

- **Each gate class rejects the right way (agents).** An agent edge proposal that (a) is a self-edge,
  (b) references a missing/archived node, (c) spans subjects, (d) duplicates a live edge, (e) only
  restates tree parentage, (f) has generic reasoning, (g) carries no evidence, (h) carries only
  medium/weak (out-of-window or non-judge-backed) evidence, or (i) is a `contrasts_with` between
  same-name siblings with zero confusion evidence (and the analogous `prerequisite` / `applied_in` /
  `related_to` cases) is rejected with the **correct** `{ gate, reason }`. One test per gate class.
- **User-edited proposal passes with only structural guards.** With `ctx.isAgent === false`, a
  proposal that fails the evidence floor / a relation gate but passes G1–G6 returns `{ ok:true }`;
  the same proposal with `ctx.isAgent === true` returns `{ ok:false }`.
- **Rejected agent proposal is folded, not dropped, and NOT live-pending.** An agent reject writes a
  propose event marked `rubric_rejected` (carrying `{ rubric_verdict }`), the inbox derive surfaces
  it in the folded / `'rubric_rejected'` bucket, and a **later valid proposal for the same edge
  `(kind, cooldown_key)` is NOT blocked** by dup-pending / cooldown (RB-7). The regression test: write
  a rubric-rejected edge proposal on key K, then a valid proposal on K, and assert the second is
  `proposed` (not `skipped:duplicate_pending`).
- **Evidence window enforced.** An agent proposal whose only evidence event is `RUBRIC_EVIDENCE_
  WINDOW_DAYS + 1` days old is rejected (`gate:'evidence_level'`); the same event 1 day inside the
  window (and judge-backed) passes the recency check. Test asserts the constant is the single source
  (no hardcoded 30 elsewhere).
- **Both write paths call the validator.** A test drives `proposeKnowledgeEdgeExecute`
  (`proposal-tools.ts:230`) and `runWriteProposal` → `writeProposalAfterGate` (`review.ts:304`/`:320`)
  and asserts each rejects an evidence-free agent edge (proving neither bypasses the service).
- **Structured verdict shape is stable for Layer-2.** A snapshot test locks the `RubricVerdict`
  union (`{ ok:true } | { ok:false; gate; reason }`) and the `gate` string set so YUK-174 can depend
  on it.
- **No FSRS / review change.** No edit under `src/server/review/**`; existing review/due tests pass
  unchanged.
- **Reject is logged.** A rubric reject writes a `tool_call_log` row (`log.ts:23`) carrying the
  verdict (evidence-first / traceable).

### Grounding contradictions / nuances flagged (per the locked design)

- **Proposals ARE event-derived (no `status` column) — the reject-folding design holds.** Verified:
  no `proposals` table; status is computed from chained `rate` events (`inbox.ts:55`). The only risk
  the event-derived model creates is the live-pending one RB-7 already addresses (a rejected event
  has no rate → would derive `'pending'` → must be marked terminal). No surprise status column was
  found.
- **§4 "agent proposal has no `evidence_event_ids`" (G7b) is documented but the schema can't express
  it.** `BaseProposal.evidence_refs` defaults `[]` (`proposal.ts:50`) and the live edge tools accept
  `evidence_event_ids` as **optional** (`proposal-tools.ts:209`). So the §4.1 evidence-presence gate
  is structurally inexpressible in Zod (which is exactly why the readiness brief rejected Option C);
  P5.4 enforces it at runtime in the validator (RB-4), faithfully implementing the locked decision.
- **`evidence_level` field from §4.5 does not exist on the payload.** §4.5 recommends adding
  `evidence_level` / `evidence_event_ids` / `subject_id` to the payload but says "Do not add this
  until the event schema and accept UI are updated together" — and they were not. So the validator
  must **compute** the level from `evidence_refs` (§3.2), not read a stored field. Faithful to the
  locked design (RB-4 says compute); flagged because §4.2's table reads as if a level were stored.
- **No user-edited proposal write path exists today (§1 fact 2).** The RB-3 structural-only branch
  is correct but currently unreachable; this is a faithful forward-looking provision, not a bug.
