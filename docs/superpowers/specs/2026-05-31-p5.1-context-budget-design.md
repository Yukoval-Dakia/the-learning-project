# Context Budget Policy (P5.1) — Design Spec

> Status: **design approved 2026-05-31** (brainstorming session). Implements roadmap **P5.1**
> ("Context Budget Policy"); a **Layer-8 Copilot prerequisite**. Refs
> [YUK-143](https://linear.app/yukoval-studios/issue/YUK-143).
> Source: 2026-05-30 P5 readiness brief §P5.1 (Option C recommendation) +
> 2026-05-31 feature-level brainstorm (numbers loosened per user decision).
> Build排期: writing-plans next; backend-only, no UI.

## 1. Problem

Every agent surface (Copilot, Dreaming, Coach, Knowledge Maintenance) reads the same DomainTool
suite, but the per-call and per-run bounds are **scattered as hardcoded constants across seven
files** with no unified policy. There is no single answer to "how much context may surface X pull
per turn / per run", no per-surface budget, and no rule that prevents a run from reading a subgraph
or a mistake list so large it **bloats the context window and distorts the agent's reasoning**
(misread priorities, signal buried under noise).

The scattered limits today (the inventory this spec consolidates):

|---|---|---|

| | `MAX_NODES` (expand cap) | 60 |
| | `query_knowledge` `limit` | 1–50, default 10 |
| | `expand_knowledge_subgraph` `maxNodes` | 1–60, default 30 |
| `src/server/ai/tools/query-mistakes.ts` | `limit` | 1–50, default 20 |
| | `PROMPT_SNIPPET_MAX` | 160 chars |
| | `sinceDays` | max 180 |
| `src/server/ai/tools/query-events.ts` | `limit` | 1–50, default 20 |

| `src/server/ai/tools/get-attempt-context.ts` | `timelineLimit` | 1–50, default 10 |
| | `listLearningRecords` limit | 25 (hardcoded) |
| `src/server/boss/handlers/dreaming_nightly.ts` | `DREAMING_MAX_PROPOSALS` | 5 |
| | `max_tool_calls` (in `buildDreamingInput`) | 8 |

Two facts about the current state are load-bearing for this design:

1. **Dreaming and Coach already enforce a run budget.** Each `*_daily.ts` handler counts
   `proposalWrites` and, in the MCP `beforeExecute` hook, returns a string
   (`"…proposal cap reached (5); stop creating proposals in this run"`) instead of executing the
   tool once the cap is hit. The model receives that string as the tool result and stops — this is a
   **soft, graceful stop, not a thrown rejection**. P5.1 keeps this mechanism; it does not introduce
   a hard error path.
2. **Copilot has no per-message budget at all.** It inherits only `maxIterations: 6` from the task
   registry. Nothing bounds total nodes/edges/event-rows a single user message can pull across its
   tool calls.

## 2. Locked decisions

| # | Decision | Rationale |
|---|---|---|
| CB-1 | **One config module `src/server/ai/tools/budgets.ts`** exposes per-surface `ContextBudget` constants. The currently-hardcoded Dreaming/Coach numbers move here unchanged; Copilot gains a new budget. Single tunable source of truth. | Today the numbers are smeared across 7 files; future tuning must be deliberate and in one place. |
| CB-2 | **Copilot per-user-message budget (loosened):** ≤ **10** tool calls, ≤ **250** total nodes+edges, ≤ **1000** event rows, per-excerpt ≤ **180** chars. | Single-user self-hosted; Copilot is user-facing and streams, so the user sees + recovers from any truncation. Generous-but-bounded beats a tight SaaS-style cap; the bound exists only to stop context bloat that would distort reasoning. 180 aligns to the existing `TEXT_SNIPPET_MAX`. |
| CB-3 | **Dreaming budget unchanged:** keep `DREAMING_MAX_PROPOSALS = 5` and `max_tool_calls = 8`, formalized as `DREAMING_CONTEXT_BUDGET`. Revisit after ~2 weeks of `dreaming_nightly` logs. | Already in production and working; tighten only with telemetry. |
| CB-4 | **Coach budget unchanged:** keep 5 proposals / 12 tool calls, formalized as `COACH_CONTEXT_BUDGET`. | Same as CB-3. |
| CB-5 | **Per-tool courtesy defaults are code-only (not user-configurable) for v1.** Each read tool declares a courtesy default limit; the tool runtime applies `min(tool default, remaining budget)`. Over-budget degrades **gracefully** (return what fits + visible truncation), never hard-rejects. | Keeps the existing soft-stop behavior; avoids profile-settings scope. User-configurable budgets are an explicit non-goal (§7). |
| CB-6 | **Counting unit = node / edge / event-row for v1.** Token-level accounting is **deferred to T-PD backlog**, not required for P5.1. | Row counting is cheap, deterministic, and adequate to prevent bloat. A token meter is a separate, larger piece of infra (already tracked as `maxCost` inactive / T-PD4). |

## 3. Design

### 3.1 `budgets.ts` shape — per-surface `ContextBudget`

A small, pure, IO-free module. No new dependencies.

```ts
// src/server/ai/tools/budgets.ts (design — not yet implemented)

export interface ContextBudget {
  /** Max DomainTool invocations counted against this budget for the run/message. */
  maxToolCalls: number;
  /** Max combined knowledge nodes + edges returned across the run/message. */
  maxNodesPlusEdges: number;
  /** Max event-table rows (query_events + query_mistakes + timelines) across the run/message. */
  maxEventRows: number;
  /** Max characters per text excerpt (prompt / answer / reasoning snippet). */
  maxExcerptChars: number;
  /** Existing Dreaming/Coach proposal-write cap; undefined for read-only surfaces. */
  maxProposals?: number;
}

export const COPILOT_CONTEXT_BUDGET: ContextBudget = {
  maxToolCalls: 10,
  maxNodesPlusEdges: 250,
  maxEventRows: 1000,
  maxExcerptChars: 180,
};

export const DREAMING_CONTEXT_BUDGET: ContextBudget = {
  maxToolCalls: 8,            // was hardcoded in buildDreamingInput
  maxNodesPlusEdges: 250,     // documents prior implicit ceiling; behavior unchanged
  maxEventRows: 1000,
  maxExcerptChars: 180,
  maxProposals: 5,            // was DREAMING_MAX_PROPOSALS
};

export const COACH_CONTEXT_BUDGET: ContextBudget = {
  maxToolCalls: 12,           // was hardcoded in buildCoachInput
  maxNodesPlusEdges: 250,
  maxEventRows: 1000,
  maxExcerptChars: 180,
  maxProposals: 5,            // was COACH_MAX_PROPOSALS
};
```

`maxNodesPlusEdges` / `maxEventRows` / `maxExcerptChars` for Dreaming/Coach are set to the same
generous ceilings as Copilot. They are **documentation of the prior implicit ceiling**, not a new
constraint: Dreaming/Coach already stop on `maxToolCalls` and `maxProposals` well before they could
approach these read totals, so adding them changes no observable behavior (acceptance criterion §8).
Keying budgets by `DomainToolSurface` (the existing union in `allowlists.ts`) is the natural fit; a
`BUDGETS_BY_SURFACE` lookup map can be added when a fourth surface (maintenance) needs an explicit
budget — until then maintenance falls back to a generous default.

### 3.2 How the tool runtime consults the budget

The MCP server is already built per surface with a `beforeExecute` hook (see
`buildMcpServerFromRegistry` usage in `dreaming_nightly.ts` / `coach_daily.ts`). P5.1 threads a
`ContextBudget` + a mutable per-run accumulator through that same construction:

1. **Courtesy default ⊓ remaining budget.** Each read tool declares a courtesy default `limit`
   (code-only, CB-5). Before execute, the runtime computes the effective limit as
   `min(toolDefaultLimit, callerRequestedLimit ?? toolDefaultLimit, remainingBudgetForThatUnit)`.
   The smaller of the tool's courtesy default and what the budget has left always wins.
2. **Graceful truncation, not rejection.** When the effective limit is below what the agent asked
   for, the tool returns **what fits** plus truncation metadata
   (`{ applied_limit, budget_remaining, truncated: true }`) — mirroring the existing
   `filter_applied` blocks the read tools already return. The agent sees it was capped and can
   re-strategise; nothing throws.
3. **Tool-call ceiling reuses the soft-stop.** `maxToolCalls` is enforced exactly like the existing
   proposal cap: once the accumulator reaches the ceiling, `beforeExecute` returns a short
   "budget reached" string instead of executing, and the agent stops. No new error class.
4. **Excerpt cap** is applied where excerpts are produced (`excerpt()` in `knowledge-readers.ts`,
   the `.slice()` calls in `query-mistakes.ts`), reading `maxExcerptChars` from the budget instead
   of file-local constants.

### 3.3 Dreaming / Coach formalization (no behavior change)

`buildDreamingInput` / `buildCoachInput` currently inline `max_tool_calls` and read
`DREAMING_MAX_PROPOSALS` / `COACH_MAX_PROPOSALS`. After P5.1 they read the same numbers from
`DREAMING_CONTEXT_BUDGET` / `COACH_CONTEXT_BUDGET`. The `beforeExecute` proposal cap keeps reading
`budget.maxProposals`. **The wire format of the `budget` block sent to the model and the
proposal-cap string are byte-identical** to today (5 / 8 and 5 / 12). This is a pure
constant-relocation refactor for these two surfaces.

### 3.4 Copilot per-message accounting — **tracked across the message's tool calls** (chosen)

A Copilot user message can fan out into several tool calls in one streamed turn. The budget must
bound the **sum across that message**, not each call independently. Two options were considered:

- **Stateless per-call** (each tool call independently respects the budget): simpler, but the model
  could issue 10 calls each pulling 250 nodes and bloat the context anyway — defeats the purpose.
- **Tracked across the message's tool calls** (one accumulator lives for the duration of one user
  message's agent turn): correctly bounds the total a single message contributes to context.

**Decision: tracked.** Justification: the entire point of the budget (CB-2) is to bound *total*
context the agent assembles before reasoning, and that total only means anything if it sums across
the calls of one message. A per-message accumulator is created when the Copilot chat endpoint starts
handling a user message, passed into the same `beforeExecute` / tool-runtime path Dreaming/Coach
already use, and discarded when the turn ends. It is the smallest stateful unit that makes the bound
meaningful, and it mirrors the per-run accumulator Dreaming/Coach already hold (`proposalWrites`).
Exactly where the accumulator is stored (StreamHandler-local vs threaded through the chat-state
object) is an implementation question — see §6.

## 4. Migration

Pure refactor for the batch surfaces; additive throttle for Copilot. No schema, no DB migration, no
UI.

1. **Create `src/server/ai/tools/budgets.ts`** with the three `ContextBudget` constants (§3.1).
2. **Dreaming/Coach:** replace the inline `max_tool_calls` and the `DREAMING_MAX_PROPOSALS` /
   `COACH_MAX_PROPOSALS` reads with reads from the budget constants. Behavior byte-unchanged
   (5 / 8, 5 / 12). Keep the constants re-exported (or aliased) so existing imports / tests don't
   break in the same commit.
3. **Read tools:** replace file-local `TEXT_SNIPPET_MAX` / `PROMPT_SNIPPET_MAX` with
   `budget.maxExcerptChars`; declare each tool's courtesy default `limit` next to its schema.
4. **Copilot:** wire the new per-message accumulator + `COPILOT_CONTEXT_BUDGET` into the chat
   endpoint's tool-runtime path. This is the only surface that gains *new* runtime behavior.
5. Update `docs/modules/agent-tools.md` to point at `budgets.ts` as the single source.

## 5. Open implementation questions (for the writing-plans stage)

1. **Where the Copilot per-message accumulator lives** — StreamHandler-local state vs threaded
   through the existing chat-state object. (§3.4 fixes *that* it is per-message and tracked; this Q
   is *how* it's stored.) Must survive parallel / out-of-order tool calls within one turn without
   double-counting.
2. **Whether truncation metadata is surfaced to the agent verbatim** — return the
   `{ applied_limit, budget_remaining, truncated }` block inside every capped tool output (lets the
   agent self-correct, costs a few tokens), or keep it server-side only (cleaner output, agent flies
   blind on truncation). Lean: surface it, consistent with the existing `filter_applied` echo.
3. **Maintenance surface budget** — give it its own `ContextBudget` now, or let it fall back to a
   generous default until it has a real consumer.

## 6. Non-goals (YAGNI)

- **No token meter.** Token-level accounting stays deferred (T-PD / the inactive `maxCost`); v1
  counts rows only (CB-6).
- **No user-configurable budgets.** Courtesy defaults and per-surface budgets are code-only for v1
  (CB-5); profile-settings exposure is out of scope.
- **No circuit-breaker / hard rejection.** Over-budget degrades gracefully (return what fits + soft
  stop), matching the existing `beforeExecute` cap. No new error class, no run-killing.
- **No allowlist changes.** `allowlists.ts` keeps owning *which* tools a surface may call; `budgets.ts`
  owns *how much* it may read. The two stay orthogonal.

## 7. Acceptance criteria

- `src/server/ai/tools/budgets.ts` is the **single source** for every number in the §1 inventory
  that this spec governs (Dreaming/Coach run caps, Copilot per-message budget, per-tool excerpt cap);
  no surface re-hardcodes them.
- **Dreaming and Coach behavior is byte-unchanged**: the `budget` block sent to the model and the
  proposal-cap string are identical before/after; existing Dreaming/Coach handler tests pass without
  edits to their assertions.
- **Copilot respects `COPILOT_CONTEXT_BUDGET`** within a single user message: a message cannot exceed
  10 tool calls, 250 nodes+edges, or 1000 event rows; over-budget reads return what fits with
  truncation metadata rather than throwing.
- **Graceful degradation is observable**: a unit test drives a Copilot turn past each ceiling and
  asserts truncated-but-valid output (not an error), plus the soft-stop string on the tool-call
  ceiling.
- Tests: budget-constant snapshot test (locks the numbers); read-tool `min(default, remaining)`
  truncation test; Copilot per-message accumulator test (sum across multiple calls in one turn);
  Dreaming/Coach no-behavior-change regression.
