## 1. Problem & scope

P5.8 builds the remaining subject eval fixtures after math + physics. The eval fixtures are the
acid test for the **generalized subject framework**: each fixture is a small corpus of grounded
questions whose answers route through the **real** judge stack (`judgeAnswer()` →
`resolveQuestionJudgeRoute` → `JudgeInvoker`), proving that a subject's `SubjectProfile` —
its `questionKinds`, `judgePolicy.preferredRoutes`, and `judgeCapabilities` — actually drives
correct routing and produces a usable `coarse_outcome`, not just that a JSON file parses.

Current state (readiness brief §P5.8, lines 525–536):

1. **Math is complete** — `src/subjects/math/fixtures/` has `index.ts` (shared + derivation
   schemas) + `data.json` (10 items) + `derivation-data.json` + tests (`index.test.ts`,
   `e2e.smoke.test.ts`, `derivation.e2e.test.ts`). It gates the **exact** and **keyword** routes
   (single_choice / fill_blank) plus the **steps@1** route (derivation).
2. **Physics is complete** — `src/subjects/physics/fixtures/` (`index.ts` + `data.json` 10 items +
   `schema.test.ts` + `e2e.smoke.test.ts`). It gates **exact** (single_choice) plus the
   **unit_dimension** route with `expected_signals` test cases.
3. **Wenyan / English / programming / reading-note are missing entirely** — no
   `src/subjects/{wenyan,english,programming,reading}/fixtures/` directory exists. Wenyan, however,
   already has a **complete validated profile** (`src/subjects/wenyan/profile.ts:1`–`:116`) and a
   curriculum (`src/subjects/wenyan/curriculum.json` — `{ version, domain, knowledge_seeds: [{slug,
   name}] }` with 7 seeds, consumed via `seed.ts` `getCurriculum()`).

**The validation gap this fixture closes.** No fixture yet exercises the **semantic** judge route.
Math gates exact/keyword/steps; physics gates exact/unit_dimension. The semantic route
(`src/server/ai/judges/question-contract.ts:262`–`:289`, `runSemanticJudge`) — the LLM-backed judge
for translation/analysis/open-interpretation answers — is **registered, runnable, but never gated by
a subject fixture**. Wenyan is the natural first subject to gate it: its profile declares
`translation` + `reading_comprehension` question kinds (`profile.ts:8`–`:14`) and lists `semantic`
in both `judgePolicy.preferredRoutes` (`profile.ts:16`) and `judgeCapabilities` (`profile.ts:115`).
A wenyan fixture is therefore the **first proof that the semantic route resolves and runs end-to-end
for a real subject**, with the LLM stubbed for determinism (§5).

**Scope.** Backend-only **test layer**: one new subject fixtures directory
(`src/subjects/wenyan/fixtures/`) mirroring the math/physics template — `index.ts` (Zod schema +
`loadWenyanFixtures()`), `data.json` (~10–12 items), `index.test.ts` (structure), `e2e.smoke.test.ts`
(real judge routing, LLM stubbed for the semantic route). **No new production schema, no migration,
no new UI, no new judge capability, no profile change** — the wenyan profile and all three judge
routes (`exact` / `keyword` / `semantic`) already exist and are registered
(`src/core/capability/judges/index.ts:1`–`:44`). ~1–2 days (brief line 571–573).

## 3. Fixture structure (mirrors math/physics)

New directory `src/subjects/wenyan/fixtures/`, mirroring `src/subjects/physics/fixtures/`:

| file | kind | content |
|---|---|---|
| `index.ts` | create | `WenyanFixtureItemSchema` (Zod) + `WenyanFixtureFileSchema` + `loadWenyanFixtures()`. Subject-local schema; does NOT touch `src/core/schema/*` (LD-6, mirrors `physics/fixtures/index.ts:4`–`:8`). |
| `data.json` | create | `{ version, subject_id:'wenyan', items:[…] }` — ~10–12 items (§4). |
| `index.test.ts` | create | Structure validation (unit partition): parse, count, per-kind counts, ref uniqueness (mirrors `physics/fixtures/schema.test.ts:1`–`:50`). |
| `e2e.smoke.test.ts` | create | DB partition: seed to `question`, `judgeAnswer()` per representative item, assert `route` + `coarse_outcome`; semantic items use the `runTaskFn` stub (§5). Mirrors `physics/fixtures/e2e.smoke.test.ts`. NOT added to `fastTestInclude` (falls through to DB partition, like math/physics e2e.smoke). |
| `vitest.shared.ts` | modify | Add one line to `fastTestInclude`: `'src/subjects/wenyan/fixtures/index.test.ts'`, mirroring `:103`–`:106`. Required because the allowlist is explicit paths, not a glob (F-5). |

**`WenyanFixtureItemSchema` (the item schema).** Mirrors the shared math/physics shape
(`ref` / `kind` / `prompt_md` / `reference_md` / `difficulty` / `knowledge_hint`), with `choices_md`
optional (for single_choice, F-1) and `rubric_json` optional (for semantic-route items'
`required_points`, F-3). NO physics-style `expected_signals` and NO math-style derivation extension —
wenyan does not use the unit_dimension or steps judge (open question OQ-2 below resolved: base kinds
only).

```ts
// src/subjects/wenyan/fixtures/index.ts
import { Rubric } from '@/core/schema/business';   // reused like math/fixtures/index.ts:1
import { z } from 'zod';
import fixtureData from './data.json' with { type: 'json' };

export const WenyanFixtureItemSchema = z.object({
  ref: z.string().min(1),
  // 'fill_blank' is a canonical QuestionKind (business.ts:19); it routes 'keyword'
  // ONLY when rubric_json.keywords is non-empty (question-contract.ts:146), else
  // 'exact'. Required for the H1/AC-5 keyword-route item.
  kind: z.enum([
    'single_choice',
    'translation',
    'reading_comprehension',
    'short_answer',
    'fill_blank',
  ]),
  prompt_md: z.string().min(1),
  choices_md: z.array(z.string().min(1)).optional(),   // F-1: present for single_choice
  reference_md: z.string().min(1),
  rubric_json: Rubric.optional(),                       // F-3: required_points for semantic items
  difficulty: z.number().int().min(1).max(5),
  knowledge_hint: z.string().min(1),                    // maps to curriculum.json seed (F-6)
});
export type WenyanFixtureItem = z.infer<typeof WenyanFixtureItemSchema>;

export const WenyanFixtureFileSchema = z.object({
  version: z.string(),
  subject_id: z.literal('wenyan'),
  items: z.array(WenyanFixtureItemSchema).min(10).max(12),
});

export function loadWenyanFixtures(): WenyanFixtureItem[] {
  return WenyanFixtureFileSchema.parse(fixtureData).items;
}
```

The `kind` enum lists the wenyan profile's own kind strings (`profile.ts:8`–`:14`) — `single_choice` /
`translation` / `reading_comprehension` / `short_answer` — PLUS `fill_blank`, which is a **canonical
`QuestionKind`** (`business.ts:19`), not a wenyan profile kind, included solely to gate the keyword
route (H1/AC-5; routes `keyword` iff `rubric_json.keywords` is non-empty, `question-contract.ts:146`).
The judge contract bridges profile-vs-canonical kinds at routing time (F-1, F-2). `multiple_choice` is
in the profile but is **not** seeded in Phase 1 (single_choice covers the exact route; multi-select
adds no new route — §9 OQ).

## 4. Fixture content plan (~10–12 items)

Grounded in canonical classical-Chinese texts (e.g. 《论语》《孟子》《岳阳楼记》《出师表》《六国论》);
each item names a `knowledge_hint` from `curriculum.json`'s `knowledge_seeds[].name` (`:4`–`:12`).
Route per kind is the
**real** route `resolveQuestionJudgeRoute` produces (verified against `question-contract.ts`):

| kind | count | ref prefix | judge route | why that route (anchor) |
|---|---|---|---|---|
| `single_choice` | ~5 | `wenyan-choice-` | **exact** | has `choices_md` → structural short-circuit (`question-contract.ts:130`–`:131`). Tests 实词/虚词/句式/文学常识 objective recall. |
| `translation` | 3–4 | `wenyan-trans-` | **semantic** | `translation` ∈ `QuestionKind` → `:155`–`:156`, wenyan prefers semantic (`profile.ts:16`). Tests 翻译 (古文→现代文). |
| `reading_comprehension` | 2–3 | `wenyan-read-` | **semantic** | not in `QuestionKind` enum → fallback `short_answer` → `:141`,`:155`–`:156`. Tests 阅读理解 / 赏析 / 开放解释. |
| `fill_blank` (optional H1) | 0–1 | `wenyan-short-` | **keyword** (rubric has `keywords`) | `fill_blank` + non-empty `rubric_json.keywords` → `:146` (keyword); empty → exact. A plain `short_answer` would route semantic (`:155`–`:156`), so `fill_blank` is the kind that reaches keyword — see H1 below. |

This maps directly onto the wenyan profile's `judgeCapabilities: ['exact', 'keyword', 'semantic']`
(`profile.ts:115`) and `judgePolicy` notes (`profile.ts:17`–`:21`: 客观题→exact; 术语/要点题→keyword;
翻译/赏析/开放→semantic). Target split: **5 single_choice + 3 translation + 2 reading_comprehension = 10**
(minimum), expandable to 12 by adding a 4th translation, a 3rd reading_comprehension, or the optional
H1 `fill_blank` keyword item.

- **H1 — keyword route (optional, recommended).** `resolveQuestionJudgeRoute` only routes to
  `keyword` for `fill_blank` (`question-contract.ts:146`) — `short_answer` goes to `semantic`
  (`:155`–`:156`), so a plain `short_answer` would NOT exercise keyword. To gate `keyword` (the
  third declared capability) the fixture can either (a) add a `fill_blank`-kind item with a
  `keywords` rubric (canonical kind, routes keyword like math `fill_blank`,
  `math/fixtures/e2e.smoke.test.ts:128`–`:139`), or (b) accept that Phase 1 gates only exact +
  semantic and defer keyword to a follow-up. **Recommendation:** add **one** `fill_blank` item
  (术语填空, e.g. 虚词功能) so all three wenyan `judgeCapabilities` are gated — making the count
  **11** (5 choice + 1 fill_blank + 3 trans + 2 read). Final inclusion is a writing-plans decision
  (§9 OQ-1).

- **Grounding discipline.** Each `reference_md` is the canonical answer; each semantic item's
  `rubric_json.required_points` lists the 2–3 scoring points the stubbed judge matches against
  (F-3). Translations use a well-known sentence whose modern-Chinese rendering is unambiguous (so
  the reference is defensible); reading items pose a short 赏析/解释 with bounded required points.

## 5. e2e.smoke design — real judge, LLM stubbed for semantic

The e2e mirrors `physics/fixtures/e2e.smoke.test.ts` structure: `beforeAll(() => { db = testDb() })`,
`beforeEach(async () => { await resetDb(); await seedWenyanFixtures(db) })`, a `toJudgeRow()` adapter,
and per-item `judgeAnswer()` calls asserting `route` + `coarse_outcome`.

**Seeding (`seedWenyanFixtures`).** Mirrors `physics/fixtures/e2e.smoke.test.ts:24`–`:64`: insert one
`knowledge` root (`domain:'wenyan'`), then loop `loadWenyanFixtures()` inserting each into `question`.
The field list below is **illustrative, not exhaustive** — `id:'q-smoke-${item.ref}'`, `kind`,
`prompt_md`, `reference_md`, `choices_md: item.choices_md ?? null`,
`rubric_json: item.rubric_json ?? null` (F-3 — needed so the semantic input has `required_points`),
`knowledge_ids:[ROOT_KID]`, `difficulty`, `source:'wenyan_fixture_smoke'`, `variant_depth:0`,
`figures:[]`, `image_refs:[]`, `structured:null`, `metadata:{ fixture_ref, knowledge_hint }`,
`version:0`. Columns with NOT-NULL DB defaults — e.g. `layout_quality`
(`schema.ts:122`, `.notNull().default('structured')`) — can be **omitted** and left to the default,
exactly as the physics seed does; only the routing-relevant fields above must be set explicitly.

**Exact route (single_choice) — no stub.** Identical to physics single_choice
(`physics/fixtures/e2e.smoke.test.ts:106`–`:136`):

```ts
const { route, result } = await judgeAnswer({ db, question: toJudgeRow(row), answer_md: '<correct>', subjectProfile: wenyanProfile });
expect(route).toBe('exact');
expect(result.coarse_outcome).toBe('correct');   // and an incorrect-answer case → 'incorrect'
```

**Semantic route (translation + reading_comprehension) — LLM STUBBED via `runTaskFn`.** The stub is
the per-call seam (F-4), the exact pattern `derivation.e2e.test.ts:83` uses for steps:

```ts
const mockRunTaskFn = async () => ({
  text: JSON.stringify({
    score: 0.9,
    coarse_outcome: 'correct',
    confidence: 0.9,
    feedback_md: '译文准确，要点齐全。',
    evidence_json: { matched_points: ['p1', 'p2'], missing_points: [] },
  }),   // matches SemanticJudgeOutput — question-contract.ts:24-34
});

const { route, result } = await judgeAnswer({
  db,
  question: toJudgeRow(row),     // a translation item (no choices_md, kind='translation')
  answer_md: '<student translation>',
  subjectProfile: wenyanProfile,
  runTaskFn: mockRunTaskFn,      // intercepts runSemanticJudge — invoker.ts:124-126
});
expect(route).toBe('semantic');                 // THE routing assertion (F-2)
expect(result.coarse_outcome).toBe('correct');  // normalizeSemanticResult — :217-250
expect(result.capability_ref.id).toBe('semantic');
```

- The stub returns a `correct` shape for a good answer and a `partial`/`incorrect` shape for a weak
  one (a second `it`), proving `normalizeSemanticResult` (`question-contract.ts:217`–`:250`) maps the
  outcome correctly. The reading_comprehension items use the **same** stub seam and assert
  `route === 'semantic'` (proving the `short_answer` fallback path, F-2).
- **No live LLM, ever.** `runTaskFn` short-circuits `defaultRunTaskFn` (`question-contract.ts:252`–
  `:260`, which imports `@/server/ai/runner`), so no `ANTHROPIC_API_KEY` and no Agent SDK call. This
  is the hermetic guarantee — same as the math derivation LLM-path test.
- **Keyword route (if H1's `fill_blank` is included).** No stub; assert `route === 'keyword'` and
  `coarse_outcome` per keyword presence, exactly as `math/fixtures/e2e.smoke.test.ts:128`–`:139`.

**The routing assertion is the deliverable.** The smoke's primary value is asserting that the wenyan
profile drives `resolveQuestionJudgeRoute` to the **correct route per kind** (`exact` for
single_choice, `semantic` for translation/reading_comprehension, optional `keyword` for fill_blank) —
this is the first end-to-end proof the framework generalizes to the semantic-route subject.

## 6. What is explicitly OUT

- **English fixture.** Deferred (LD-2). Blocked on a judge-route decision — grammar Q&A might want
  `exact`/`keyword` while translation/analysis wants `semantic`; needs a profile + route strategy
  first (§9 Q3). No `src/subjects/english/`.
- **Programming fixture.** Deferred (LD-2). Blocked on `code_execution` — there is **no such judge
  capability** (the registry has only `exact`/`keyword`/`semantic`/`steps`/`unit_dimension`,
  `src/core/capability/judges/index.ts:1`–`:44`). Stubbing it would test nothing real. No
  `src/subjects/programming/`.
- **Reading-note fixture.** Deferred AND **semantically unclear** (LD-2). Likely a Living-Note
  artifact **fixture TYPE**, not a subject — needs product-owner clarification (§9 Q1, Q4) before any
  file is created. No `src/subjects/reading/`.
- **New judge capability / `code_execution` / `ai_flexible`.** Out. `ai_flexible` is in wenyan
  `judgePolicy.preferredRoutes` (`profile.ts:16`) but NOT in `judgeCapabilities` (`:115`) and is a
  documented future route (`question-contract.ts:18`–`:22`, `FUTURE_JUDGE_ROUTES`); Phase 1 gates
  only the three runnable wenyan routes.
- **`multiple_choice`, `derivation`, `expected_signals`, unit_dimension/steps.** Not used by wenyan;
  no fixture rows. The wenyan schema is base-kind only (no physics `expected_signals`, no math
  derivation extension).
- **No new production schema, no migration, no new UI, no profile change.** The wenyan profile and
  all three judge routes already exist. The fixture is a subject-local test corpus + two test files;
  `pnpm audit:schema` is unaffected (no `src/db/schema.ts` field added).

## 7. Acceptance criteria

- **AC-1 (structure, LD-3/LD-4).** `data.json` parses against `WenyanFixtureFileSchema`;
  `loadWenyanFixtures()` returns 10–12 items; `index.test.ts` asserts per-kind counts (≥5
  single_choice, ≥3 translation, ≥2 reading_comprehension, optional 1 fill_blank) and global ref
  uniqueness — mirroring `physics/fixtures/schema.test.ts`.
- **AC-2 (exact route).** A single_choice item answered correctly → `route:'exact'`,
  `coarse_outcome:'correct'`, `capability_ref.id:'exact'`; a wrong answer → `'incorrect'` (no LLM
  call).
- **AC-3 (semantic route — THE frontier, LD-5).** A translation item routed via `judgeAnswer` with a
