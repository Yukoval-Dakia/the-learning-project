> Status: **design approved 2026-05-31 · revised 2026-05-31 (grounded-lean mechanism)** (feature-level
> decision). Implements roadmap **P5.6** ("Copilot suggestion semantics: proactive vs corrective vs
> accept_suggestion"). A **Layer-8** Copilot/Coach quality layer. Refs
> [YUK-143](https://linear.app/yukoval-studios/issue/YUK-143) (P5 chain) +
> ADR-0011 v2 §2.1 (the `suggestion_kind` discriminator this spec operationalizes).
> Source: 2026-05-31 user decision (KPI excludes corrective; field-scope = audit-derived;
> TeachingDrawer can be corrective after N failures; Mechanism = Option B explicit field) +
> 2026-05-30 P5 readiness brief §P5.6.
> Build 排期: writing-plans next; backend-only, no new UI (`SuggestionKindTag` already shipped).
>
> **Revision note (2026-05-31, grounded-lean).** The first draft's emit mechanism ("Coach prime +
> deterministic post-process fallback on soft-fail / `result_count===0`") failed adversarial review
> against the real architecture and is **dropped**. Grounding: (1) the post-process fallback had **no
> wiring site** — `runCoach` has no single "proposal-write path" to coerce in, and the ToolContext
> last-turn-outcome threading it implied is **infeasible** (the SDK loop is per-turn stateless;
> threading a turn outcome into `ToolContext` is unresolvable causality ambiguity — rejected, §2). (2)
> The `result_count===0` soft-fail trigger is a **phantom signal**: the MCP bridge computes
> `outcome = errorReason ? 'failure' : 'success'` (`mcp-bridge.ts:287`) and **never populates
> `result_count`**; a soft-fail empty read returns `outcome:'success'` (the `DomainTool` contract:
> soft-fail returns a valid empty `Output`, `types.ts:60`–`:61`), and `Output` shapes are too
> heterogeneous to introspect a uniform count. So there is **no reliable deterministic
> soft-fail→corrective signal** to build on. (3) The §6 chip writer, the TeachingDrawer
> `active_question`, and the chip-accept KPI table all referenced **data + write paths that do not
> exist** in the codebase. The **revised lean mechanism** keeps the parts that survived review (the §3
> per-kind audit, the §5.1 single-choke-point gate at `recordProposalDecisionSignal`, the §7 ND-style
> invariants, the §2 decision table) and replaces the rest with grounded, no-migration wiring:
> corrective comes ONLY from (a) the `variant_question` producer's deterministic structural floor and
> (b) **explicit model labeling** via a new optional `suggestion_kind?` arg on the 4 agent-callable
> propose tools (the model sets it when it reasons a proposal is a failure-retry); the chip writer is a
> **new `POST …/accept-chip` endpoint**; `active_question` is returned **in the turn/GET response**
> (no DB column); the chip-accept KPI is an **event-table reader** (no new table). **No migration.**

## 1. Problem & scope (Layer-8 framing)

ADR-0011 v2 (`docs/adr/0011-tool-use-and-edge-event-paths.md:113`) shipped a first-class
`AcceptSuggestionChip` event whose `payload.suggestion_kind` discriminates **`proactive`** ("agent
在成功 explain 后主动提议下一步动作", source = `explain` event, outcome=success) from
**`corrective`** ("agent 在失败 / 0 result 后提议修正参数", source = `tool_use` event,
outcome=failure 或 result_count=0). §2.1 fixed the *intent* and one KPI consequence verbatim —
"`corrective` 不计入'用户接受 AI 建议'KPI（避免修正动作虚增接受率）". What it left **operationally
underspecified** — and where the first draft's mechanism broke against the real code — is the entire
mechanism:

1. **The discriminator is shipped but never written.** `SuggestionKind` + `AcceptSuggestionChip`
   are defined (`src/core/schema/event/known.ts:331`, `:334`) with a **REQUIRED**
   `payload.suggestion_kind`, but a repo-wide search finds **no** `writeEvent(... action:
   'accept_suggestion' ...)` and **no** `writeAcceptSuggestionChip()` anywhere in `src/server/`.
   The chip schema is a dead letterbox: the UI `SuggestionKindTag` (`src/ui/primitives/SuggestionKindTag.tsx:3`)
   can render the "修正" badge, but nothing on the write path ever sets the kind.

2. **The real accept path is `action: 'rate'`, and it has no `suggestion_kind`.** Proposal accepts
   funnel through `acceptAiProposal()` (`src/server/proposals/actions.ts:480`) → `dispatchAccept()`
   (`:523`), which per kind writes a `rate` event (`rating: 'accept'`, e.g. `:406`) and calls
   `recordProposalDecisionSignal(db, proposal, 'accept', user_note)` (`:540`, `:553`, `:585`, `:669`,
   `:747`, …). The proposal payload (`AiProposalPayloadT`, `proposal.ts:110`) and the inbox row
   (`ProposalInboxRow`, `inbox.ts`) carry **no** `suggestion_kind`. So today there is **no way** to
   tell a corrective accept apart from a proactive one at accept time.

3. **Every corrective accept silently inflates the KPI that P5.4-L2 just shipped.**
   `recordProposalDecisionSignal` (`signals.ts:169`) is the incremental writer of
   `proposal_signals.accept_count` (the UPSERT at `signals.ts:226`), and `rebuildProposalDecisionSignal`
   (`:300`) re-derives the same count from the `rate` event log on replay/reconcile — the two writers
   that must agree (§5.1). That `accept_count` / `acceptance_rate` is exactly what P5.4-L2's adaptive
   bias reads: `getProposalFeedbackDigest`
   (`adaptive-bias.ts`) feeds per-`(kind, relation)` `acceptance_rate` into Dreaming / Coach /
   Copilot prompts (Facet A), and `resolveEdgeGateBump` sums it to tighten the L1 gate (Facet B). A
   *corrective* accept ("yes, redo this question after I failed it") is a **retry-on-failure
   acknowledgment**, not evidence the agent's *proactive proposal quality* is good. Counting it
   pollutes the very accept-learned signal P5.4-L2 was built to keep clean.

4. **TeachingDrawer has hardcoded suggestion_kind, but no failure-driven corrective path — and no
   live failure data.** `TeachingDrawer.tsx:73` hardcodes a `SUGGESTIONS` array mixing kinds
   (`'再讲一遍'` = `corrective`, `'出题考我'` / `'我懂了'` = `proactive`). These are static UI chips, not
   agent-emitted, and the drawer has **no** logic to flip a chip to corrective after the learner
   repeatedly fails — the "after N failed attempts → revisit-prerequisite / redo" behavior the user
   wants does not exist. **Worse, the drawer has no failure signal to drive it:** the teaching session
   does **not track the active question id** and the drawer holds no `questionId`, so it cannot look up
   `attempt_counts.failure` at all. This is a real data-flow gap, not just a missing branch.

5. **There is no chip writer and no chip-accept KPI.** The first draft assumed an existing
   `AcceptSuggestionChip` route writer and an existing "用户接受 AI 建议" chip-accept KPI reader to add
   a filter to. Neither exists. A repo search finds **no** `writeEvent(... action:'accept_suggestion'
   ...)` anywhere, and there is no metric/query that aggregates `accept_suggestion` events. So P5.6
   must *create* the chip writer (the FIRST writer) and *create* the chip-accept KPI reader, not patch
   existing ones.

**Layer-8 stake.** The Phase-3 Global Copilot Orchestrator's "proactive partner" intent depends on
unambiguous suggestion routing: a proactive next-step ("出 3 道变式") and a corrective retry ("换个
参数再查" / "after 3 fails, revisit the prerequisite") are different acts with different KPI
meanings. P5.6 makes that distinction first-class on the **write** path (set at emit — deterministically
for the one structural corrective kind, and by **explicit model labeling** for the rest — not inferred
at read) and wires the KPI exclusion at the two `proposal_signals.accept_count` writers (incremental +
reconcile) plus the new event-table chip-accept reader, so the P5.4-L2 accept-learned bias keeps
measuring what it claims to measure.

**Scope.** Backend-only. Additive optional field on the proposal payload (Mechanism = Option B);
emit semantics at the `variant_question` producer (deterministic floor) + an optional tool arg the
Coach/Copilot model sets explicitly; KPI exclusion at `recordProposalDecisionSignal` (incremental
**and** rebuild paths); a NEW `accept-chip` route writer + an event-table chip-accept KPI reader;
`active_question` surfaced in the teaching-session turn/GET responses. No new schema table, no new
event action, **no DB column, no migration**, no new UI (the badge already renders). The
`AiProposalKind` enum (`proposal.ts:4`) is unchanged.

## 2. Decision summary

P5.6 = **make `suggestion_kind` a first-class, write-set, optional field on the proposal payload,
read it at accept, and branch corrective accepts OUT of the accept-learned KPI while still recording
them as full events.** Three user-locked decisions anchor this:

| # | Locked decision | Where it lands |
|---|---|---|
| **LD-1** | **KPI excludes corrective — fully (both numerator and denominator).** A corrective decision is still a recorded event (the `rate` event is written exactly as today), but it must NOT feed the P5.4-L2 accept-learned signal at all: a corrective **accept** does not increment `accept_count`, AND a corrective **dismiss** does not increment `dismiss_count`. Since `acceptance_rate = accept_count/(accept_count+dismiss_count)`, excluding only accepts would let a dismissed corrective edge depress the cell rate and tighten the L1 gate via `resolveEdgeGateBump` — re-introducing, in the denominator, exactly the pollution LD-1 removes (re-critique issue 4). So corrective is simply **not a KPI signal**, on either side. | §5.1 — gate at BOTH `accept_count`/`dismiss_count` writers: the incremental `recordProposalDecisionSignal` (`signals.ts:169`) and the replay `rebuildProposalDecisionSignal` (`:300`); the `rate` event still writes, and a corrective **dismiss still persists its `cooldown_until`** (the gate skips KPI *counting*, not the cooldown write). |
| **LD-2** | **`suggestion_kind` field scope = audit-derived, NOT assumed.** §3 audits all 14 `AiProposalKind`s by semantics. Outcome: **only `variant_question` is structurally corrective** (it is emitted ONLY after a failed attempt). The brief's straw-man (`defer` / `plan_adjustment` / `relearn`) is **revised**: those are all proactive maintenance, never failure-retries. The field is therefore **optional on all kinds**, REQUIRED-corrective on none of the *proposal* kinds — but the **TeachingDrawer chip** (LD-3, not a proposal kind) is the live corrective producer. | §3 — per-kind audit table + recommended field-scope set. |
| **LD-3** | **TeachingDrawer suggestions CAN be corrective.** When the active question's cumulative `failure` attempt total reaches **N = 3** (signal = `lifecycle.attempt_counts.failure`, schema `context-readers.ts:505`, computed/assigned `:623`), the teaching-drawer's redo / revisit-prerequisite chip is emitted as `corrective` and routed through the SAME `suggestion_kind` + KPI-exclusion path. (Note: `attempt_counts` are **cumulative totals**, not consecutive — §4.3.) | §4.3 — failed-attempt trigger; §5.2 — chip-accept exclusion path; §6 — chip→accept mapping (the new accept-chip endpoint). |

**Mechanism (revised, lean) = Option B (explicit field), set deterministically at exactly one
producer + by explicit model labeling everywhere else.**
- **Option B (explicit optional field, set at write / read at accept).** Add an optional
  `suggestion_kind?: SuggestionKind` to `BaseProposal` (`proposal.ts:47`) so every kind inherits it
  via the discriminated union (`:110`). The emitter sets it at write; the accept handler reads it
  off `proposal.payload`. No event-chain inference at accept time (rejected Option A), no new event
  action (rejected Option C — would need an ADR-0011 erratum to add `accept_correction`).
- **Where corrective comes from (the two sources — there is no deterministic post-process fallback).**
  1. **Structural floor (deterministic).** The `variant_question` producer
     (`producers.ts:32`–`:65`) **hard-sets `suggestion_kind:'corrective'`** in its `writeAiProposal`
     payload. This is the only fully-deterministic corrective: the producer only ever fires after a
     failed attempt (it requires `source_attempt_event_id` + `source_question_id`, §3 / SK-3).
  2. **Explicit model labeling (the agent-driven source).** The 4 agent-callable propose
     `DomainTool`s — `propose_knowledge_edge` (`proposal-tools.ts:214` input schema),
     `propose_knowledge_mutation` (`:493`), `propose_record_links` (`:1063`),
     `propose_record_promotion` (`:1162`) — gain an **optional `suggestion_kind?: SuggestionKind`
     input arg**. The MODEL sets it explicitly when it reasons a proposal repairs a failure it itself
     observed; `execute()` threads `input.suggestion_kind ?? 'proactive'` into the proposal payload.
     The Coach/Copilot prompts are primed to set it (§4.1/§4.2), but there is **no deterministic
     coercion** — the structural floor (variant_question) is the only deterministic corrective, and it
     is sufficient because it is the one kind that is *structurally* a failure-retry.
- **REJECTED: deterministic post-process fallback on soft-fail / `result_count===0`.** The first
  draft proposed coercing `corrective` in "`runCoach`'s proposal-write path" on a soft-failed /
  zero-result tool turn. **This is dropped as ungrounded.** (a) There is **no single proposal-write
  path** in `runCoach` to coerce in. (b) The signal it keyed on is phantom: `mcp-bridge.ts:287`
  computes `outcome = errorReason ? 'failure' : 'success'` and **never populates `result_count`**; a
  soft-fail empty read returns `outcome:'success'` per the `DomainTool` soft-fail contract
  (`types.ts:60`–`:61`, "Soft-fail (empty result) returns a valid Output; hard-fail throws"), and
  `Output` shapes are too heterogeneous to introspect a uniform count. (c) Threading a turn outcome
  into `ToolContext` to recover the trigger is **infeasible** — the SDK loop is per-turn stateless, so
  a tool's `execute()` cannot reliably attribute "the previous turn soft-failed" to "this proposal is
  its retry" without unresolvable causality ambiguity. The model-labeling arg replaces this with an
  honest, auditable source.
- **Flow stays Layer-8.** Coach / Copilot already call `writeAiProposal` via the producers /
  `proposal-tools.ts` `execute()` path; the field rides that existing path (producer hard-set for
  `variant_question`, input arg for the 4 propose tools). No new agent write surface.

## 3. `suggestion_kind` field design — per-kind audit + field-scope set

LD-2 forbids assuming the scope. The audit below classifies **every** `AiProposalKind`
(`proposal.ts:4`) by *its emitter's trigger semantics* — is it ever fired by a failure / zero-result
handler (corrective), or always by a proactive next-step / maintenance signal?

### 3.1 Per-kind audit table

### 3.2 Recommended field-scope set

- **Schema:** `suggestion_kind?: SuggestionKind` is **OPTIONAL on `BaseProposal`** (SK-1), so all 14
  kinds carry it uniformly. No kind makes it required — a discriminated-union `superRefine` or a
  per-branch required field would (a) break every existing producer & fixture and (b) contradict the
  audit: 13 of 14 kinds are *always* proactive, so a default-to-proactive (§7) is the correct, not
  lax, contract.
- **Producers / tools that SET it:**
  - `variant_question` → producer hard-sets `suggestion_kind:'corrective'` (SK-3, the only
    always-corrective proposal kind; `producers.ts:32`–`:65`).
  - The 4 agent-callable propose tools — `propose_knowledge_edge`, `propose_knowledge_mutation`,
    `propose_record_links`, `propose_record_promotion` — gain an **optional `suggestion_kind?` input
    arg** the model fills explicitly; `execute()` threads `input.suggestion_kind ?? 'proactive'` into
    the payload (§4.1/§4.2). For these, corrective is **model-labeled** (the agent declares "this
    repairs a failure I observed"), not deterministically coerced. Default (model omits) = proactive.
  - The remaining proposal kinds (maintenance / appeal / goal-scope producers in `producers.ts`) →
    **leave it unset** at the producer (default = proactive, §7). They are audited always-proactive
    (§3.1), so no producer change is needed.
- **Outside the proposal payload:** the TeachingDrawer chip (LD-3 / §4.3) carries `suggestion_kind`
  on the `AcceptSuggestionChip` event directly, not through a proposal — it is a chip-accept, not a
  proposal-accept (§6). It is the live corrective producer for the human-facing surface, written by
  the new `accept-chip` route (§5.2 / §6).

> **Net field-scope:** optional on all 14 proposal kinds; deterministically hard-corrective at exactly
> one producer (`variant_question`); model-labeled corrective via the optional tool arg on the 4
> propose tools; and a separate chip-level corrective on the TeachingDrawer path. The brief's
> `defer`/`plan_adjustment`/`relearn` straw-man is **rejected by the audit** — those are proactive
> maintenance. There is **no** emitter-context (soft-fail) coercion (the discarded fallback, §2).

## 4. Emit semantics — exact trigger rules per emitter

The discriminator follows ADR-0011 §2.1 verbatim:

> | `suggestion_kind` | 上下文 | source_event 形态 |
> |---|---|---|
> | `proactive` | agent 在成功 explain 后主动提议下一步动作 | source = `explain` event, outcome=success |
> | `corrective` | agent 在失败 / 0 result 后提议修正参数 | source = `tool_use` event, outcome=failure 或 result_count=0 |
> — `docs/adr/0011-tool-use-and-edge-event-paths.md:117`

### 4.1 Coach (`coach_daily.ts` / `runCoach` path) — explicit model labeling, no fallback

Coach emits via the MCP bridge (`coach_daily.ts:225` `buildMcpServer` with a `beforeExecute` hook
that counts proposals, `:235`/`:257`); the bridge mirrors tool_use with `outcome: errorReason ?
'failure' : 'success'` (`mcp-bridge.ts:287`) and **never populates `result_count`**.

1. **Prime (the only mechanism).** Extend `COACH_DAILY_OBJECTIVE` / `COACH_WEEKLY_OBJECTIVE`
   (`coach_daily.ts:58`/`:59`) with a clause instructing the model to set the propose-tool
   `suggestion_kind` arg on each proposal: `proactive` when proposing a next step off a successful
   read (the default); `corrective` only when the proposal repairs a failure the model itself just
   observed. (Appended alongside the existing `COACH_PROPOSAL_FEEDBACK_GUIDANCE`, `:56`.) The model
   sets the optional `suggestion_kind?` input arg on `propose_knowledge_edge` /
   `propose_knowledge_mutation`; `execute()` writes `input.suggestion_kind ?? 'proactive'`.
2. **Zero-result is NOT a corrective trigger (explicit model instruction).** The prompt must state: a
   read that returns **zero results is a legitimate `outcome:'success'`** (no error) — it means "I
   looked and found nothing", not "I failed". The model must NOT label a proposal `corrective` just
   because an upstream read came back empty. Only a genuine repair-of-an-observed-failure is
   corrective.

> **Why no deterministic fallback (the discarded §2 mechanism).** The first draft coerced `corrective`
> in "`runCoach`'s proposal-write path" on a soft-fail / `result_count===0` turn. There is no such
> single write path, the `result_count` field the trigger keyed on is never populated by the bridge
> (`mcp-bridge.ts:287`), a soft-fail empty read is `outcome:'success'` (the `DomainTool` contract,
> `types.ts:60`), and the per-turn-stateless SDK loop makes threading a turn outcome into the tool's
> `ToolContext` infeasible. So there is no auditable deterministic soft-fail→corrective signal to
> build a floor on. Coach's actable proposal kinds (`COACH_ACTABLE_KINDS`, `coach_daily.ts:66`) are
> audited proactive (§3); in steady state Coach emits proactive, and corrective only when the model
> explicitly labels a genuine repair.

### 4.2 Copilot (`copilot/chat.ts` `runCopilotChat`) — same explicit model labeling

Copilot proposes `knowledge_edge` / `record_links` / `record_promotion` (all proactive by default,
§3) via the DomainTool path (`proposal-tools.ts`), and writes `experimental:copilot_user_ask` /
`experimental:copilot_chip_trigger` events (`chat.ts:182`/`:199`).

1. **Prime.** The `CopilotTask` system prompt (`task-prompts.ts`) gets the same clause: set the
   propose-tool `suggestion_kind` arg per proposal; proactive for next-step suggestions (default),
   corrective only when the proposal repairs a failure the model observed within the message.
2. **Same zero-result instruction.** As §4.1.2 — a zero-result read is `outcome:'success'`, not a
   corrective trigger. There is no deterministic fallback; the model labels the optional input arg on
   `propose_knowledge_edge` / `propose_record_links` / `propose_record_promotion`, and `execute()`
   writes `input.suggestion_kind ?? 'proactive'`. Copilot's per-message agent turn is bounded by the
   existing `ContextBudgetTracker` (`chat.ts:217`); the arg is a pure per-write field, no new loop
   state.

### 4.3 TeachingDrawer — failed-attempt corrective trigger (LD-3)

Today the drawer's `SUGGESTIONS` array (`TeachingDrawer.tsx:73`) is static, with `'再讲一遍'`
hardcoded `corrective` and the rest `proactive`. P5.6 makes the **redo / revisit-prerequisite** chip
*dynamically* corrective off the live failure signal. **The blocker the revision resolves: the drawer
has no failure data today** — the teaching session does not track the active question id and the
drawer holds no `questionId`, so it cannot look up `attempt_counts.failure`. The lean fix (NO DB
column, NO migration) is to surface that data in the existing turn/GET responses.

- **Data flow (lean fix — no `learning_session.active_question_id` column).** Return
  `active_question_id` **and** its `attempt_counts` in the `POST
  /api/teaching-sessions/[id]/turn` response **and** the `GET /api/teaching-sessions/[id]` response
  payload; the drawer stores them in component state. `attempt_counts` is read via the existing
  `getQuestionContext` aggregate (`context-readers.ts:505` schema; built at `:599`–`:604` and assigned
  at `:623`). **Explicitly rejected:** adding a `learning_session.active_question_id` DB column — it is
  heavier than needed (a migration + a write path for a value the response can already carry), and the
  drawer only needs the value for the current turn, which the response already produces.
- **Signal.** `attempt_counts: { success, partial, failure }`; `AttemptOnQuestion.outcome` includes
  `'failure'` (`known.ts:25`). **These are CUMULATIVE totals** — `getQuestionContext` iterates over
  *all* attempts in the question timeline and increments per-outcome (`context-readers.ts:599`–`:604`),
  so `attempt_counts.failure` is the total failure count over the question's history, NOT a
  consecutive-failure streak.
- **Trigger (N = 3, total).** When the active question's cumulative `attempt_counts.failure >= 3`, the
  drawer surfaces (or re-labels) a **redo / revisit-prerequisite** chip with
  `suggestion_kind:'corrective'`. Below 3, the chip stays proactive ("出题考我") or is not surfaced as a
  correction.
  - **Why N=3 (total):** three failures is the point where "try again" stops being normal practice and
    becomes "this prerequisite is not landing — revisit it." It is a small, tunable single-source const
    `TEACHING_CORRECTIVE_FAILURE_N` (= 3); revisit after real session logs, mirroring P5.1's "tune
    after ~2 weeks" stance.
  - **Total vs consecutive — pinned.** The trigger uses **total** `failure >= N`, because that is what
    the field provides. A *consecutive*-failure trigger ("still failing now") is **NOT derivable** from
    `attempt_counts` (it is a sum, not a streak). Consecutive-failure is noted as a **future refinement
    requiring a new attempt-stream query** (read the ordered attempt timeline and count the trailing
    failure run) — it is **not claimed now** (§12).
- **Routing.** When the user clicks the corrective chip, the drawer's chip `onClick` calls the new
  `POST /api/teaching-sessions/[id]/accept-chip` endpoint (§5.2 / §6), which writes the
  `AcceptSuggestionChip` event with `suggestion_kind:'corrective'`; that flows through the chip-accept
  KPI exclusion (§5.2). The proactive chips write the chip event with `suggestion_kind:'proactive'`
  (which DOES count). For accept-chips this `onClick` **replaces** the chat-turn send (it is a chip
  accept, not a teaching turn).

> The drawer chips are NOT proposals — they are chip-accepts (§6). The corrective KPI exclusion
> therefore applies to BOTH the proposal-accept path (§5.1) and the chip-accept path (§5.2), unified by
> the `suggestion_kind === 'corrective'` test.

## 5. Accept + KPI exclusion — exact call-sites

LD-1: a corrective accept is recorded as an event but excluded from `acceptance_rate` /
`proposal_signals`. There are two accept surfaces; both gate on `suggestion_kind === 'corrective'`.

**Gate predicate — full exclusion across the whole decision set.** `decisionFromRate` (the rebuild's
classifier) treats `accept | reverse | change_type` as the accept-family (it increments `acceptCount`
for all three, `:347`–`:357`) and `dismiss` as the denominator side. The gate (both the incremental
early-return and the rebuild skip) reads as **"any decision on a corrective proposal is excluded from
the KPI counts"** — accept-family → no `accept_count`, dismiss → no `dismiss_count` — not just literal
`'accept'`. This is the full-exclusion fix (re-critique issue 4): excluding only accepts would let a
dismissed corrective depress the cell `acceptance_rate` denominator and tighten the L1 gate, mirroring
the pollution LD-1 removes. In practice edges are audited proactive (§3), so the corrective set is
mostly `variant_question` + model-labeled record/mutation kinds — but the predicate must cover the
whole decision set so neither a corrective `reverse`/`change_type` credit nor a corrective `dismiss`
debit can sneak into the rate. (A corrective dismiss still writes `cooldown_until` — only the count is
skipped.)

Because `recordProposalDecisionSignal` (incremental) and `rebuildProposalDecisionSignal` (replay) are
the **only** two writers of `proposal_signals.accept_count` / `dismiss_count` (verified: `signals.ts:226`
is the sole incremental UPSERT, and the rebuild is the only absolute recompute), gating BOTH
guarantees a corrective decision (accept OR dismiss) never reaches `acceptance_rate`, hence never reaches
`getProposalFeedbackDigest` / `resolveEdgeGateBump` (P5.4-L2 `adaptive-bias.ts`). The P5.4-L2 signal
stays clean by construction, and a `ensureProposalDecisionSignal` reconcile cannot reintroduce the
inflation.

### 5.2 Chip-accept path (TeachingDrawer / Copilot chip → `AcceptSuggestionChip`)

The `AcceptSuggestionChip` event (`known.ts:334`–`:350`) carries `payload.suggestion_kind` (REQUIRED).
**Neither the writer nor the KPI reader exists today** (§1.5) — P5.6 creates both:

- **Writer (NEW).** `POST /api/teaching-sessions/[id]/accept-chip` (+ a Copilot counterpart) writes
  the `AcceptSuggestionChip` event on chip click (§6). This is the **first** code that ever writes
  `action:'accept_suggestion'`.
- **KPI reader (NEW, event-table — no new table, SK-7).** A reader function queries the event table
  directly: aggregate `action='accept_suggestion'` events `GROUP BY payload->>'suggestion_kind'`,
  filtering `WHERE payload->>'suggestion_kind' <> 'corrective'` for the "用户接受 AI 建议" rate. Do
  **NOT** build a `chip_accept_signals` table — no consumer exists yet (YAGNI / anti-overengineering,
  CLAUDE.md scope discipline). A materialized signals table is a future optimization only if a consumer
  appears (§12).
- If a chip-accept *also* materializes a proposal accept (e.g. a corrective chip that triggers
  `acceptAiProposal` on a `variant_question`), the §5.1 gate already excludes it from
  `proposal_signals` — the two gates compose (a corrective signal is excluded at *whichever* path it
  takes), and there is exactly one logical rule: `suggestion_kind === 'corrective'` ⇒ no KPI credit.

## 6. proposal → AcceptSuggestionChip mapping

This is the bridge the brief flagged as "no proposal → suggestion_kind mapping". The mapping is
**directional and explicit**, not inferred:

| source | what carries `suggestion_kind` | who writes it | counts toward KPI? |
|---|---|---|---|
| Inbox proposal accept (`variant_question`, edges, …) | `proposal.payload.suggestion_kind` (SK-1) | producer (`variant_question` hard-sets corrective; the 4 propose tools take the model-labeled optional arg; the rest default proactive) | proactive: yes (via `proposal_signals`); corrective: **no** (§5.1 gate) |
| TeachingDrawer chip click | `AcceptSuggestionChip.payload.suggestion_kind` (`known.ts:342`) | **NEW `POST …/accept-chip` endpoint**, on click, from the chip's kind (LD-3 / §4.3) | proactive: yes; corrective: **no** (§5.2 reader filter) |
| Copilot chip trigger | `AcceptSuggestionChip.payload.suggestion_kind` (when a chip is accepted) | NEW Copilot accept-chip endpoint | same |

**The NEW chip writer (resolves the dead-letterbox + the brief open-Q "where is the chip written").**
`AcceptSuggestionChip` was never written by any code (§1.5). P5.6 adds a `POST
/api/teaching-sessions/[id]/accept-chip` endpoint (and a Copilot counterpart) — NOT inside
`coach_daily.ts` / `copilot/chat.ts` (those write proposals + ask/chip-trigger events, never the
chip-accept). The drawer's chip `onClick` calls this endpoint instead of the chat-turn send for
accept-chips (§4.3). On click the endpoint:
- **re-validates session ownership + state** (single-user `x-internal-token`, plus the session must be
  active — return **409 if the session has ended**, mirroring the existing teaching-session route
  guards);
- writes the `AcceptSuggestionChip` event with `action:'accept_suggestion'` and `payload`
  `{ suggestion_kind, chip_label, source_event_id }` from the chip's declared kind (static for
  proactive chips; LD-3-computed for the corrective redo chip);
- sets `source_event_id` per **ADR-0011 §2.1**: the agent **`explain`** event for a `proactive` chip,
  the **`tool_use`** event for a `corrective` chip;
- if the chip's `target_tool` materializes a proposal accept (e.g. a corrective `propose_variant`
  chip), the endpoint **also calls `acceptAiProposal`** so §5.1's gate applies to the proposal-signal
  side (the two gates compose; a corrective signal is excluded at whichever path it takes).

So the proposal-payload `suggestion_kind` (write-set at emit) and the chip-event `suggestion_kind`
(write-set at click from the chip's kind) are **two faces of the same discriminator**, and the single
KPI rule (`=== 'corrective'` ⇒ no credit) covers both faces.

## 7. Invariants (additive / ND-style)

- **ND-SK-1 — optional, absence = proactive.** `suggestion_kind` is optional everywhere. A proposal
  or chip with no `suggestion_kind` is treated as `proactive` (counts toward KPI). No producer is
  *forced* to set it; the default is the safe, audit-correct one (§3: 13/14 proposal kinds are always
  proactive).
- **ND-SK-2 — never changes proposal acceptance logic, only KPI attribution.** A corrective
  `suggestion_kind` does NOT change whether a proposal can be accepted, what the accept *does* (the
  edge/variant/etc. is still materialized identically), or any L1 rubric / P5.4-L2 gate. It changes
  **only** whether the accept increments `proposal_signals.accept_count` / `acceptance_rate`. Accept
  side-effects are byte-identical with or without the field.
- **ND-SK-3 — corrective is still a full event.** A corrective accept ALWAYS writes its `rate` event
  (or `AcceptSuggestionChip` event) with the real `rating:'accept'` / outcome — it is excluded from
  the *KPI aggregate*, never from the *event log*. Evidence-first / reversible (CLAUDE.md): you can
  always replay the log and see the corrective accept happened.
- **ND-SK-4 — KPI exclusion is at exactly one logical rule, full (both sides).** `suggestion_kind ===
  'corrective'` ⇒ no KPI count credit on EITHER side — no `accept_count` for an accept-family decision
  AND no `dismiss_count` for a dismiss — enforced at **both** `proposal_signals` count writers (the
  incremental `recordProposalDecisionSignal` early-return / count-skip AND the
  `rebuildProposalDecisionSignal` corrective-`Set` skip, §5.1) and at the single event-table chip-accept
  KPI reader filter (§5.2). No place may bump `accept_count` *or* `dismiss_count` for a corrective; the
  predicate covers the full decision set (`accept | reverse | change_type | dismiss`). A corrective
  dismiss still persists `cooldown_until` (re-surfacing suppression is independent of KPI accounting).
- **ND-SK-5 — additive / cold-start / regression-safe.** Every existing producer that omits the
  field behaves exactly as today (default proactive). The pure pre-P5.6 accept path with no corrective
  proposal is byte-identical. P5.4-L2's `adaptive-bias.ts` reads the same `proposal_signals` shape,
  just with corrective accepts no longer polluting it.

## 8. Acceptance criteria

- **AC-1 (LD-1, the core invariant — full exclusion).** A unit/DB test accepts a `corrective` proposal
  and asserts: (a) the `rate` event with `rating:'accept'` IS written; (b) `proposal_signals.accept_count`
  for that `(kind, cooldown_key)` is **unchanged** (no increment); (c) `getProposalAcceptanceRates` /
  `getProposalFeedbackDigest` therefore exclude it. The same proposal as `proactive` (or field-absent)
  DOES increment `accept_count`.
- **AC-1b (LD-1 denominator — dismiss exclusion + cooldown preserved).** A test **dismisses** a
  `corrective` proposal and asserts: (a) `proposal_signals.dismiss_count` is **unchanged** (the
  denominator is not distorted — re-critique issue 4); (b) `cooldown_until` IS still written (the
  corrective proposal is suppressed from re-surfacing); (c) the resulting `acceptance_rate` for the cell
  is identical to never having seen the corrective decision. A `proactive` dismiss DOES increment
  `dismiss_count` as today.
- **AC-2 (field scope, audit-faithful).** A test asserts `suggestion_kind` is optional on
  `BaseProposal` (parsing a payload without it succeeds and a downstream reader sees `proactive`), and
  that the `variant_question` producer always emits `suggestion_kind:'corrective'`. A
  table/parameterized test pins the §3.1 classification (every kind's expected corrective-possibility)
  so a future producer change that flips a kind is caught.
- **AC-3 (explicit model labeling — no fallback).** A test asserts that a propose tool called with the
  optional `suggestion_kind:'corrective'` input arg writes a `corrective` payload, and that the same
  tool called **without** the arg writes `proactive` (the `input.suggestion_kind ?? 'proactive'`
  default). There is NO soft-fail / `result_count===0` coercion test (that mechanism is dropped, §2);
  a zero-result read does NOT, by itself, produce a corrective proposal.
- **AC-3b (rebuild / reconcile regression — the gate holds on replay).** A DB test accepts a
  `corrective` proposal, then runs `ensureProposalDecisionSignal` (the rebuild path,
  `rebuildProposalDecisionSignal`) for that `(kind, cooldown_key)` and asserts `accept_count` is
  **UNCHANGED** — the rebuild's corrective-proposal `Set` skip (§5.1) reproduces the gated count, so a
  reconcile cannot reintroduce the inflation. A sibling proactive proposal on the same key still
  contributes.
- **AC-4 (TeachingDrawer N=3, LD-3).** A test with cumulative `attempt_counts.failure < 3` asserts the
  redo chip is proactive (or not surfaced as corrective); at `>= 3` asserts the corrective redo /
  revisit-prerequisite chip is surfaced. (Trigger is total failures, not consecutive — §4.3.)
- **AC-5 (accept-chip endpoint + event-table chip-KPI exclusion, §5.2).** A test POSTs to the new
  `/api/teaching-sessions/[id]/accept-chip` endpoint with a `corrective` chip and asserts: the
  `AcceptSuggestionChip` event is written (`action:'accept_suggestion'`, `source_event_id` set per
  §2.1); the event-table chip-accept KPI reader (GROUP BY `suggestion_kind`, filter `<> 'corrective'`)
  **excludes** it; a `proactive` chip-accept counts. A second test asserts the endpoint returns **409**
  when the session has ended.
- **AC-6 (ND-SK-2 — accept side-effects unchanged).** A test accepts the SAME proposal as proactive
  vs corrective and asserts the materialized result (edge row / variant question / etc.) is identical;
  only `proposal_signals` differs.
- **AC-7 (no migration).** No DB migration; `proposal_signals` columns unchanged; no new table;
  `pnpm audit:schema` passes (`suggestion_kind` is a payload field on an existing event + an optional
  tool input; `active_question` rides the turn/GET response; the chip-KPI is an event-table reader, not
  a column/table). `AiProposalKind` enum unchanged. **TRUE by the lean choices** (no
  `learning_session.active_question_id` column, no `chip_accept_signals` table).
- **AC-8 (regression).** The full existing accept/signal/P5.4-L2 suites pass unedited for proactive /
  field-absent proposals (ND-SK-5), including the existing `rebuildProposalDecisionSignal` tests for
  all-proactive keys.

## 11. ADR-0011 erratum note

ADR-0011 v2 §2.1 (`:113`) signed off the `suggestion_kind` discriminator and the KPI consequence
("`corrective` 不计入接受率") but documented **no handler spec / write path / KPI choke point** — and
the `AcceptSuggestionChip` event it defined was never written by any code (§1.1 / §1.5). P5.6 is the
**operationalizing erratum**:

1. **`suggestion_kind` is realized as an OPTIONAL field on `BaseProposal`** (`proposal.ts:47`) and as
   an optional input arg on the 4 agent-callable propose tools, not only on `AcceptSuggestionChip`.
   The discriminator now rides the proposal payload (the actual Coach/Copilot output) in addition to
   the chip event. (Option B, not a new `accept_correction` action — §2.1's
   single-`accept_suggestion`-action decision is preserved.)
2. **The KPI exclusion has concrete choke points:** the proposal-signal `accept_count` is gated at
   **both** its writers — `recordProposalDecisionSignal` (incremental, `signals.ts:169`) and
   `rebuildProposalDecisionSignal` (replay/reconcile, `:300`) — and the chip-accept KPI is an
   event-table reader that filters `suggestion_kind <> 'corrective'`. §2.1's "不计入接受率" is now
   mechanized end-to-end (incremental + reconcile + chip), not aspirational.
3. **The `AcceptSuggestionChip` writer is added** as a NEW `POST …/accept-chip` endpoint (§6) — the
   schema was a dead letterbox until P5.6. The §2.1 `source_event_id` contract (`explain` for
   proactive, `tool_use` for corrective) is honored at write.
4. **TeachingDrawer corrective is failure-count-driven** (cumulative `attempt_counts.failure >= 3`,
   total — not consecutive, which is not derivable from the field, §4.3), with the active question id +
   counts surfaced in the turn/GET response (no DB column). This gives §2.1's "corrective" branch a
   live, non-hardcoded producer on the human-facing surface.

No change to the §2.1 *semantic table* (proactive/corrective definitions + source-event forms) — P5.6
implements it verbatim. The erratum is purely the missing handler/write/KPI layer. (The first draft's
soft-fail / `result_count===0` deterministic trigger is NOT part of this erratum — it was dropped as
ungrounded, §2.)

## 12. Open implementation questions (for the writing-plans stage)

Most of the first draft's open questions are now **RESOLVED** by the grounding pass:

- **RESOLVED — chip writer location.** A NEW `POST /api/teaching-sessions/[id]/accept-chip` endpoint
  (+ a Copilot counterpart) is the first writer of `AcceptSuggestionChip` (§6 / call-site 12). It has
  the chip's `suggestion_kind` (from the drawer/Copilot chip) and resolves `source_event_id` per
  ADR-0011 §2.1 (explain/tool_use).
- **RESOLVED — chip-accept KPI reader.** An event-table reader: aggregate `action='accept_suggestion'`
  events `GROUP BY payload->>'suggestion_kind'`, filter `<> 'corrective'` (SK-7 / §5.2). No
  `chip_accept_signals` table.
- **RESOLVED — `result_count` is dropped.** The bridge never populates it (`mcp-bridge.ts:287`); a
  soft-fail read is `outcome:'success'` (`types.ts:60`). It stays in `ToolUseQuery` (`known.ts:438`)
  as an **unpopulated future extension point** only; the corrective trigger no longer keys on it.
- **RESOLVED — `ProposalSignalSource` widening.** It is the narrowed `{ cooldown_key?: string }`
  today (`signals.ts:56`–`:62`) and **must be widened** to also carry `suggestion_kind?:
  SuggestionKindT` (call-site 5). The rebuild reconstructs the marker from the **proposal payload**
  (a `Set<correctiveProposalId>`), not from the `rate` event (call-site 7).
- **RESOLVED — TeachingDrawer N + semantics.** `N = 3` as a single-source const
  `TEACHING_CORRECTIVE_FAILURE_N`; the trigger is **total** `attempt_counts.failure >= N` (the field
  is a cumulative sum, `context-readers.ts:599`–`:604`/`:623`), not consecutive.

Genuinely-open items for writing-plans:

1. **Reset / re-surfacing UX for the corrective chip.** Once `attempt_counts.failure >= N` fires the
   corrective chip, what dismisses it — a later `success` answer, or does it persist until the question
   leaves the active slot? (The count is cumulative, so it does not decrease; the drawer decides
   surfacing, not the count.) Pin the UX in the writing-plans stage.
2. **Consecutive-failure refinement (deferred).** A "still failing now" trigger would need a new
   attempt-stream query (read the ordered attempt timeline, count the trailing failure run) — not
   derivable from `attempt_counts`. Note as a future refinement; not built now (§4.3).
3. **Copilot accept-chip endpoint shape.** Whether the Copilot counterpart is a sibling route under
   `app/api/copilot/*` or shares a handler with the teaching-sessions one; confirm the Copilot chip
   carries the same `source_event_id` provenance (the `copilot_chip_trigger` / `explain` event).
4. **Double-count guard (proactive chip + proposal accept).** If a proactive chip both writes
   `AcceptSuggestionChip` AND triggers `acceptAiProposal`, confirm the two aggregates (event-table
   chip-KPI vs `proposal_signals.accept_count`) are not summed into one "the" acceptance metric;
   document which is the headline KPI to avoid double attribution.
5. **Materialized `chip_accept_signals` table — future only.** If a high-frequency consumer of the
   chip-accept rate appears, revisit SK-7 and materialize an aggregate; until then the event-table
   reader is sufficient (YAGNI).
6. **Corrective-dismiss count-skip mechanism (re-critique issue 4 impl detail).** The incremental gate
   must skip the `dismiss_count` increment for a corrective dismiss **while still writing
   `cooldown_until`**. `recordProposalDecisionSignal`'s dismiss path today does both in one UPSERT —
   pin the split: either thread a `countsTowardKpi: false` flag into `recordProposalDecisionSignalLocked`
   that zeroes the count delta but keeps the cooldown write, or branch a cooldown-only write for
   corrective dismisses. Choose at writing-plans; both keep ND-SK-4 intact.
7. **`getQuestionContext` is a DomainTool, not a free function (re-critique issue 2).** The
   `attempt_counts` aggregate lives inside `executeGetQuestionContext(ctx, raw)` (`context-readers.ts`
   ~`:556`–`:604`), which needs a `ToolContext`. The turn/GET route cannot call it as a drop-in reader.
   Pin: extract a plain `countAttemptOutcomes(db, questionId)` helper from `:597`–`:604`, OR call
   `getQuestionTimeline` + filter directly in the route. Small, but name the chosen path.
8. **`attempt_counts` is only non-zero on later turns (re-critique issue 3).** On the turn that *creates*
   an `ask_check` question, that question's `attempt_counts.failure` is 0 (no attempts yet); the failure
   total that should drive the corrective chip accrues on subsequent attempts. So the corrective trigger
   is evaluated against the count surfaced by the **GET poll / the turn AFTER an attempt lands**, not the
   question-creation turn — wire the GET path as the primary source (the turn-response value is a
   convenience, not the trigger source).
9. **accept-chip endpoint guard: use `assertActive`, not `assertAcceptingTurns` (re-critique issue 5).**
   `assertAcceptingTurns` (`conversation.ts:262`–`:279`) auto-resumes an idle session and writes a
   `conversation.resumed` job event as a side-effect — undesirable for a chip click. Prefer `assertActive`
   (`:293`, returns 409-on-ended without the resume side-effect). Pin the choice at writing-plans.
10. **Corrective marker must survive the rubric round-trip (re-critique issue 6).** `proposeKnowledgeEdgeExecute`
    runs `parseAiProposalPayload` → `validateProposalQuality` (`proposal-tools.ts:391`–`:402`) BEFORE
    `writeAiProposal`, so `suggestion_kind` flows through the rubric parse. Add to AC-3 an assertion that
    a corrective-labeled edge still parses + passes the rubric with the marker intact (so a future rubric
    edit can't silently strip it before the write).
11. **Citation hygiene.** `context-readers.ts` and `mcp-bridge.ts` live under `src/server/ai/tools/`
    (not `src/server/knowledge/`); use the full path at first mention so the implementer opens the right
    tree (re-critique issue 1).
