## 1. Problem & scope (Layer-8 fixture-gating framing)

The P5.x Layer-8 surfaces — Copilot Drawer, Dreaming, Coach, Knowledge Maintenance — are all
*agent loops over the DomainTool suite* (23 tools registered in
`src/server/ai/tools/bootstrap.ts:41`–`:65`). Their quality depends on a property the current test
layer does NOT assert: **that a tool's `Output` is intelligible to an agent** — that the key insight
fields the agent must reason on are present and non-null, that any ids the output cites resolve to
real seeded rows, and that an empty read is a legitimate success rather than a malformed blob.
`knowledge.md §5` lists **4** fixtures (`:403`–`:408`); the brief's "10-question" framing
(`P5.5`, brief line 349) *expands* those 4 plus 6 more — the §5 doc table is not itself a 10-row list.
Phase 1 implements the 4 §5 fixtures; the **remaining 6** come from the brief, not §5 (§6, LD-1).
Today "only the knowledge fixture landed" (`read-tools-m2.test.ts`, which asserts row shape, not
agent-readability). The brief frames this as
**Layer-8 critical: without fixtures it is impossible to verify tool outputs are intelligible to
agents (not merely that SQL returns rows)** (brief lines 313–323).

The stake is specific and code-checkable. Today:

1. **No agent-readability assertion exists.** `read-tools-m2.test.ts` asserts `output.nodes.length`,
   `output.edges[0].relation_type`, etc. — *structural* checks. Nothing asserts that the documented
   **key insight fields** (e.g. `mistakes[].cause`, `get_attempt_context.cause`,
   `learning_item.evidence[].id`) are present and non-null for the scenario, that ids the output cites
   (`evidence_event_ids`, `knowledge_ids`, `evidence[].id`) **resolve** to seeded rows, or that no
   null leaks into a field the agent must reason on. An agent can be handed a structurally-valid blob
   that is semantically unreadable, and no test catches it.

2. **No multi-tool chain coherence test exists.** The Layer-8 loops are *chains*
   (`query_mistakes` → `get_attempt_context` → `attribute_mistake` → `propose_variant`). The
   readiness brief lists "E2E chained-tool tests" as missing (brief line 332). Each stage's output is
   the next stage's input; if a hand-off field is null or unresolvable, the chain silently breaks and
   the agent produces garbage. No test asserts stage N's output actually feeds stage N+1.

3. **Zero-result semantics are untested at the fixture layer.** Per P5.6 (`mcp-bridge.ts:287`:
   `outcome = errorReason ? 'failure' : 'success'`), a soft-fail empty read returns
   `outcome:'success'` — a zero-result is a **legitimate success the agent must treat as "I looked and
   found nothing", NOT a corrective trigger**. No fixture pins this contract, so a regression that
   makes an empty read throw (or that an agent misreads it) would not be caught here.

4. **Cost labeling is never asserted.** Every tool declares a `costClass` on the `DomainTool` struct
   (`types.ts:59`; e.g. `query-mistakes.ts:263` `costClass:'local'`), and exposes a `summarize()`
   folded label (`types.ts:62`–`:63`). No fixture asserts the cost label is present and well-shaped,
   so a tool shipped without a cost class (or with a malformed summary) is invisible to the gate.

**Layer-8 stake.** Phase-3 Global Copilot Orchestrator and the P5.x drawer/coach launches depend on
these tools being *agent-usable*, not just *SQL-correct*. P5.5 Phase 1 builds the fixture harness that
**gates** that property for the 4 highest-value scenarios, and establishes the reusable
agent-readability assertion contract (§3) that Phase 2 and all future tools reuse. The fixtures become
the **tool-contract source-of-truth**: when a tool's `Output` shape changes, the fixture is the test
that must be updated in lockstep (§5.4, the maintenance note).

**Scope.** Backend-only server **test layer**. One new DB test file (`fixtures.test.ts`) + one
reusable assertion helper. **wenyan only** (the existing knowledge-graph corpus). **Happy-path +
KNOWN errors only** (zero-result success, duplicate-edge rejection) — NOT malformed-input /
permission / rate-limit recovery (§6). Reuses the existing `resetDb()` / direct-`db.insert` seed
primitives. **No new schema, no migration, no new UI, no new tool, no production-code change** —
every tool under test already exists and is registered. ~600 lines (brief line 372).

## 3. The agent-readability assertion contract (the key deliverable — LD-4)

This is the spec's primary output. "Agent-readable" is made **code-checkable** as a single reusable
helper the fixtures call. It is grounded in the real `Output` shapes (§Ground / per-tool anchors), not
an abstract notion.

### 3.1 Signature

```ts
// src/server/ai/tools/fixtures-assert.ts  (NEW test-support module, co-located)
interface AgentReadableSpec {
  /** Dotted paths that MUST be present and non-null for THIS scenario,
   *  e.g. ['mistakes.0.cause', 'mistakes.0.cause.primary_category']. */
  keyInsightFields: string[];
  /** Id-bearing dotted paths whose value(s) must resolve to a seeded row,
   *  paired with the table to look them up in. Arrays are checked elementwise. */
  idRefs: Array<{ path: string; table: SeededTable; column?: string }>;
  /** Dotted paths that must NOT be null/undefined (the fields the agent
   *  reasons on). Superset-friendly: keyInsightFields are auto-included. */
  nonNullFields?: string[];
  /** When true (zero-result scenarios), key-insight CONTAINERS may be
   *  empty but must be structurally present (e.g. mistakes:[] ok, not null). */
  allowEmptyContainers?: boolean;
}

async function assertAgentReadable(
  db: Db,
  tool: DomainTool<unknown, unknown>,
  input: unknown,
  output: unknown,
  spec: AgentReadableSpec,
): Promise<void>;
```

`assertAgentReadable` runs **four limbs**, each a hard assertion (LD-4 a–d):

### 3.2 Limb (a) — key-insight fields present + non-null

For each `keyInsightFields` path, assert the resolved value is **present and non-null** (or, when
`allowEmptyContainers`, present-but-possibly-empty for the *container*, never `null`). The
per-scenario key-insight sets come straight from the documented "Key insight fields" in §Ground.
The table below is the **Phase-1 tools actually under test** (6 tools across the 4 scenarios);
`get_subject_graph_overview` and `query_knowledge` are **Phase-2 reference rows — NOT asserted here**
(included only because §5's `wenyan-zhi-confusion` doc behavior names `query_knowledge`; Phase 1
realises that question via the diagnostic chain in scenario (i) instead — §4.1):

| tool | key-insight fields asserted | anchor |
|---|---|---|
| `query_mistakes` | `mistakes[].cause` (`source`/`primary_category`/`confidence`), `mistakes[].review_state` (`due_at`/`is_due`), `mistakes[].knowledge_ids` | `query-mistakes.ts:54`–`:75` |
| `get_attempt_context` | `attempt`, `cause` (`source`/`primary_category`/`analysis_md`/`confidence`), `timeline[].event_id` | `get-attempt-context.ts:76`–`:89`, `:35`–`:42`, `:47` |
| `get_learning_item_context` | `item.status`, `hierarchy.children`, `evidence[].id`/`path`/`summary`, `knowledge_context[].knowledge_id` | `context-readers.ts:848`–`:905`, `:881`–`:884`, `:896`–`:904` |
| `attribute_mistake` | `status`, `judge_event_id` (on `written`), `cause` (`primary_category`/`confidence`/`analysis_excerpt`) | `proposal-tools.ts:689`–`:701` |
| `propose_variant` | `status`, `proposal_ids`/`mistake_variant_ids` (arrays; `variant_question_ids` is hardcoded `[]` — H3, not asserted non-empty) | `proposal-tools.ts:829`–`:848`, `:875`–`:881` |
| `propose_knowledge_edge` | `status`, `proposal_id` (on `proposed`), or `cooldown_key`/`reason` (on skip) | `proposal-tools.ts:214`–`:242`, `:316`–`:324` |
| `propose_learning_item_completion` | `status`, `proposal_id` + `learning_item_id` (on `proposed`) | `proposal-tools.ts:941`–`:969` |
| _Phase-2 ref (not asserted):_ `get_subject_graph_overview` | `clusters[].name`/`child_count`; `weak_node_count`/`recent_failure_count_30d` are `.optional()` and ONLY set when `includeWeaknessSummary:true` (`:318`–`:323`) | `knowledge-readers.ts:261`–`:282` |
| _Phase-2 ref (not asserted):_ `query_knowledge` | `nodes[].path`, `nodes[].stats` (when included), `edges[].relation_type` | `knowledge-readers.ts:342`–`:392` |

### 3.3 Limb (b) — cited ids resolvable

For each `idRefs` entry, resolve the path (elementwise for arrays) and assert each non-null id value
**exists** in the named seeded table/column. Examples per scenario:
- scenario (i): `mistakes[].event_id` → `event`; `mistakes[].question_id` → `question`;
  `mistakes[].knowledge_ids` → `knowledge`; chained `judge_event_id` → `event`.
- scenario (iv): `item.id` → `learning_item`; `evidence[].id` → `completion_evidence`;
  `knowledge_context[].knowledge_id` → `knowledge` (the field is `knowledge_id`, NOT `.id` —
  `context-readers.ts:899`).

**Documented limitation (F-3).** `query_knowledge.edges[].evidence_event_ids` is an **empty `[]`** in
the current implementation (`knowledge-readers.ts:378` declared, `:499` filled empty — "no actual
event tracing yet"). The contract therefore asserts the field is **present (an array)** and that
*every element* (vacuously, none) resolves — it does NOT require non-empty. The fixture carries a
comment naming this as the current impl reality so a future tracing change tightens the fixture (LD-7).

### 3.4 Limb (c) — no nulls in critical fields

`nonNullFields` (∪ `keyInsightFields`) must each resolve non-null/non-undefined. This catches the
"structurally valid but semantically blank" blob (e.g. a `cause` object present but
`cause.primary_category === null`). Zero-result containers are exempted only when
`allowEmptyContainers` is set (scenario ii) — and even then the *container* must be a non-null empty
array, never `null` (F-5).

### 3.5 Limb (d) — cost label present + well-shaped (LD-6 / F-2)

Assert on the **`DomainTool` struct**, not the `Output`:
- `tool.costClass` is one of the valid `ToolCostClass` enum values — the full enum is
  `'local' | 'cheap_llm' | 'expensive_llm'` (`types.ts:30`; the `costClass` field is `types.ts:59`); the
  Phase-1 tools use `'local'` (reads) and `'cheap_llm'` (`attribute_mistake` / `propose_variant`);
- `tool.summarize(input, output)` returns a **non-empty string ≤120 chars** — the contract comment says
  "Must not exceed **~**120 chars" (`types.ts:62`), so the limb pins a hard `≤120` as a test-layer floor
  on that soft guidance (a one-line comment notes the `~` is intentionally tightened to a hard bound here).

This is presence + shape only — **no exact cost value, no latency** (LD-6; perf is §6-deferred). The
limb is cheap and runs for every tool in every scenario, so a tool shipped without a cost class or
with a malformed summary fails the gate.

> **Net contract.** `assertAgentReadable` makes "intelligible to an agent" mean exactly: *the agent
> can read the documented insight fields (a), follow every id it cites to a real row (b), never trips
> on a null in a field it must reason on (c), and the tool self-describes its cost (d).* This helper is
> the reusable Phase-2 / future-tool foundation.

## 4. The 4 core scenarios

Each scenario: tool sequence → minimal seed → chain hand-offs → key assertions → which §5 fixture
question it answers. All seeding reuses §5.2 primitives; all output checks route through §3.

> **First-class precondition — the P5.4 agent rubric (applies to EVERY agent-gated propose).** `ctx()`
> uses `callerActor.kind:'agent'`, so any propose that runs `validateProposalQuality(…, {isAgent:true})`
> is held to the P5.4 rubric. **Which tools this gates matters and is verified per tool:**
> - `propose_knowledge_edge` **IS** rubric-gated (`proposal-tools.ts:391`–`:402`). The rubric
>   **REJECTS** an edge with `evidence_event_ids.length === 0` → gate `'evidence_missing'`
>   (`rubric-validator.ts:511`–`:518`), folded to `status:'skipped:rubric_rejected'`
>   (`proposal-tools.ts:457`–`:463`). So any propose-edge seed that wants `status:'proposed'` MUST seed
>   in-window judge-backed confusion evidence AND pass non-empty `evidence_event_ids` — mirroring
>   `proposal-tools.test.ts:296`–`:317` (`seedConfusionEvidence` ×2, `evidence_event_ids:['conf_1','conf_2']`,
>   concrete-signal reasoning). This is enforced in scenario (iii).
> - `propose_variant` is **NOT** rubric-gated — it goes through `runVariantGen` (`proposal-tools.ts:859`),
>   whose own guards require an existing judge / targetable cause / depth caps (the `no_judge_yet` path,
>   §4.1 stage 4). It still makes a live-LLM call → §5 mock (B2).
> - `propose_learning_item_completion` is **NOT** rubric-gated — its preconditions are item-found +
>   state ∈ `{pending,in_progress}` + no prior pending completion + `writeCompletionProposal` returns an
>   id (`proposal-tools.ts:941`–`:968`); no `validateProposalQuality`, no LLM (scenario iv).

### 4.1 Scenario (i) — mistake-confusion chain (CHAIN, LD-5)

**Answers §5 question:** `wenyan-zhi-confusion` ("我为什么老错『之』？" — the agent inspects the
mistake, its attempt context, attributes a cause, and proposes a variant) — extends the §5 row beyond
its `query_knowledge` example into the full diagnostic chain.

**Tool sequence:** `query_mistakes` → `get_attempt_context` → `attribute_mistake` →
`propose_variant`.

**Minimal seed:** one wenyan `knowledge` row (`k_zhi`, the §5.2 `seedGraph` shape); one `question`
(`knowledge_ids:['k_zhi']`); one **failure** attempt via `seedAttempt({ outcome:'failure',
question_id, knowledge_ids:['k_zhi'] })` (`event-seed.ts:58`, which also mirrors a
`learning_record(kind='mistake')`); one `experimental:user_cause` via `seedUserCause({
attempt_event_id, primary_category:'concept' })` (`event-seed.ts:129`) so a cause exists for stage 1/2
to surface. (No pre-seeded judge — stage 3 `attribute_mistake` writes it, via a **stubbed** runner.)

**LLM stub (B2 — MANDATORY).** Before stage 3, `mockRunner.runTask.mockResolvedValueOnce(<AttributionTask
JSON>)`; before stage 4, `mockRunner.runTask.mockResolvedValueOnce(<variant-gen JSON>)` — see §5.2.1.
The LLM is never live in this fixture.

**Chain hand-offs (each asserted non-null before threading, F-4):**

| from stage | output field | → into stage | input field |
|---|---|---|---|
| 1 `query_mistakes` | `mistakes[0].event_id` | 2 `get_attempt_context` | `attemptEventId` |
| 2 `get_attempt_context` | `attempt.event_id` | 3 `attribute_mistake` | `attempt_event_id` |
| 3 `attribute_mistake` | `judge_event_id` (≠ null after write) | 4 `propose_variant` | takes `attempt_event_id` (the SAME attempt id, `proposal-tools.ts:825`), NOT `judge_event_id`; stage 3's write is the *precondition* `runVariantGen` re-reads — judge must exist or `skipped:no_judge_yet` |

**Key assertions:**
- Stage 1: `assertAgentReadable(query_mistakes, …)` with key-insight = `mistakes[0].cause` +
  `review_state` + `knowledge_ids`; id-refs resolve `event_id`/`question_id`/`knowledge_ids`.
- Stage 2: `cause` non-null, `timeline[].event_id` resolvable; `assertAgentReadable`.
- Stage 3: `status === 'written'`, `judge_event_id` non-null and resolves in `event`; `cause` populated
  (`proposal-tools.ts:722`–`:803`).
- Stage 4: `status === 'generated'` with **non-empty `proposal_ids` (and/or `mistake_variant_ids`)** —
  **do NOT assert `variant_question_ids` non-empty: it is hardcoded `[]` on success in the current impl
  (`proposal-tools.ts:879` — H3)**, mirror `proposal-tools.test.ts:570`–`:573`. OR an explicit documented
  skip (`proposal-tools.ts:830`–`:842`) asserted by status — the chain asserts the judge precondition
  holds so `no_judge_yet` does NOT fire.

### 4.2 Scenario (ii) — zero-result-corrective (ISOLATED, KNOWN ERROR, LD-3 / F-5)

**Answers §5 question:** `wenyan-zero-result` ("最近有没有『焉』的错题？" — the read returns 0 results;
the agent must treat it as a legitimate "found nothing", NOT a corrective trigger — ties directly to
P5.6 suggestion semantics).

> **Doc-wording note (M6).** §5's expected-behavior text says the read "returns 0 result **with
> corrective query suggestions**" (`knowledge.md:407`). The current `query_mistakes` zero-result Output
> has **no `suggestions` field** — it returns only `mistakes:[]` / `total:0` / `filter_applied`. That §5
> wording predates P5.6; **this fixture pins the P5.6 empty-success semantics (no suggestions field), NOT
> the §5 "corrective suggestions" phrasing.** A future §5 doc-wording fix should drop "with corrective
> query suggestions" (tracked as a §9 follow-up).

**Tool sequence:** `query_mistakes` with `{}` (no matching data) — isolated single-tool assert.

**Minimal seed:** a single wenyan `knowledge` row, **NO failure attempts, NO questions** — the read
must legitimately return empty.

**Key assertions:**
- `execute()` **does NOT throw** (soft-fail contract, `types.ts:60`).
- Output is the valid empty shape: `mistakes` is `[]` (present, non-null array), `total === 0`,

## 5. Harness design

### 5.1 Location & partition

- **New file:** `src/server/ai/tools/fixtures.test.ts`, co-located with `read-tools-m2.test.ts`.
- **New test-support module:** `src/server/ai/tools/fixtures-assert.ts` (the `assertAgentReadable`
  helper, §3) — a non-`.test.ts` helper so it can be imported by both Phase-1 fixtures and Phase-2.
- **Partition:** DB partition (`vitest.db.config.ts`). It imports `tests/helpers/db`, live tool
  modules, and `drizzle` — per CLAUDE.md it MUST NOT be in the unit config; `pnpm audit:partition`
  enforces this (F-7). Run via `pnpm test:db:watch src/server/ai/tools/fixtures.test.ts`.

### 5.2 `ctx()` / seed pattern reused

- **`ToolContext`:** reuse the `ctx()` factory shape (`read-tools-m2.test.ts:38`–`:44`):
  `{ db: testDb(), taskRunId, callerActor:{ kind:'agent', ref:'agent:copilot' } }`. Proposal tools
  need an agent caller (matches `proposal-tools.test.ts:60`–`:66`).
- **Reset:** `beforeEach(resetDb)` (`tests/helpers/db.ts:49`–`:54`, `TRUNCATE … CASCADE` over
  `ALL_TABLES`) for hermetic isolation. DB config is `singleFork` (CLAUDE.md) — do not assume isolated
  databases per file.
- **Seed primitives (no new helpers unless a per-scenario factory pays for itself):**
  - knowledge/edges: direct `db.insert(knowledge|knowledge_edge)` in the `seedGraph` shape
    (`read-tools-m2.test.ts:61`–`:98`).
  - attempts + mistakes: `seedAttempt` (`event-seed.ts:58`, auto-mirrors `learning_record`).
  - causes: `seedUserCause` (`event-seed.ts:129`).
  - learning objects: the `seedLearningObjects` shape (`read-tools-m2.test.ts:189`–`:264`).
- **Fixture data lives inline** in `fixtures.test.ts` (the `read-tools-m2.test.ts` convention), NOT a
  separate `.json` — the seeds are small and scenario-specific; a shared data file is YAGNI for 4
  scenarios (revisit in Phase 2 if the corpus grows — §9).

#### 5.2.1 LLM is ALWAYS stubbed (MANDATORY — B2)

Scenario (i) stages 3 (`attribute_mistake`) and 4 (`propose_variant`) make **live LLM calls**:
`attribute_mistake` → `runAttributionAndWriteJudgeEvent`, `propose_variant` → `runVariantGen`, both
calling `defaultRunTaskFn` → `runner.ts` `runTask` (needs `ANTHROPIC_API_KEY` + the live Agent SDK).
**There is NO per-call injection point on `tool.execute()`** — the fixture calls the real tool, which
calls the real runner. So the harness MUST stub at the module level, exactly as
`proposal-tools.test.ts:34`–`:56` does:

```ts
const mockRunner = vi.hoisted(() => ({ runTask: vi.fn() }));
vi.mock('@/server/ai/runner', () => ({ runTask: mockRunner.runTask }));
vi.mock('@anthropic-ai/claude-agent-sdk', () => ({ /* createSdkMcpServer + tool stubs */ }));
```

and set a fresh result **per stage** with `mockRunner.runTask.mockResolvedValueOnce(...)` before each
LLM-backed `execute()` (the AttributionTask JSON for stage 3 — `proposal-tools.test.ts:466`–`:477`; the
variant-gen JSON for stage 4 — `:557`–`:568`). **The LLM is ALWAYS stubbed; there is no CI-safe "real"
option** — this is what keeps the fixture hermetic and within **LD-6 "no LLM cost"**. (Read-only tools —
`query_mistakes`, `get_attempt_context`, `get_learning_item_context`, `propose_knowledge_edge`,
`propose_learning_item_completion` — touch no LLM and need no per-call mock, but the module-level mocks
must still be declared so the SDK import does not spawn Claude.)

### 5.3 Cost-label assertion approach (LD-6 / §3.5)

The cost limb reads `tool.costClass` (struct field, `types.ts:59`) and `tool.summarize(input, output)`
(`types.ts:62`) — both are on the `DomainTool` the fixture already holds. It asserts enum-membership +
non-empty-≤120-char string. **No** assertion against `Output` (there is no cost field there) and **no**
exact value (§6). This runs inside `assertAgentReadable`, so every scenario covers it automatically.

### 5.4 Maintenance — fixtures are the tool-contract source-of-truth (LD-7, process note)

When a tool's `Output` shape, status enum, key-insight fields, or `costClass` changes, the change MUST
update the corresponding scenario in `fixtures.test.ts` (and its `keyInsightFields`/`idRefs` spec) in
the same PR. This is a **documented process expectation**, enforced by the test failing — **not** a new
cron/lint/automation (CLAUDE.md scope discipline; no new infra). A short note is added to
`docs/modules/knowledge.md §5` pointing at `fixtures.test.ts` as the executable form of those fixtures.
