# Subject-scoped vs Global Brief Refresh (P5.2) — Design Spec

> Status: **design approved 2026-05-31** (feature-level decision). Implements roadmap **P5.2**
> ("Subject-scoped vs global brief note 并行刷新"); feeds the **Layer-8 Copilot orchestrator** +
> **North-Star** Coach/Dreaming surfaces. Refs
> [YUK-143](https://linear.app/yukoval-studios/issue/YUK-143).
> Source: 2026-05-30 P5 readiness brief §P5.2 (Option A, EXPLORE-phase) + 2026-05-31 user decision
> (drop goal-coupling; gate on **recent activity** instead).
> Build 排期: writing-plans next; backend-only, no UI.

## 1. Problem

`memory_brief_note` (ADR-0017 Layer 2) is the pre-summarized "方向感" the Layer-8 orchestrator,
Dreaming, and Coach read before consulting SoT. A brief is only useful if it is **reasonably fresh**
for subjects the user is actually working on, and **not expensive to maintain** for subjects that
have gone quiet.

The current refresh path has two gaps for the per-subject case:

1. **The nightly sweep is freshness-gated, not activity-gated.** `buildMemoryBriefSweepHandler`
   (`src/server/memory/triggers.ts:152`) calls `listStaleBriefScopes` (`brief.ts:121`), which
   selects every existing `memory_brief_note` row whose `refreshed_at IS NULL OR < now() - 24h` and
   enqueues a regen for each. The per-scope regen handler then calls `scopeHasNewEvidence`
   (`brief.ts:132`) to skip no-op LLM calls. So the *cost gate* exists, but it is keyed on
   "does this **already-existing** brief row have an event newer than its `latest_evidence_at`" —
   it does **not** discover which *subjects* became active, and it never creates a per-subject brief
   for a subject that has activity but no row yet.

2. **`subject:` scopes are barely populated by real learning activity today.** `affected_scopes`
   is computed at event-write by `computeAffectedScopes` (`src/server/memory/scope_tagger.ts:38`,
   called from `writeEvent` `queries.ts:990`). It emits `subject:<slug>` **only** when the event
   payload carries `subject_id` / `subject` / `domain`. But the two core learning events —
   `AttemptOnQuestion` and `ReviewOnQuestion` (`src/core/schema/event/known.ts`) — carry only
   `referenced_knowledge_ids` in their payloads, **no** `subject_id` / `domain`. So a real attempt
   or review today tags `global` + `topic:<knowledge_id>` and (in the common case) **no
   `subject:X`**. There is no path that resolves those `referenced_knowledge_ids` to a subject and
   refreshes that subject's brief. The per-subject brief layer is effectively dormant for the
   subjects the user actually studies.

Net: a blind daily refresh of all subjects (readiness-brief Option B) would waste LLM cost on
dormant subjects, and the existing freshness-gated sweep never reaches the subjects that *are*
active because their events don't carry a `subject:` scope. P5.2 closes both.

## 2. Locked decision

**Per-subject briefs refresh when the subject has recent ACTIVITY** (activity-gated), **not**
goal-gated, and **not** a blind daily refresh of all subjects. This generalizes the readiness-brief
EXPLORE-phase Option A by dropping the North-Star coupling: a subject earns a refresh by being
*worked on*, regardless of whether it has an active goal.

Sub-defaults (user delegated specifics; stated explicitly so they can be adjusted at spec-review):

| # | Decision | Rationale |
|---|---|---|
| BR-1 | **Activity gate = since-last-refresh delta.** A subject's per-subject brief is refreshed when that subject has **≥ 1 qualifying activity event with `created_at` strictly after that brief's last refresh**. If no per-subject brief / `refreshed_at` exists for the subject yet, treat it as **needing an initial refresh** (any qualifying activity in a bounded lookback window triggers the first build — see BR-5). | The delta is the natural cost gate: a subject with zero new activity since its brief was last written has nothing new to summarize. Reuses the existing `latest_evidence_at`-style "is there anything newer" semantics already proven in `scopeHasNewEvidence`. |
| BR-2 | **Dormant subjects are skipped.** A subject with zero qualifying activity since its last refresh incurs **no** refresh cost this run. Its existing brief (if any) is left untouched; its `refreshed_at` / `latest_evidence_at` staleness is itself the signal "this subject has been quiet" (ADR-0017 §"Cron daily sweep" reads staleness as a quiet-signal). | Cost only where the user is active. Aligns with ADR-0017's stated cost concern (~$0.20–0.60/day) and its long_term cost-overrun risk. |
| BR-3 | **Qualifying activity = subject-resolvable learning events.** The actions that count: `attempt` (`AttemptOnQuestion`), `review` (`ReviewOnQuestion`), `experimental:record_capture` (direct `/record` entry), and the ingestion enroll path (whatever event the enroll pipeline writes — confirm at impl, see §5). An event qualifies for subject **S** when its `referenced_knowledge_ids` (or, for capture/enroll, its knowledge ids) **resolve to S** via the knowledge→subject resolution (BR-4). | These are the events that mean "the user did learning work on this subject". `judge` / `propose` / `generate` / `rate` are agent-side or meta and are intentionally excluded from the gate (they don't represent the user studying the subject). |
| BR-4 | **Subject resolution = `knowledge_ids → getEffectiveDomain → resolveSubjectProfile.id`** — the exact bridge already used by T-CS cross-subject scheduling (`src/server/review/due-list.ts:125` `batchResolveSubjectIds`). Orphan / unresolvable knowledge ids fall back to the default subject profile id (YUK-56), so they refresh the default subject's brief rather than being dropped. | This is the project's one canonical "which subject does this knowledge belong to" answer. Reusing it keeps P5.2 consistent with the review scheduler and avoids a second, drifting subject-resolution path. It is also exactly the bridge the current `computeAffectedScopes` lacks (§1.2). |
| BR-5 | **Initial-build lookback window = bounded (default 30 days).** When a subject has qualifying activity but no brief row yet, the "since-last-refresh" delta is undefined, so use a bounded lookback (default **30 days**) to decide it needs an initial build and to seed the event window the brief summarizes. | Avoids re-summarizing a subject's entire history on first build, and avoids an unbounded "treat null as epoch" scan. 30d matches the existing `recent_months` window granularity intent (ADR-0017 3-window model). Tunable. |
| BR-6 | **Global brief refresh behavior is unchanged.** P5.2 **adds** the per-subject activity-gated layer on top of the existing `global` sweep/regen. | BR-7 | **Parallel-safe by construction.** Per-subject briefs are keyed by a **distinct `scope_key` per subject** (`subject:<id>`), one DB row each, mirroring the existing `memory_brief_note_scope_key_unique` index + per-scope `singletonKey: memory.regen.<scope_key>`. Refreshing `subject:A` and `subject:B` concurrently touches different rows and different singleton locks → **no clobber**. | The schema + queue already give per-scope isolation; P5.2 only needs to enqueue one regen per active subject and (optionally) allow the regen handler to drain more than one job at a time. |
| BR-8 | **Runs in the existing nightly brief-refresh handler (batch, not real-time).** Extend `buildMemoryBriefSweepHandler` (or a sibling sweep) to additionally: compute the set of subjects with qualifying activity since their brief's last refresh, and enqueue one `MEMORY_BRIEF_REGEN_QUEUE` job per active subject. No new cron surface, no read-time refresh. | Reuses the proven nightly batch path (ADR-0017 trigger #3). Read-time / on-demand refresh (readiness-brief Option C) was rejected there for latency/complexity; P5.2 keeps the batch model. |
| BR-9 | **Per-run and per-brief activity read are budgeted** (via P5.1 budgets.ts pattern). Add `BRIEF_REFRESH_BUDGET` constant with `maxSubjectsPerRun` (default: 12) and `maxEventsPerBrief` (default: 50, formalizing the current hardcoded `.limit(50)` in `brief.ts:86`). When more active subjects exist than `maxSubjectsPerRun`, prioritize by activity recency/volume and defer the rest to the next night (no starvation, just spread out). Per-brief regen reads at most `maxEventsPerBrief` recent events. | Bounded nightly cost/latency even under an activity burst. Consistent with the P5.1 budgets.ts single-source pattern (src/server/ai/tools/budgets.ts §3.1). | BR-10 | **Subject regen reads a knowledge-resolved event list and uses a knowledge-resolved freshness check — NOT `affected_scopes`.** The nightly activity-detection query (BR-1 / §3.2) is the single source of truth for both "is subject X fresh" and "which events seed its brief". The per-subject regen path therefore injects an explicit event list resolved by knowledge-id→subject (via `regenerateMemoryBrief`'s **existing** `loadEvents` injection param, `brief.ts:156`) and gates on a knowledge-resolved freshness check; it does **not** reuse the `affected_scopes @> [subject:X]` filter that `loadEventsFromDb` / `scopeHasNewEvidence` apply for `global`. | **Load-bearing — without this the layer is non-functional.** `computeAffectedScopes` (`scope_tagger.ts:38`) tags `subject:` **only** when an event payload carries `subject_id` / `subject` / `domain`; the two core learning events (`AttemptOnQuestion` / `ReviewOnQuestion`) carry only `referenced_knowledge_ids`, so they **never** tag `subject:`. Tagging `subject:` at event-write (option a) is rejected for P5.2 because it would force a DB read into the deliberately-pure synchronous `writeEvent` path (§6 future work). |

### Rationale vs the readiness-brief options

The 2026-05-30 readiness brief recommended **Option A** (global always + per-subject **on active
goals or recent activity**). The user's decision keeps Option A's shape but **decouples it from
North-Star goals**: the gate is purely *recent activity*. This is strictly more general — a subject
the user studies without any declared goal still gets a fresh brief — and it removes a brittle
goal↔refresh coupling. Options **B** (blind daily, wastes budget on dormant subjects), **C**
(read-time lazy, latency/complexity), and **D** (drop the global brief) remain rejected for the
reasons in the readiness brief §P5.2.

## 3. Design

### 3.1 Per-subject brief storage (scope_key scheme)

No schema change. Per-subject briefs reuse `memory_brief_note` with `scope_key = 'subject:<id>'`,
where `<id>` is the `resolveSubjectProfile(...).id` (BR-4). This matches:

- the existing scope taxonomy (`brief.ts` `prefixForScope` already maps `subject:` → the `subject`
  template; `subjectForScope` already extracts the id into the `subject_id` column),
- the `MemoryBriefScopeKey` business validator (`src/core/schema/business.ts:98`), whose regex is
  `^(global|subject:[a-zA-Z0-9_-]+)$` — i.e. it **already** restricts brief-refresh-event scope keys
  to exactly `global` + `subject:*`. **This is a constraint, not an accident**: the
  `experimental:memory_brief_refresh` trace event (`src/core/schema/event/experimental.ts:81`) will
  reject any other scope. Subject ids must therefore match `[a-zA-Z0-9_-]+`; the profile ids
  resolved via BR-4 already do (confirm at impl, §5).

> Note: `brief.ts` `BRIEF_TEMPLATES` also defines `topic:` / `mistake_cluster:` / `meta:` prefixes,
> but those are **out of P5.2 scope** and are not permitted by the `MemoryBriefScopeKey` validator.
> P5.2 governs only `global` (unchanged) + `subject:*` (the new activity-gated layer).

### 3.2 Activity-detection query (active subjects since last refresh)

A new read in `src/server/memory/` (sibling to `brief.ts`), pure-ish over the `event` table +
the knowledge→subject resolver. Conceptual shape:

1. **Load existing per-subject brief refresh watermarks.** `SELECT scope_key, subject_id,
   refreshed_at FROM memory_brief_note WHERE scope_key LIKE 'subject:%'` → a map
   `subjectId → lastRefreshedAt`.
2. **Scan qualifying activity events.** Select `event` rows where `action IN ('attempt','review',
   'experimental:record_capture', <enroll-action>)`, newer than a floor = `min over subjects of
   (lastRefreshedAt)` or the BR-5 lookback for never-built subjects (bounded, default 30d), pulling
   `id`, `created_at`, and the knowledge ids (`payload.referenced_knowledge_ids` / capture knowledge
   ids).
3. **Resolve each event to a subject** via the BR-4 bridge. Reuse the `batchResolveSubjectIds`
   pattern (`due-list.ts:125`): resolve each **unique first knowledge id** once
   (`getEffectiveDomain → resolveSubjectProfile.id`), memoised, to avoid an N+1 parent-chain walk.
   Orphan ids → default subject (YUK-56).
4. **Group by subject, keep the max `created_at`.** A subject is **active** iff its newest
   qualifying event is **strictly after** its brief's `refreshed_at` (or it has no brief row and has
   activity within the BR-5 window). Emit the set `{ subjectId, scopeKey: 'subject:'+id }`.

This is intentionally the *same* knowledge→subject resolution the review scheduler uses, so "which
subject is active" and "which subject does the review pool round-robin over" can never diverge.

> **Why not just read `affected_scopes`?** Because (§1.2) attempt/review events do **not** tag a
> `subject:` scope today — they tag only `global` + `topic:<id>`. This is the *same* root cause that
> forces the subject **regen** path off `affected_scopes` (BR-10): both the activity *detection* here
> and the event *loading* in regen must resolve subjects from knowledge ids, not from
> `affected_scopes`. A future enhancement could make `computeAffectedScopes` resolve `subject:` from
> knowledge ids (so event-ingest also enqueues per-subject regen in near-real-time), but that drags a
> DB read into the synchronous `writeEvent` path and is deferred (§6 future work); the nightly
> activity-detection query above is the SoT for P5.2 and does not depend on it.

### 3.3 Refresh trigger in the nightly handler

Extend the nightly sweep (`buildMemoryBriefSweepHandler`, `triggers.ts:152`, scheduled
`0 3 * * *` Asia/Shanghai). After the existing `listStaleBriefScopes` enqueue loop (which keeps
`global` + any existing stale rows fresh, BR-6), add:

```
for (const { scopeKey } of await listActiveSubjectsSinceRefresh(db, { lookbackDays: 30 })) {
  await enqueueBriefRegen(boss, scopeKey);   // singletonKey memory.regen.<scopeKey>
}
```

`enqueueBriefRegen` (`triggers.ts:86`) already dedups per scope_key on a 6-min window, so a subject
that is also caught by `listStaleBriefScopes` is enqueued at most once.

The per-job regen handler (`buildMemoryBriefRegenHandler`, `triggers.ts:122`) **must branch by scope
prefix** (BR-10): it is shared by `global` and `subject:*` jobs off the same
`MEMORY_BRIEF_REGEN_QUEUE`, but the two scopes resolve their events differently.

- **`global` (and any legacy affected_scopes-tagged scope) — unchanged.** Still gated by
  `scopeHasNewEvidence(scopeKey)` and still loads via `loadEventsFromDb` (`affected_scopes @>
  [scopeKey]`). BR-6.
- **`subject:<id>` — knowledge-resolved (BR-10).** The handler must **not** call the affected_scopes
  `scopeHasNewEvidence` guard (it returns false for active subjects, §1.2 / BR-10) nor the
  affected_scopes `loadEventsFromDb` (it returns ~0 rows). Instead it injects an explicit event list
  resolved by knowledge-id→subject — the **same** qualifying events the activity-detection query
  (§3.2) already gathered for this subject — through `regenerateMemoryBrief`'s existing `loadEvents`
  param (`brief.ts:156`), capped at `BRIEF_REFRESH_BUDGET.maxEventsPerBrief`. Freshness is already
  decided upstream (the subject is in the active set only because its newest qualifying event is
  newer than `refreshed_at`), so no separate `scopeHasNewEvidence` call is needed for subject scopes;
  if a redundant guard is desired, it must be a knowledge-resolved check, not the affected_scopes one.

For a **never-built** subject, `regenerateMemoryBrief` (`brief.ts:153`) already handles first insert
via `onConflictDoUpdate` on the unique `scope_key`, and the injected knowledge-resolved event list
is non-empty (it is exactly why the subject was flagged active), so the initial build produces a
real brief with a non-null `latest_evidence_at` — no empty-brief / re-generate-every-night leak.

### 3.4 Parallel-safety

- **Distinct rows per subject** (BR-7): each `subject:<id>` is one `memory_brief_note` row guarded
  by `memory_brief_note_scope_key_unique`. Concurrent regen of `subject:A` / `subject:B` writes
  disjoint rows.
- **Distinct singleton locks**: `singletonKey: memory.regen.subject:A` ≠ `memory.regen.subject:B`,
  so they are not collapsed into one job.
- **Concurrency knob**: the only change needed to actually run subjects in parallel is the regen
  worker's `batchSize` (currently `1`, `triggers.ts:251`). Bumping it (e.g. to 2–3) lets pg-boss
  hand multiple `subject:*` jobs to the handler loop. Each iteration is independent (own scope_key,
  own row, own Mem0 search filter), so a higher batchSize is safe. **Whether to bump it, and to
  what, is an implementation question** (§5) — the design is correct at `batchSize:1` (serial,
  still no clobber) and merely *faster* at >1.

### 3.5 How the global brief stays unchanged

The `global` scope is never produced by the activity-detection query (it groups by *subject*).
`global` continues to be refreshed exactly as today: `listStaleBriefScopes` includes it once its
`refreshed_at` crosses 24h, and `scopeHasNewEvidence('global')` gates the LLM call. No code on the
`global` path is touched; the per-subject loop is purely additive (BR-6). Acceptance §7 asserts this
explicitly.

### 3.6 Brief-refresh budget (BR-9)

Tie P5.2's per-run and per-brief activity scope into the P5.1 budgets.ts single-source pattern
(`src/server/ai/tools/budgets.ts`, already merged to main; see the P5.1 spec §3.1 for the
`ContextBudget` interface).

**Add a new constant to `budgets.ts`:**

```ts
export interface BriefRefreshBudget {
  /** Max subjects whose briefs refresh in a single nightly sweep run. When exceeded, prioritize
   *  by activity recency/volume; defer remainder to the next night (no starvation). */
  maxSubjectsPerRun: number;
  /** Max recent activity events fed into a single subject brief's summarization.
   *  Formalizes the current loadEventsFromDb `.limit(50)` hardcoded in brief.ts:86. */
  maxEventsPerBrief: number;
}

export const BRIEF_REFRESH_BUDGET: BriefRefreshBudget = {
  maxSubjectsPerRun: 12,      // Never bites at today's 3–5 active subjects; forward-looking guard
  maxEventsPerBrief: 50,      // Existing hardcoded limit in brief.ts:86; proven bound
};
```

**Enforcement in the nightly sweep** (`buildMemoryBriefSweepHandler`, `triggers.ts:152`):

- After calling `listActiveSubjectsSinceRefresh(db, { lookbackDays: 30 })`, sort the result by
  activity recency (max `created_at` per subject, descending) and take the top
  `BRIEF_REFRESH_BUDGET.maxSubjectsPerRun`. Enqueue briefs for those only; defer the rest to the
  next run (they remain active and will be eligible again tomorrow, so no starvation).
- In `loadEventsFromDb` (`brief.ts:80`), the `.limit(50)` already honors `maxEventsPerBrief`. If
  a caller wishes to parameterise it, use `BRIEF_REFRESH_BUDGET.maxEventsPerBrief` instead of the
  hardcoded 50.

**Decision: global brief and the per-subject budget.** The global brief is **not** subject to
`maxSubjectsPerRun` (it is always enqueued via the existing 24h-stale gate and should always
complete). The global brief **may** inherit `maxEventsPerBrief` if desired (decide at impl time:
whether to read more events for the global summary or keep it at 50). This spec does not mandate a
choice; the codebase can decide per-scope at the `regenerateMemoryBrief` call site.

## 4. Migration

Additive; no schema change, no DB migration, no UI.

1. **Update `src/server/ai/tools/budgets.ts`** (P5.1 single-source): add the `BriefRefreshBudget`
   interface and `BRIEF_REFRESH_BUDGET` constant (§3.6). Export both so `triggers.ts` and
   `brief.ts` can import them. The values (12 subjects / 50 events) are tunable here only.
2. **Add `listActiveSubjectsSinceRefresh(db, opts)`** in `src/server/memory/` (new module or added
   to `brief.ts`), implementing §3.2 and reusing the `batchResolveSubjectIds` resolution pattern
   from `due-list.ts` (extract it to a shared helper if it must be imported, rather than duplicating
   the parent-chain walk). It returns, per active subject, the `scopeKey`, the max `created_at`
   (for the BR-9 recency sort), and **the resolved qualifying event window** (capped at
   `BRIEF_REFRESH_BUDGET.maxEventsPerBrief`) so the regen step (step 4, BR-10) can summarize from it
   without re-querying.
3. **Extend `buildMemoryBriefSweepHandler`** (`triggers.ts:152`) to call it, sort by activity
   recency, and enqueue only the top `BRIEF_REFRESH_BUDGET.maxSubjectsPerRun` active subjects
   (§3.6). Defer the rest to the next run. Existing `listStaleBriefScopes` loop stays (global brief
   always enqueued, BR-6).
4. **Branch `buildMemoryBriefRegenHandler`** (`triggers.ts:122`) by scope prefix (BR-10 / §3.3).
   For `subject:*`: resolve the qualifying event window by knowledge-id→subject (reuse step 2's
   resolver) and inject it via `regenerateMemoryBrief`'s existing `loadEvents` param; do **not** call
   the affected_scopes `scopeHasNewEvidence` guard or `loadEventsFromDb` for subject scopes. For
   `global` (and any legacy affected_scopes scope): unchanged. This is the load-bearing change —
   without it subject briefs are empty and never refresh (BR-10).
5. **(Optional, impl decision §5)** bump `MEMORY_BRIEF_REGEN_QUEUE` `batchSize` from 1 for parallel
   drain.
6. In `brief.ts` `loadEventsFromDb` (line 86), optionally replace the hardcoded `.limit(50)` with
   `BRIEF_REFRESH_BUDGET.maxEventsPerBrief` for consistency (pure refactor; no behavior change).
7. Update `docs/modules/` (memory module doc) to describe the activity-gated per-subject layer and
   reference the budgets.ts constants for per-run and per-brief limits.

No change to `regenerateMemoryBrief`'s signature (the `loadEvents` injection param already exists),
the `subject` template, the `memory_brief_note` schema, the `experimental:memory_brief_refresh`
event, the `global` path, or `scopeHasNewEvidence` itself (it is simply not called for `subject:*`).

## 5. Open implementation questions (for the writing-plans stage)

1. **Budget tuning (BR-9).** The defaults in `BRIEF_REFRESH_BUDGET` (12 subjects, 50 events per brief)
   are tunable constants in `budgets.ts`. Confirm they are sensible defaults from early runs (12 will
   never bind at 3–5 active subjects; 50 events was the hardcoded limit and is proven). If a burst to
   20+ active subjects occurs, the prioritization logic spreads them across nights (no starvation). If
   a subject has >50 recent events, the per-brief read caps at 50 and the brief is summarized from
   that window (no rejection, graceful degradation per the P5.1 pattern). Adjust in budgets.ts if
   needed after Phase B; no other code path hardcodes these numbers (the single-source is respected).
2. **Enroll event action name.** §1/BR-3 list "ingestion enroll" as qualifying activity but the
   exact `action` literal (and whether enroll even writes an `event` row with resolvable knowledge
   ids) must be confirmed against `src/server/ingestion/**`. If enroll has no knowledge-resolvable
   event, drop it from the gate or resolve its subject another way.
3. **Activity floor vs since-last-refresh edge cases.** The scan floor in §3.2 step 2 must be chosen
   so that (a) a subject whose `refreshed_at` is very old still only re-summarizes new evidence
   (not its whole history), and (b) a never-built subject uses the BR-5 30d window. Confirm the
   exact `WHERE created_at > greatest(brief.refreshed_at, now() - lookback)` per-subject predicate
   vs a single global floor + per-subject post-filter (the latter is one cheaper query).
4. **~~How per-subject brief content differs from global~~ — RESOLVED (BR-10).** This was the
   load-bearing risk; it is now a locked decision, not an open question. The subject regen path does
   **not** use the `affected_scopes @> [subject:X]` filter (`loadEventsFromDb` / `scopeHasNewEvidence`)
   — those return ~0 rows / `false` for active subjects because attempt/review never tag `subject:`
   (§1.2). Instead it injects a knowledge-resolved event list via `regenerateMemoryBrief`'s existing
   `loadEvents` param and skips the affected_scopes freshness guard for subject scopes (option **b**).
   Option **a** (tag `subject:` at event-write) is rejected for P5.2 (§6). What remains for
   writing-plans is purely *which knowledge-id of a multi-knowledge event resolves the subject* (BR-4
   already says "first knowledge id"; confirm that matches `batchResolveSubjectIds`' own choice) and
   *whether the subject template prompt needs tuning* for a knowledge-resolved (vs affected_scopes)
   event list — neither is load-bearing.
5. **Regen `batchSize`** (BR-7 / §3.4): keep 1 (serial, correct) or bump to 2–3 (parallel, faster)?
   If bumped, add a concurrent-refresh test asserting no cross-subject clobber.
6. **mem0 interaction (ADR-0017).** Per-subject regen calls `client.search('memory brief
   subject:X', { filters: { scope_key } })` for curated facts. Confirm Mem0 facts are scoped/filtered
   per subject scope_key the same way; if Mem0 has no per-subject facts yet, the subject brief is
   summarized from raw events only (acceptable — ADR-0017 "input mode (b)").
7. **Global brief and `maxEventsPerBrief`.** The global brief is not subject to `maxSubjectsPerRun`,
   but does it also read at most 50 events? Decide at impl: either extend `loadEventsFromDb` to take
   a `limit` parameter and pass `BRIEF_REFRESH_BUDGET.maxEventsPerBrief` for all scopes, or keep
   global uncapped and per-subject capped (the spec does not mandate, per BR-9).

## 6. Non-goals (YAGNI)

- **Not goal-coupled.** Refresh is gated on activity, never on North-Star goal presence (the explicit
  decision vs readiness-brief Option A). Goals may still *soft-bias* what Coach/Dreaming do with a
  brief (ADR-0025 ND-5), but they do not decide *whether* a brief refreshes.
- **Not real-time.** Refresh stays in the nightly batch (BR-8). No read-time / on-demand refresh
  (readiness-brief Option C rejected).
- **No per-subject brief UI.** Briefs remain agent-facing memory; P5.2 adds no surface, route, or
  component.
- **No new brief scopes.** `topic:` / `mistake_cluster:` / `meta:` stay out (and are not permitted by
  `MemoryBriefScopeKey`). P5.2 is `global` (unchanged) + `subject:*` only.
- **No long-term staleness rule.** 30d/90d/N-days-without-evidence expiry is **P5.3**, explicitly
  downstream of this spec.
- **Not tagging `subject:` at event-write (option a).** Extending `computeAffectedScopes`
  (`scope_tagger.ts:38`) to resolve `subject:` from `referenced_knowledge_ids` — which would let
  event-ingest enqueue per-subject regen in near-real-time — is **rejected for P5.2** (BR-10). It
  would force a DB read (`getEffectiveDomain` parent-chain walk) into the deliberately-pure,
  synchronous `writeEvent` hot path, and would only benefit future events (not historical rows).
  P5.2 resolves subjects in the nightly query instead. The event-write tagging is a candidate future
  enhancement (its own ADR if pursued), not part of this slice.

## 7. Acceptance criteria

- **Active subject → its brief refreshes.** A subject with ≥ 1 qualifying activity event
  (attempt/review/capture/enroll) whose knowledge ids resolve to it, created after that subject's
  brief `refreshed_at` (or within the 30d window if no brief exists), is enqueued for regen by the
  nightly sweep, and `regenerateMemoryBrief` upserts its `subject:<id>` row.
- **Dormant subject → skipped.** A subject with zero qualifying activity since its last refresh is
  **not** enqueued; its existing brief row is untouched (no LLM call, no `refreshed_at` bump).
- **Subject brief is non-empty for the common case (BR-10).** A subject made active purely by
  `attempt` / `review` events that carry only `referenced_knowledge_ids` (no `subject_id` in payload)
  produces a **non-empty** brief with a non-null `latest_evidence_at`. A test asserts that
  `loadEventsFromDb('subject:X')` (the affected_scopes path) returns 0 rows for such a subject, while
  the P5.2 knowledge-resolved loader returns the qualifying events — proving the regen path reads the
  resolved list, not `affected_scopes`. A regression test asserts a never-built active subject does
  **not** loop into regenerating an empty brief every night (the BR-10 cost-leak this prevents).
- **Per-run budget enforced (BR-9).** When more than `BRIEF_REFRESH_BUDGET.maxSubjectsPerRun` (default: 12)
  subjects are active-since-last-refresh, the nightly sweep enqueues only the top 12 by activity
  recency; the rest are deferred to the next run (no starvation). A test asserts the sort-by-recency
  logic and the defer behavior under an activity burst (20+ active subjects).
- **Per-brief read budget enforced (BR-9).** Each subject brief regen reads at most
  `BRIEF_REFRESH_BUDGET.maxEventsPerBrief` (default: 50) recent qualifying events via
  `loadEventsFromDb` for summarization. A test asserts that when a subject has >50 recent events,
  the brief is summarized from exactly 50 (the oldest 50, most recent first) and the LLM call
  succeeds (graceful truncation, not rejection).
- **Global brief unchanged.** The `global` brief's refresh path is byte-unchanged: the
  `listStaleBriefScopes` + `scopeHasNewEvidence('global')` behavior and timing are identical
  before/after; `global` is always enqueued (not subject to `maxSubjectsPerRun`); existing
  global-brief tests pass without edits.
- **Parallel refresh, no clobber.** Enqueuing `subject:A` and `subject:B` in the same sweep produces
  two independent regen jobs writing two distinct rows; a test asserts both rows are written with
  correct content and neither overwrites the other (at whatever `batchSize` is chosen).
- **Subject resolution matches the review scheduler.** A unit test asserts `listActiveSubjectsSince
  Refresh` resolves the same knowledge-id→subject mapping as `batchResolveSubjectIds`, including the
  default-profile fallback for orphan ids.
- **Single-source budgets.** `src/server/ai/tools/budgets.ts` is the only place the per-run and
  per-brief limits are defined; no hardcoded fallbacks or per-file constants for these numbers.
- Tests: activity-gate delta (active vs dormant), initial-build window (never-built subject within /
  outside 30d), per-run budget (over-cap → top-N + defer), per-brief budget (>50 events → truncate),
  global-path no-behavior-change regression, parallel no-clobber, and (if BR-3 enroll is included)
  enroll-event qualification.
