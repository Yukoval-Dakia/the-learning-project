## 1. Problem

The brief system (ADR-0015 §2 + ADR-0017) keeps three windows per `scope_key`:
`recent_week_md` / `recent_months_md` / `long_term_md`. The two recent windows are time-bounded
by convention (recent_week ≈ 7d, recent_months ≈ 3mo — **convention, not ADR-codified**), but
`long_term_md` has **no freshness signal**. As supporting SoT evidence ages, a long-term claim can
stay rendered as fresh ground truth indefinitely. Dreaming (T-DR) and the Copilot `/today` drawer
(Layer 8) both read brief text through the `query_memory_brief` DomainTool, so an old-evidence
long-term paragraph is treated as current with no way for a consumer to know it may be dated.

There is no existing demotion/eviction/aging mechanism in either ADR. The only aging mechanics on
record are supersede-via-wholesale-overwrite (each regen recomputes all three windows from scratch,
"upsert 不留历史") and Phase-C whole-scope archival. Per-window long-term staleness was explicitly
deferred out of Wave 1 (T-37 brief-writer-driver). This spec fills that gap.

**Non-goals (deferred — see §11):** per-prefix half-life override, an explicit re-verification task,
per-evidence confidence weighting, the actual UI "may be dated" section, and feeding staleness *into*
the generate prompt (the larger Option-2 design).

## 2. Decision — evidence-decay *freshness score*, surfaced for render-time annotation

Compute a single `long_term_freshness_score` over the long-term paragraph's backing SoT evidence
ages, **store it on `memory_brief_note`**, and **surface it additively** through `query_memory_brief`.
When the brief LLM writer + UI land, Dreaming / Copilot can annotate "Earlier patterns (may be dated)"
at **render time** using the score — **without mutating the paragraph**. The mechanism is exactly:
**compute → store → surface**. There is no row mutation, no archive, no blanking.

This is the readiness brief's Option B *score* without Option B's column-mutation packaging. The
P5.3 value is preserved and is actually cleaner: a quantified freshness signal on the long-term
paragraph so the orchestrator stops treating an old-evidence paragraph as fresh ground truth.

### 2.1 Why scoring, not timestamp eviction or human-in-the-loop

- **Hard timestamp eviction** is too rigid for a learning system — one high-quality event can stay
  true for years; a `now - latest_ts > 90d` cutoff would evict it on a single quiet quarter.
- **Lazy / human-in-the-loop** is too passive — the user rarely reviews their own brief, so dated
  claims accumulate silently and Dreaming keeps citing them at face value.
- **A decay score** balances automation + transparency and respects ADR-0017's evidence_ids-as-SoT
  commitment: the score is derived purely from SoT `event.created_at` timestamps already reachable at
  regen time, it is *interpretable* (score + half-life + count), and it auto-recovers when fresh
  evidence arrives (the next regen recomputes it from the then-current evidence ids).

### 2.2 Why NOT mutate the paragraph (the critical reframe)

An earlier draft of this spec proposed, when the score fell below threshold, **blanking** `long_term_md`
to `''` and **archiving** its text into a new `long_term_stale_claims` JSONB column. **That design is
incoherent and is dropped.** The LLM `generate` step (`params.generate`, `brief.ts:197`) **rewrites all
three windows including `long_term_md` from scratch on every regen**:

- `brief.ts:197` — `draft` (including `long_term_md` + `long_term_evidence_ids`) comes wholesale from
  `params.generate` each regen.
- `brief.ts:203-214` — the row is built by spreading `...draft`.
- `brief.ts:100-122` — `upsertBriefInDb` overwrites all three md windows + all three `*_evidence_ids`
  in both the INSERT `.values(row)` and the `onConflictDoUpdate({ set })` block, every time.

So **no content persists across regens independently of `generate`.** Any post-generate row mutation
(blanking + archiving) would be silently clobbered by the very next regen: the archive would just be a
lagging copy, and the live `long_term_md` would flip back to fresh text on the next run. A demotion
that un-demotes itself one regen later is not a mechanism — it's a bug. The coherent design is
render-time annotation off a stored score, which §7.1 of this spec already described; the
column-blanking idea contradicted §7.1 and is removed.

### 2.3 Production status — ships inert-until-`generate`-lands (honest staging)

P5.3 is **inert in production until the brief LLM writer (T-37 brief-writer-driver) ships.**
`defaultGenerateBrief()` **throws** today (`triggers.ts:329-331`:
`throw new Error('memory brief LLM generator is not configured')`), and `registerMemoryHandlers` is
called in production with only `(boss, db)` (`handlers.ts:50`) so `deps.generateBrief` is undefined and
`generateBrief = deps.generateBrief ?? defaultGenerateBrief` (`triggers.ts:341`) resolves to the
throwing default. **The entire brief-regen path — including this P5.3 score — is exercised ONLY by
injected test fakes and produces no observable production behavior yet.** This is pre-existing: the
brief subsystem was staged ahead of its LLM writer. P5.3 does **not** introduce this; it just lands the
freshness score consistently with the rest of the staged-but-inert subsystem. Reviewers should expect
acceptance to be verified via injected `generate` fakes, not via prod behavior.

### 2.4 Evidence = SoT ids ONLY (ADR-0017 namespace separation, confirmed)

Freshness is computed over `long_term_evidence_ids` only — these are SoT (`event` / `learning_record`)
ids per the ADR-0017 Evidence contract (L38). Mem0 fact-ids are **not** consulted and **not** stored
in any brief column. The score's decay terms come exclusively from SoT `event.created_at`.

### 2.5 No new LLM call

The freshness score runs inside `regenerateMemoryBrief`, **reusing the evidence already fetched at
regen time** (`events` array + at most one batched timestamp lookup, §4.3). It adds zero LLM calls and
zero embedding calls — pure arithmetic over SoT `created_at`, consistent with the existing pure-SQL
gates (`scopeHasNewEvidence`, `listStaleBriefScopes`). The (currently inert) `generate` step and the
embedding-backed `searchFacts` remain the only model-bearing steps; P5.3 touches neither.

## 3. Tunable constants (single source: `budgets.ts`)

Add a new block to `src/server/ai/tools/budgets.ts`, alongside `BRIEF_REFRESH_BUDGET` (the established
"one typed interface + one named const, no class/factory" pattern — `budgets.ts:137-151`). A single
object const, **not** a per-prefix `Record` (per-prefix override is Phase-C deferred, §11):

```ts
// ── P5.3 long-term brief freshness/decay budget ─────────────────────────────
// Single tunable source for the long_term_md evidence-decay freshness score
// (docs/superpowers/specs/2026-05-31-p5.3-long-term-brief-stale-design.md §4).
// Pure arithmetic over SoT event.created_at — no LLM/embedding call.
export interface LongTermFreshnessBudget {
  /** Exponential decay half-life in days. freshness contribution of one
   *  evidence row = exp(-ln(2) * ageDays / halfLifeDays). 60d = balanced
   *  (readiness brief default): a 60d-old event contributes 0.5, 120d → 0.25. */
  halfLifeDays: number;
  /** ADVISORY render-annotation boundary: a stored score below this is the
   *  signal a render-time consumer uses to surface the paragraph as
   *  "may be dated". 0.3 = balanced default. NOT a mutation gate — nothing in
   *  the regen path acts on it; it only informs §7 surfacing. With halfLife 60d
   *  a single-event paragraph crosses 0.3 at ≈104d old. */
  freshnessThreshold: number;
}

export const LONG_TERM_FRESHNESS_BUDGET: LongTermFreshnessBudget = {
  halfLifeDays: 60,
  freshnessThreshold: 0.3,
};
```

Defaults `halfLifeDays = 60`, `freshnessThreshold = 0.3` are the readiness-brief recommendations
(§P5.3 "User decisions needed" → balanced row). These are **autonomous technical choices**, not
user-blocking: single-user self-hosted, backend-only, fully reversible by editing one const, and the
score is stored so the effect is auditable and re-tunable after real data accrues (the P5.1 "revisit
after ~2 weeks of logs" stance). Per-prefix override is deferred to Phase C (§11). Naming both knobs
`LONG_TERM_FRESHNESS_*` keeps them greppable as one unit.

> Decay form note: the readiness brief writes `exp(-days/60)`. We implement the half-life form
> `exp(-ln(2) * ageDays / halfLifeDays)` so the named const `halfLifeDays = 60` means *literal
> half-life* (a 60d event contributes exactly 0.5). Same exponential curve, just with the constant
> chosen so the knob is interpretable; the threshold 0.3 is calibrated against this form. Documented
> so the const name is not misread as a raw `1/τ`.

## 4. Mechanism

### 4.1 Insertion point in `regenerateMemoryBrief` (`brief.ts:182-222`)

The freshness score slots **after `draft` is produced (`brief.ts:197`)** and **alongside the
`latestEvidenceAt` computation (`brief.ts:198-201`)**, feeding **one** new field into the row build
(`brief.ts:203-214`). No demotion, no override of `draft`'s long-term fields — they pass through
verbatim.

```ts
const draft = await params.generate({ scopeKey: params.scopeKey, template, events, facts }); // :197
const latestEvidenceAt = events.length === 0 ? null : new Date(Math.max(...));               // :198-201

// ── P5.3 long-term freshness score (no LLM/embedding call; no row mutation) ──
const evidenceTimestamps = await resolveEvidenceTimestamps(   // §4.3
  draft.long_term_evidence_ids,
  events,
  params,                                                     // lazy: uses loadEventTimestamps OR db, only if needed
);
const { score } = scoreLongTermFreshness(evidenceTimestamps, now, LONG_TERM_FRESHNESS_BUDGET);

const row: BriefRow = {
  id: idForScope(params.scopeKey),
  scope_key: params.scopeKey,
  subject_id: subjectForScope(params.scopeKey),
  ...draft,                              // long_term_md + long_term_evidence_ids UNTOUCHED
  long_term_freshness_score: score,      // NEW column (§5); number | null
  source_event_id: events[0]?.id ?? null,
  latest_evidence_at: latestEvidenceAt,
  evidence_count: events.length,
  refreshed_at: now,
  created_at: now,
  updated_at: now,
};
```

`regenerateMemoryBrief` still returns `{ wrote: true, row }` unconditionally — no freshness
short-circuit. The existing no-op gating lives in the callers (`scopeHasNewEvidence` /
`subjectScopeHasNewEvidence`), unchanged.

### 4.2 Decay function signature

Pure, IO-free, unit-testable (lives in `brief.ts` or a small `brief-freshness.ts` sibling — no DB
import, so it stays unit-test-clean):

```ts
/**
 * freshness = (1 / knownCount) * Σ exp(-ln(2) * ageDays_i / halfLifeDays)
 * over evidence rows whose created_at is known. ageDays = (now - created_at)/86_400_000, floored at 0.
 * score is NULL when knownCount === 0 (no judgeable evidence) — distinct from a scored 0.
 */
export function scoreLongTermFreshness(
  evidenceTimestamps: { id: string; created_at: Date | null }[],
  now: Date,
  budget: LongTermFreshnessBudget,
): { score: number | null; knownCount: number; unknownCount: number };
```

The return `score` is **`number | null`**, matching the nullable column (§5). There is no `stale`
boolean in the return — staleness is an advisory render-time judgement (`score != null && score <
budget.freshnessThreshold`) made by consumers, not a mutation gate computed here.

Edge cases (all unit-tested, §8):

- **No evidence ids** (`long_term_evidence_ids === []`) → `knownCount = 0` → **`score = null`**: an
  empty long-term window has nothing to score (and the `GenerateBrief` test fakes emit `[]`).
- **All timestamps known, all very recent** → `score → 1`.
- **All known, all old** → `score → 0` (low; a render consumer would annotate "may be dated").
- **Mixed known + unknown** → unknown ids are **excluded from both the numerator and `knownCount`**.
- **All ids unknown** (none resolvable, §4.3) → `knownCount = 0` → **`score = null`** (fail-safe: we do
  not fabricate a date and we do not score 0). Under render-annotation framing this just means "never
  annotated dated" — low harm (§4.4 caveat).
- **Denominator** is `knownCount` with the `knownCount === 0 ⇒ null` guard removing any divide-by-zero.

### 4.3 Where evidence timestamps come from (the #1 data-flow risk, addressed head-on)

`draft.long_term_evidence_ids` are **not guaranteed** to all be present in the in-memory `events`
array. The repo's own unit test proves it: `brief.test.ts:24` has `generate` return
`long_term_evidence_ids: ['evt_0']` while `loadEvents` (`brief.test.ts:30-39`) returns only `evt_1`
— so **`evt_0 ∉ events`**. The injectable `params.loadEvents` / `params.generate` seams break any
subset guarantee outright; nothing validates that returned evidence ids are members of `events`. Even
in the DB path the events are biased toward the newest-50 window (`loadEventsFromDb` caps at
`BRIEF_REFRESH_BUDGET.maxEventsPerBrief = 50`, `desc created_at`, `brief.ts:81-89`).

**Resolution — `resolveEvidenceTimestamps(ids, events, params)`:**

1. Build an in-memory map `events[].id → events[].created_at` (zero-cost; already loaded).
2. Partition `long_term_evidence_ids` into **present** (id in the map → use that `created_at`, no
   query) and **missing**.
3. **Lazily** resolve the missing set — and only then. If `missing.length === 0`, do nothing (the
   common DB-path case). If `missing.length > 0`:
   - if an injected `params.loadEventTimestamps` seam exists, call it once with exactly the missing id
     set;
   - else if `params.db` is present, issue **one batched query** `inArray(event.id, missingIds)`
     (`SELECT id, created_at FROM event WHERE id = ANY($ids)`) — no N+1, no LLM, mirroring the existing
     pure-SQL gate shape and the `inArray` import already used in `active-subjects.ts:23`;
   - else (no loader, no db — the injected-`loadEvents` unit-test path) leave them unresolved.

   **Crash fix:** do **not** eagerly construct a `defaultLoadEventTimestampsFromDb(params.db)` when
   `params.db` is undefined (the prior draft's §4.1 did, which would dereference an undefined db on the
   injected-`loadEvents` unit path). Resolution is gated on `missing.length > 0 && (loader || db)`.
4. Any id still unresolved after the lookup (an id `generate` invented, or a `learning_record` id —
   see note) → `created_at: null` → excluded from the score per §4.2.

`loadEventTimestamps` is an **injectable seam** so the unit test drives the decay path with in-memory
timestamps and never touches Postgres (keeping `brief.test.ts` in the unit partition, §8).

> **Scope-bias caveat (documented, not fixed here):** because the loaded window is the newest-50, the
> genuinely-old long-term tail of a high-activity scope is invisible to `generate`, so the ids it cites
> skew recent and the score measures recency-among-the-newest rather than true historical age. Under
> render-annotation framing this is acceptable: the worst case is the paragraph is *never annotated
> dated* (the cited evidence is, if anything, fresher than the true tail) — no false demotion is even
> possible because nothing demotes. Widening the window is a `maxEventsPerBrief` tuning question owned
> by P5.2, not P5.3. Explicitly **unvalidated** until the `generate` writer ships (§2.3).
>
> **`learning_record` ids:** ADR-0017 allows `evidence_ids` to reference `event` OR `learning_record`
> SoT rows. v1 resolves only `event.created_at` (the batched query targets `event`). A
> `learning_record`-only id resolves to `null` → excluded. Because there is **no row mutation now**,
> the consequence is benign: such a scope just may not get a low score, i.e. "never annotated dated" —
> low harm for v1. Resolving `learning_record.created_at` is a small additive follow-up (§11), not
> built speculatively.

### 4.4 Reset / re-confirmation semantics (free, by construction)

Because every regen recomputes the score from the **current** `draft.long_term_evidence_ids` and their
live `created_at`, recovery is automatic and stateless:

- When fresh SoT evidence arrives, the next regen's `generate` cites newer ids → `score` rises.
- A scope that stays quiet keeps re-scoring the same evidence with a slowly-decaying `score` each
  regen (idempotent), so a render consumer can truthfully say "last confirmed N days ago" using
  `latest_evidence_at`.
- There is **no sticky flag and no stored "stale" state** — there is nothing to reset, because nothing
  was mutated in the first place. The paragraph text is always whatever the latest `generate` produced.

## 5. Schema + migration

One new column on `memory_brief_note` (`src/db/schema.ts:278-305`):

```ts
long_term_freshness_score: real('long_term_freshness_score'), // nullable; null = unjudgeable (§4.2)
```

- **Type is `real`, NOT `doublePrecision`.** Two hard reasons:
  1. `real` is the project's pervasive float type (`acceptance_rate` / `ai_score` / `mastery` /
     `weight` / `cost` / `extraction_confidence` all use `real`) and is **already imported** in
     `schema.ts:10`. `doublePrecision` is **not imported** → would tsc-fail unless added.
  2. The `audit:schema` parser's field regex (`scripts/audit-schema-writes.ts:332`) recognizes
     `text|integer|real|jsonb|boolean|timestamp|smallint|bigint|date|numeric|varchar|json|uuid|bytea`
     — it recognizes **`real`** but **NOT `doublePrecision`**. A `doublePrecision` column would be
     **invisible to `audit:schema`**, silently breaking this spec's "reports `live`, no allowlist"
     claim. `real` (4 bytes) is ample for a `[0,1]` score we never do exact arithmetic on.
- **Nullable** (no default). `null` is the "unjudgeable" state (no known backing timestamps,
  `knownCount === 0`, §4.2) and is distinct from a scored `0`.

**`BriefDraft` vs computed:** `long_term_freshness_score` is **NOT** added to `BriefDraft` (it is not
the `generate` LLM's output) — it is **computed in `regenerateMemoryBrief`** and spread into
`BriefRow`. Extend `BriefRow` (`brief.ts:52-62`) with `long_term_freshness_score: number | null`.

**`audit:schema` requirement (mechanical gate) → `live`, no allowlist.** It must appear in BOTH:

1. INSERT — rides in via `.values(row)` in `upsertBriefInDb` (`brief.ts:102-103`) because it is part of
   the `row` object, and the field name literally appears in `brief.ts` (the file-level string scanner
   needs only that).
2. UPDATE — add it to the `onConflictDoUpdate({ set: {...} })` block (`brief.ts:106-120`):
   ```ts
   long_term_freshness_score: row.long_term_freshness_score,
   ```
   satisfying the `.set(` UPDATE path.

→ `audit:schema` reports it `live`; **no `scripts/audit-schema-allowlist.json` entry needed**. (The
pre-existing `manual`-stub allowlist entries for `memory_brief_note.*` are unrelated and untouched;
this spec does not wire the `generate` writer, so they remain valid stubs.)

**Migration.** Run `pnpm db:generate` (drizzle-kit) → emits **whatever the next sequence number is**
(the latest on disk is `0023_lovely_thing.sql`, so it will be `0024_*`, but do not hardcode it — let
drizzle-kit assign and name the file) + sibling `meta/00NN_snapshot.json` + `_journal.json` entry.
Shape (single `ADD COLUMN`):

```sql
ALTER TABLE "memory_brief_note" ADD COLUMN "long_term_freshness_score" real;
```

Tests apply migrations via `pnpm db:migrate` in `tests/global-setup.ts` (shared container) and the
from-scratch `migration-smoke.test.ts` container. Bump `SCHEMA_VERSION` in
`src/server/export/constants.ts` (currently `'4.0'`) and confirm the export round-trip
(`app/api/_/_round_trip.test.ts`) — the column rides the whole-row dump/restore automatically.

## 6. `meta:orchestrator_self` (procedural memory) — score computed, not subject to dated-annotation

The `long_term_freshness_score` is **still computed and stored** for `scope_key ===
'meta:orchestrator_self'` (observability). But procedural memory is **how-to knowledge** (how the
orchestrator should behave / which routes work), not time-bounded evidence about a learner's state. A
how-to heuristic does not "go dated" because no new event referenced it in 60 days. So consumers
**must not** apply the "may be dated" render annotation to the procedural scope — its score is
informational only.

Because there is **no row mutation** in this design, this is **correct-by-construction insurance, not a
fix for a live bug**: **no consumer currently reads the `meta:orchestrator_self` brief row's
`long_term_md`** (Copilot reads `recent_week_md` only — `copilot-summary.ts:173`; Dreaming reads global
brief text). There is nothing today that could mis-annotate the procedural paragraph. The exclusion is
a one-line note for the future render consumer, not a code branch in `regenerateMemoryBrief`.

## 7. Surfacing + consumer back-compat

### 7.1 Render annotation (the intended use of the score)

The "split" between fresh and dated long-term content is realised as a **render-time annotation off the
stored score**, never a structural change to `long_term_md`:

- `long_term_md` is always the latest `generate` output (string, as today).
- When the brief writer + UI land, a render consumer that finds `long_term_freshness_score != null &&
  < freshnessThreshold` (and the scope is not procedural, §6) can present that paragraph under an
  **"Earlier patterns (may be dated)"** heading with the score + `latest_evidence_at` age. This is a
  pure presentation choice made at read time; the SoT row is untouched.

The `BRIEF_TEMPLATES` exact-key assertion in `brief.test.ts:6-12`
(`['global','meta:orchestrator_self','mistake_cluster','subject','topic']`) is **unchanged** — P5.3
adds **no new scope prefix**, so that assertion does not need editing.

### 7.2 `query_memory_brief` surfacing (additive, confirmed no breakage)

- **`query_memory_brief` / `executeMemoryBrief`** (`context-readers.ts:1080-1114`,
  `MemoryBriefOutputSchema:1051-1072`) is a pure passthrough by explicit field mapping. New DB columns
  are silently dropped unless mapped, and `.parse()` strips unknown keys. To expose freshness to the
  orchestrator, make an **additive** extension: add **only** `long_term_freshness_score?: number |
  null` to `MemoryBriefOutputSchema.note` (the always-returned `.note` block at :1052-1064) and map it
  in `executeMemoryBrief` (:1092-1103). Additive optional fields break no existing reader.
  - Note: the bare score on `.note` is fine and **not** evidence-gated. Because we **dropped**
    `long_term_stale_claims` (which would have carried evidence ids), the earlier concern about leaking
    evidence ids into the always-returned `.note` block is gone — a scalar score is not provenance.
- **`long_term_md` stays a non-nullable `z.string()` column** (default `''`, `schema.ts:286`). It is
  never null and is never blanked by P5.3. No consumer breaks.
- **`loadCopilotSummary`** (`copilot-summary.ts:173`) reads **only `recent_week_md`** — never
  `long_term_md`. Untouched. (File path is `src/server/today/copilot-summary.ts`.)
- **`TodayCopilotDrawer`** consumes the derived `brief_global_md` string — no `long_term_md` knowledge.
  Untouched.
- **Scheduling/sweep paths** (`active-subjects.ts`, `brief.ts` stale loops) read only metadata columns
  (`scope_key`, `refreshed_at`, `latest_evidence_at`) — indifferent to the new column.
- **Export** (`export/constants.ts`) whole-row dump — the new column rides along (SCHEMA_VERSION bump
  per §5).

Every reader's contract is intact. The only structure-sensitive parse anywhere (`recent_week_md`
first-paragraph slice in copilot-summary) is not on the long-term path.

## 8. Test plan

**Unit (`brief.test.ts`, already in `fastTestInclude` `vitest.shared.ts:93` — no partition edit; it
imports only `vitest` + `./brief` (`brief.test.ts:1-2`), uses DI seams + an injected
`loadEventTimestamps`, no DB import, stays unit-clean):**

- `scoreLongTermFreshness` table-driven cases:
  - empty ids → `score: null` (knownCount 0).
  - non-empty, all evidence today → `score ≈ 1`.
  - single evidence exactly `halfLifeDays` (60d) old → term `= 0.5`, `score = 0.5`.
  - single evidence ≈104d old → `score ≈ 0.3` (the advisory boundary).
  - all-old → `score → 0`.
  - mixed known + unknown timestamps → unknown excluded from numerator AND `knownCount`; assert
    `knownCount` / `unknownCount`.
  - all-unknown timestamps with non-empty ids → `score: null` (fail-safe).
- `resolveEvidenceTimestamps`: ids all present in `events` → injected `loadEventTimestamps` fake is
  **not** called; some missing → fake called once with exactly the missing id set (batched); id
  unresolved after lookup → `created_at: null`; **no db + no loader + missing ids → no crash**, those
  ids resolve to `null` (the crash-fix regression, §4.3).
- `regenerateMemoryBrief` score persistence (no mutation): inject `generate` returning an old-aged
  long-term draft + a `loadEventTimestamps` fake; assert the upserted row (`upsertBrief` seam) has

## 11. Deferred (Phase C / follow-up)

- **Per-prefix half-life / threshold override** (e.g. `mistake_cluster:*` aggressive, `subject:math`
  conservative) — readiness brief estimates ~4pt; the `LongTermFreshnessBudget` const is shaped so this
  becomes a `Record<BriefScopePrefix, LongTermFreshnessBudget>` lookup without a migration.
- **Explicit re-verification task** — auto-enqueue a "re-confirm this dated pattern" task when a
  paragraph stays low-score for N regens. Out of scope for v1 (no task surface wired).
- **Per-evidence confidence weighting** — weight each decay term by an evidence-quality signal rather
  than equal weights. Deferred (no per-evidence confidence column exists yet).
- **`learning_record.created_at` resolution** — `resolveEvidenceTimestamps` v1 resolves only
  `event.created_at`; add a `learning_record` lookup if those ids appear in real
  `long_term_evidence_ids` (currently excluded → fail-safe, §4.3).
- **The actual UI "Earlier patterns (may be dated)" section** — rendering the dated split in the drawer
  is gated on the pending UI redraw; v1 ships the data (the stored score) + the additive
  `query_memory_brief` field; UI consumes later.
- **Feeding staleness INTO the `generate` prompt** (the larger Option-2 design — let the writer itself
  soften dated claims) — bigger design, deferred.
- **Widening `maxEventsPerBrief`** to reduce the newest-50 recency bias (§4.3 caveat) — owned by the
  P5.2 brief-refresh budget, not P5.3.
