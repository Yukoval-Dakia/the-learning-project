## 1. Problem

Layer 1 (P5.4, merged) closed the **deterministic** rubric floor: `validateProposalQuality`
(`src/server/knowledge/rubric-validator.ts:404`) runs the §4.1/§4.2/§4.3 gates, rejects medium/weak
agent evidence, and folds rejects as a queryable `rubric_verdict` marker
(`proposal-tools.ts:370` `foldRubricRejectedEdge`; `review.ts:405`). T-AR
(`src/server/proposals/signals.ts`) added the accept/dismiss **signal** and a per-`kind` roll-up,
and the Dreaming nightly handler already feeds that roll-up into its prompt as an additive bias.

What is still missing — and what L2 closes — is **learning from the accumulated signal**:

1. **The acceptance feed is rate-only and kind-only.** `getProposalAcceptanceRates`
   (`signals.ts:139`) groups `proposal_signals` purely by `kind` and returns
   `{ kind, accept_count, dismiss_count, total, acceptance_rate }`. Dreaming
   (`dreaming_nightly.ts:129` `proposal_acceptance_rates`) injects the **top-8 kinds' numbers** with
   a one-line bias string (`DREAMING_ACCEPTANCE_RATE_BIAS_GUIDANCE`, `:100`). The agent learns
   *"`knowledge_edge` is accepted 40% of the time"* but **not why** a proposal was dismissed, and
   **not** that quality varies sharply *within* `knowledge_edge` by relation. `knowledge_edge` is a
   single `kind`, yet a `contrasts_with` edge between same-name siblings and a well-evidenced
   `prerequisite` edge have wildly different accept rates — the kind-level rate averages them into
   noise.

2. **Only Dreaming gets the feed.** Coach (`coach_daily.ts:84` `buildCoachInput`) injects
   `active_goals` but **no** acceptance signal at all. Copilot (`copilot/chat.ts:88` `runCopilotChat`)
   passes only `{ surface, triggered_by, user_message, chip_kind }` to `CopilotTask` — **no**
   structured feedback field whatsoever. All three call the same `propose_knowledge_edge` write path
   (`proposeKnowledgeEdgeExecute`, `proposal-tools.ts`), so all three would benefit from the same
   reason-feedback, but only one receives even the rate.

3. **The L1 reject signal is collected but never fed back.** Every L1 reject writes a
   `rubric_verdict { ok:false; gate; reason }` to the propose event (`proposal-tools.ts:390`), and
   each dismiss records a free-form `dismiss_reason` (the user's `user_note`,
   `actions.ts:364` `recordProposalDecisionSignal(db, proposal, 'dismiss', user_note)`,
   persisted to `proposal_signals.dismiss_reason`, `schema.ts:594`). These two streams — *machine
   gates* (why the rubric rejected) and *user reasons* (why the human dismissed) — are exactly the
   "failure modes" an agent could self-correct against, but nothing surfaces them to a prompt and
   nothing uses them to tune the borderline gate.

Net (roadmap `:880`): L2 should "读 `proposal_signals` 调 borderline 门槛 + 把 dismiss/reject reason
喂回 agent prompt + inbox 按接受率排序;纯加性软偏置,不覆盖 L1 硬门". This spec covers the first
two (gate-bias + reason-feedback); inbox accept-rate ordering already exists
(`inbox.ts:218` `proposalAcceptanceRateExpr`, ranked in `loadProposalEvents`) and is out of L2's
new scope (§7).

Three facts about the current state are load-bearing for this design:

1. **Per-`(kind, relation)` is already derivable from `proposal_signals.cooldown_key` — no schema
   change.** Edge proposals build `cooldown_key = knowledge_edge:${from}|${to}|${relationType}`
   (`proposal-tools.ts:82` `edgeCooldownKeys`; `review.ts:355`; test fixture
   `writer.test.ts:68` `'knowledge_edge:k1:k2:prerequisite'`). The `relation_type` is the **last
   `|`-delimited segment**. So a per-`(kind, relation)` roll-up is a pure derive over the existing
   aggregate (the mastery-view precedent `getProposalAcceptanceRates` itself cites, `signals.ts:119`)
   — `relation_type` is NOT a stored column on `proposal_signals` (its columns are `kind`,
   `cooldown_key`, `accept_count`, `dismiss_count`, `acceptance_rate`, `dismiss_reason`,
   `cooldown_until`, `schema.ts:585`) and does not need to be.

2. **`validateProposalQuality`'s signature cleanly takes an optional adaptive input.** It is
   `validateProposalQuality(payload, db, ctx: { isAgent, actorRef })` returning
   `RubricVerdict = { ok:true } | { ok:false; gate; reason }` (`rubric-validator.ts:404`, `:63`).
   Adding an **optional** 4th parameter (or an optional field on a widened ctx) is additive and
   default-omittable, so the pure L1 path (no adaptive input) stays byte-identical and separately
   unit-testable (§4, acceptance §8).

3. **`dismiss_reason` is free-form text, not categorical.** It is the user's `user_note`
   (`actions.ts:57` `user_note?: string`), persisted verbatim to `proposal_signals.dismiss_reason`
   (`signals.ts:242`, `:277`). The digest must summarize free text (top-N recent, truncated),
   which ties to the future embedding option (§6 ML section).

## 2. Locked decision

**L2 = an additive soft layer on top of T-AR + L1.** L1's deterministic hard floor stays **pure**;
L2 reads `proposal_signals` (rates + dismiss reasons) and the `rubric_rejected` event bucket and
produces (1) a per-`(kind, relation)` **feedback digest** injected into Dreaming/Coach/Copilot
prompts, and (2) a per-`(kind, relation)` **gate-bump decision** passed as an OPTIONAL adaptive input
to L1's validator. **L1 alone = deterministic; L1 + L2 input = adaptive.** Core invariant (ND-5,
mirroring the existing Dreaming goal/acceptance feeds): the bias is additive only, never suppresses
L1, and is a no-op on cold start.

**Architecture:** a new `src/server/proposals/adaptive-bias.ts` exporting (a) a per-`(kind, relation)`
acceptance/reason digest reader (an extension of / sibling to `getProposalAcceptanceRates`), and
(b) a `computeGateBump(...) → adaptive input` decision. `validateProposalQuality` gains an optional
adaptive parameter; when omitted it behaves exactly as today (pure L1).

| # | Decision | Rationale |
|---|---|---|
| AB-1 | **New module `src/server/proposals/adaptive-bias.ts`** owns both products: the per-`(kind, relation)` digest (facet A) and the gate-bump decision (facet B). It reads `proposal_signals` (`signals.ts`) + the `rubric_rejected` propose-event bucket (`inbox.ts:85` `isRubricRejected` / the `rubric_verdict` marker). L1's `rubric-validator.ts` stays pure and gains only an optional input param; it does **not** import `adaptive-bias.ts`. | One module owns the soft layer; L1 determinism stays isolated and separately testable (acceptance §8). Mirrors how T-AR put the signal read in `signals.ts` and the bias in the handler, not in the write path. |
| AB-2 (Facet A, PRIMARY) | **Reason feedback to agents.** Add a per-`(kind, relation)` digest = `{ acceptance_rate, accept_count, dismiss_count, top_dismiss_reasons[], top_rubric_gates[] }` injected into the proposal prompts of **Dreaming + Coach + Copilot**. `top_dismiss_reasons` are recent `proposal_signals.dismiss_reason` strings; `top_rubric_gates` are recent L1 `rubric_verdict.gate` values from the `rubric_rejected` bucket. The agent learns the specific *failure mode* (e.g. "your `related_to` edges are dismissed for `related_to_dumping_ground`") and self-corrects upstream. Additive / ND-5 / cold-start (no signal) = no-op, exactly as today's Dreaming feed. | In-context learning by a frontier LLM is the actual "learning" here (§6). The kind-only rate (`signals.ts:139`) is too coarse — failure mode and relation dimension are what let the agent change behavior. |
| AB-3 (Facet B = B1, tighten-only / never-lock) | **Gate adaptation.** When given the optional adaptive input, the validator applies for **borderline (`medium`) evidence** on a `(kind, relation)`: if `acceptance_rate < threshold` **and** the sample is `≥ minSamples`, raise the bar one notch (medium-allowed → strong-required) for that `(kind, relation)`. **Invariants:** (i) NEVER loosens — a learned "user likes X" can NOT unlock an L1 hard gate or lower the evidence floor; (ii) NEVER locks — `strong` evidence always passes, the bar is only ever raised; (iii) cold-start / below-`minSamples` = L1 default (no change). | The only safe direction for an aggregate signal to move a deterministic gate is **stricter on borderline cases**. Loosening would let accept-rate override correctness (anti-pattern); locking would permanently kill an edge the rubric itself says is fine. B1 is the conservative intersection. |
| AB-4 (Facet C, dimension + reach) | **Per-`(kind, relation_type)` signal + Coach/Copilot reach.** The signal is rolled up to per-`(kind, relation_type)` (a new variant of `getProposalAcceptanceRates` that also derives `relation_type` from `cooldown_key`, §1 fact 1), because `knowledge_edge` is one kind but quality varies by relation. The feed extends from Dreaming-only to **Coach + Copilot** (Copilot per-message, within the P5.1 ContextBudget for that surface). | A kind-level rate averages a good `prerequisite` and a bad `contrasts_with` into meaningless noise. All three surfaces share the edge write path, so all three should learn from the same signal. |
| AB-5 (`PROPOSAL_FEEDBACK_BUDGET`) | **Single-source budget in `budgets.ts`** (the P5.1 const+interface pattern) caps the digest injected into prompts: `maxKindRelations`, `maxDismissReasonsPerCell`, `maxRubricGatesPerCell`, `maxChars`. Graceful no-op on cold-start. For Copilot, the digest read is bounded by, and counts against, the per-message `COPILOT_CONTEXT_BUDGET` (`budgets.ts:36`) the same way other reads do. | Bounded prompt cost; consistent with P5.1's single-source `budgets.ts` (the merged file already holds `ContextBudget` + `BriefRefreshBudget`). The digest must never bloat the window it is meant to sharpen. |
| AB-6 | **No schema change; derive per-relation from `cooldown_key`.** Prefer deriving `relation_type` from the existing `proposal_signals.cooldown_key` aggregate (§1 fact 1) over adding a column. ### Rationale vs the readiness-brief / L1 split

The 2026-05-31 two-layer split (L1 spec §2) deferred exactly this layer to YUK-174 and pinned the
coupling contract: L2 "reads rejected-vs-accepted-vs-dismissed per `(kind, relation, cooldown_key)`
off `proposal_signals` (`signals.ts:139`) + the rejected bucket to tune thresholds and feed agent
prompts" (L1 spec RB-9, `2026-05-31-p5.4-rubric-enforcement-design.md:103`). L2 implements that
contract verbatim and adds the facet split (A reason-feedback / B gate-bump / C dimension+reach) and
the no-trained-model rationale (§6) the user asked for. The stable `RubricVerdict` shape L1 froze
(RB-9) is the only contract L2 needs from L1; L2 adds an **optional input** to the validator and does
not change that output shape.

## 3. Design

### 3.1 `adaptive-bias.ts` — the per-`(kind, relation)` digest (Facet A + C)

A new read-only module, sibling to `signals.ts`. It extends the T-AR roll-up with a relation
dimension and the two reason streams.

```ts
// src/server/proposals/adaptive-bias.ts (design — not yet implemented)

export interface ProposalFeedbackCell {
  kind: string;                  // e.g. 'knowledge_edge'
  relation: string | null;       // derived from cooldown_key for edges; null for non-edge kinds
  accept_count: number;
  dismiss_count: number;
  total: number;
  acceptance_rate: number;       // accept_count / total (T-AR semantics)
  top_dismiss_reasons: string[]; // recent proposal_signals.dismiss_reason (free-form), bounded
  top_rubric_gates: string[];    // recent rubric_verdict.gate from the rubric_rejected bucket, bounded
}

export async function getProposalFeedbackDigest(
  db: DbLike,
  budget: ProposalFeedbackBudget,
): Promise<ProposalFeedbackCell[]>;   // cold start (no signal) → []
```

- **Rate + counts (per `(kind, relation)`).** Read `proposal_signals` and `GROUP BY kind` plus a
  derived `relation` extracted from `cooldown_key`. For `kind = 'knowledge_edge'`, the relation is
  the substring after the last `|` (or, defensively, after the last `:` to also match the
  test-fixture-style `knowledge_edge:k1:k2:prerequisite` form — pin the exact delimiter at impl,
  §5). Non-edge kinds carry `relation = null` and roll up exactly like today's per-kind row. This is
  the AB-4 / AB-6 derive: it reuses the same `SUM(accept_count) / SUM(dismiss_count)` shape as
  `getProposalAcceptanceRates` (`signals.ts:145`) with one extra grouping expression — **no new
  column**.
- **Top dismiss reasons.** For each cell, the recent non-null `proposal_signals.dismiss_reason`
  values for that `(kind, relation)`, deduped, truncated to `maxChars`, capped at
  `maxDismissReasonsPerCell` (AB-5). Free-form text summarization is "take the most recent N,
  truncate each" for v1 (§5; embedding-cluster is future, §6).
- **Top rubric gates.** For each cell, the recent `rubric_verdict.gate` strings from the
  `rubric_rejected` propose events on that `(kind, relation)`. The bucket is queryable exactly as L1
  already does (`review.ts:315` `(payload->'rubric_verdict'->>'ok') = 'false'`, keyed on
  `payload->'ai_proposal'->>'kind'` / `->>'cooldown_key'`), so L2 reuses that predicate with a
  relation derive on `cooldown_key`, capped at `maxRubricGatesPerCell`.
- **Cold start.** No `proposal_signals` rows (and no `rubric_rejected` events) → `[]`, identical to
  `getProposalAcceptanceRates`'s honest empty-list cold-start (`signals.ts:151`). Every downstream
  feed degrades to a no-op.

### 3.2 Facet A — how the digest extends the EXISTING Dreaming feed (without duplicating it)

This is the subtle part: Dreaming **already** feeds `proposal_acceptance_rates`. L2 must extend, not
recreate.

**Existing (do NOT recreate):** `dreaming_nightly.ts:129` builds `proposal_acceptance_rates` from
`getProposalAcceptanceRates`, takes the top `DREAMING_ACCEPTANCE_RATE_TOP_N` (8, `:43`), maps each to
`{ kind, acceptance_rate, accept_count, dismiss_count }`, and pairs it with the bias string
`DREAMING_ACCEPTANCE_RATE_BIAS_GUIDANCE` (`:100`). That is **per-kind, rate-only, no reasons**.

**L2 delta (the only additions):**
1. **Swap the reader** `getProposalAcceptanceRates` → `getProposalFeedbackDigest` (the per-`(kind,
   relation)` reader, §3.1). The injected key becomes `proposal_feedback` (or `proposal_acceptance_rates`
   gains `relation`, `top_dismiss_reasons`, `top_rubric_gates` fields — pin the key name at impl,
   §5, but the bias string must be updated in lockstep so the model is told what the new fields mean).
   The per-kind rate is still present (a cell with `relation:null` for non-edge kinds, and the
   edge cells now split by relation), so the existing behavior is a strict subset of the new feed.
2. **Extend the bias guidance string** to additionally say: "for each `(kind, relation)`, the
   `top_dismiss_reasons` and `top_rubric_gates` tell you the specific failure mode — avoid
   repeating it; additive only, never suppress signal-driven proposals (ND-5)". The existing
   ND-5 / cold-start-no-op clause is preserved verbatim.
3. **Bound it** by `PROPOSAL_FEEDBACK_BUDGET` (AB-5) instead of the ad-hoc `DREAMING_ACCEPTANCE_RATE_TOP_N`
   slice (which `maxKindRelations` subsumes; keep `TOP_N` as an alias or fold it into the budget,
   §5).

The Dreaming feed swap is otherwise a like-for-like substitution: same `LoadProposalAcceptanceRatesFn`
dependency-injection seam (`dreaming_nightly.ts:71`, `:84`) so DB tests still inject fixtures, same
"empty → no-op" contract.

### 3.3 Facet A — Coach + Copilot reach (Facet C)

- **Coach** (`coach_daily.ts:84` `buildCoachInput`): add a `proposal_feedback` field to the input
  object (parallel to `active_goals`, `:99`) sourced from `getProposalFeedbackDigest`, and extend
  `COACH_DAILY_OBJECTIVE` / `COACH_WEEKLY_OBJECTIVE` (`:44`/`:45`) with the same ND-5 reason-feedback
  clause. Coach has no acceptance feed today, so this is purely additive; cold-start = the field is
  `[]` and the objective clause is inert.
- **Copilot** (`copilot/chat.ts:88` `runCopilotChat`): the run input today is only
  `{ surface, triggered_by, user_message, chip_kind }` (`:185`). Add a `proposal_feedback` field
  (the digest), bounded by and counted against the per-message `COPILOT_CONTEXT_BUDGET`
  (`budgets.ts:36`, resolved at `:156` via `resolveContextBudget(surface)`) — the digest read goes
  through (or alongside) the existing `ContextBudgetTracker` (`:156`) so a large digest degrades
  gracefully, never bloats the message budget. Copilot's `CopilotTask` system prompt
  (`task-prompts.ts:531` → `tasks.CopilotTask.systemPrompt`) gains the same ND-5 reason-feedback
  clause. Because `promptFromInput` serializes the input object verbatim
  (`runner.ts:181` `JSON.stringify(input)`), the `proposal_feedback` field reaches the model with
  no new plumbing — the same mechanism Dreaming/Coach already rely on.

> **Where the digest is read for Copilot (P5.1 interaction, §5).** Copilot is per-message and
> user-facing; the digest is read once when `runCopilotChat` builds the input (before the agent
> turn), so it is a single bounded read, not per-tool-call. It is capped by
> `PROPOSAL_FEEDBACK_BUDGET.maxChars` and must fit inside the message's `COPILOT_CONTEXT_BUDGET`
> headroom; the exact accounting (count digest chars against `maxExcerptChars` vs a dedicated
> sub-budget) is an impl question (§5).

### 3.4 Facet B — the optional adaptive input to L1's pure validator (B1)

L1's `validateProposalQuality(payload, db, ctx)` (`rubric-validator.ts:404`) gains an **optional 4th
parameter** carrying the gate-bump decision for the proposal's `(kind, relation)`:

```ts
// rubric-validator.ts (design — additive, optional)
export interface AdaptiveGateInput {
  /** True when this (kind, relation)'s aggregate says: raise the bar one notch
   *  for borderline (medium) evidence. Computed by adaptive-bias.ts (AB-3). */
  tightenMediumToStrong: boolean;
}

export async function validateProposalQuality(
  payload: AiProposalPayloadT,
  db: DbLike,
  ctx: RubricValidatorCtx,
  adaptive?: AdaptiveGateInput,   // omitted → pure L1, byte-identical to today
): Promise<RubricVerdict>;
```

**Where it applies (and ONLY there):** the §4.2 evidence-floor check. Today
(`rubric-validator.ts:529`) an agent edge passes only at `level === 'strong'` (plus the
relation-scoped explicit-single-event rescue). L1 already rejects medium for agents, so B's
gate-bump can only ever **fire on a case L1 would otherwise PASS via the rescue path** — i.e. the
`prerequisite` / `contrasts_with` single-event explicit-analysis rescue (`:523`
`explicitSingleEventRescue`). B1 says: if the adaptive input is present and
`tightenMediumToStrong === true` for this `(kind, relation)`, **suppress that rescue** (require a
genuine `strong` instead), with the reject reason annotated as adaptive (`gate:'evidence_level'`,
reason mentions the low acceptance rate). It does **NOT** introduce a new gate string, so the
`RubricGate` enum (`:43`) and the frozen verdict shape (RB-9) are unchanged.

**The three invariants, mapped to code:**
- **Never loosens.** B can only turn a would-pass into a reject (suppress the rescue / raise the
  bar). There is NO code path where `adaptive` flips an L1 reject into a pass. A unit test asserts
  this (acceptance §8): for any payload that L1 rejects with `adaptive` omitted, the same payload
  with ANY `adaptive` value still rejects with the same hard gate.
- **Never locks.** `strong` evidence ignores the adaptive input entirely — the bump only touches the
  medium/rescue branch, so a genuinely strong-evidenced edge always passes regardless of how low the
  `(kind, relation)` acceptance rate is.
- **Cold-start / below-`minSamples` = L1 default.** `computeGateBump` (in `adaptive-bias.ts`)
  returns `tightenMediumToStrong: false` when `total < minSamples` or `acceptance_rate >= threshold`,
  and the call sites pass `undefined` (or the false-bump) when there is no signal — so the validator
  runs the pure L1 path.

**`computeGateBump` (AB-3, in `adaptive-bias.ts`):**

```ts
export interface GateBiasConfig {
  acceptanceThreshold: number;   // below this → tighten (e.g. 0.30; §5)
  minSamples: number;            // sample guard for cold start (e.g. 5; §5)
}

export function computeGateBump(
  cell: ProposalFeedbackCell | undefined,
  config: GateBiasConfig,
): AdaptiveGateInput;             // { tightenMediumToStrong: false } when cell undefined / below guard
```

**Wiring at the call sites.** Each agent write path resolves the digest cell for the proposal's
`(kind, relation)` and passes `computeGateBump(cell, config)` as the 4th arg to
`validateProposalQuality`:
- DomainTool: `proposeKnowledgeEdgeExecute` (`proposal-tools.ts:336`) — it already has
  `relation_type` (`input.relation_type`) and `ctx.db`; resolve the cell, pass the bump.
- Legacy MCP: `writeProposalAfterGate` (`review.ts:376`) — has `edgePayload.relation_type` and the
  tx handle; same resolution. **Performance note:** to avoid a per-proposal digest read, the cell
  can be resolved once per run and passed in (Dreaming/Coach are batch; Copilot is one message) —
  pin the read granularity at impl (§5).

## 4. Migration

Additive; **no schema change, no DB migration, no UI**. L1 stays pure (optional param defaults off).

1. **Create `src/server/proposals/adaptive-bias.ts`** (AB-1) exporting `ProposalFeedbackCell`,
   `getProposalFeedbackDigest` (§3.1), `GateBiasConfig`, `AdaptiveGateInput`, and `computeGateBump`
   (§3.4). Read-only; reuses the `getProposalAcceptanceRates` SUM-over-aggregate shape plus a
   relation derive on `cooldown_key` (AB-6) and the `rubric_rejected` predicate from `review.ts:315`.
2. **Add `PROPOSAL_FEEDBACK_BUDGET` to `src/server/ai/tools/budgets.ts`** (AB-5, P5.1 single-source):
   a `ProposalFeedbackBudget` interface (`maxKindRelations`, `maxDismissReasonsPerCell`,
   `maxRubricGatesPerCell`, `maxChars`) + one named const, same pattern as the merged
   `ContextBudget` / `BriefRefreshBudget` (`budgets.ts:18`, `:136`).
3. **Extend the Dreaming feed (Facet A, §3.2)** — in `dreaming_nightly.ts`, swap
   `getProposalAcceptanceRates` for `getProposalFeedbackDigest` behind the existing
   `loadProposalAcceptanceRatesFn` seam (`:84`), inject the per-`(kind, relation)` digest with reasons
   under the bounded budget, and extend `DREAMING_ACCEPTANCE_RATE_BIAS_GUIDANCE` (`:100`) to describe
   the new reason fields. Keep the empty→no-op contract.
4. **Add the feed to Coach (§3.3)** — `buildCoachInput` (`coach_daily.ts:84`) gains
   `proposal_feedback`; extend `COACH_DAILY_OBJECTIVE` / `COACH_WEEKLY_OBJECTIVE` with the ND-5
   reason-feedback clause. New additive field; cold-start inert.
5. **Add the feed to Copilot (§3.3)** — `runCopilotChat` (`copilot/chat.ts:88`) adds
   `proposal_feedback` to the run input (`:185`), bounded by/counted against `COPILOT_CONTEXT_BUDGET`
   via the existing `ContextBudgetTracker` (`:156`); extend the `CopilotTask` system prompt
   (`task-prompts.ts:531`) with the ND-5 clause.
6. **Add the optional adaptive input to L1 (Facet B, §3.4)** — extend `validateProposalQuality`
   (`rubric-validator.ts:404`) with the optional 4th param `adaptive?: AdaptiveGateInput`; apply it
   ONLY to suppress the medium/rescue pass on the §4.2 floor (`:523`/`:529`). Omitted → unchanged.
   No new `RubricGate` string; verdict shape frozen (RB-9).

## 6. Why not train an ML model

The user explicitly asked to capture this. A trained model has ~no value here and would actively
hurt:

1. **Single-user → tiny, sparse data.** This is a single-user self-hosted tool. Most
   per-`(kind, relation)` cells have **0–5 samples** (the cold-start guard `minSamples` exists
   precisely because cells are usually below it, §5). Cold-start is the *norm*, not the exception. A
   trained model needs volume this domain structurally cannot produce; it would overfit noise or sit
   permanently in cold-start.
2. **The frontier LLM agent IS the learner.** Facet A is **in-context learning**: the agent reads the
   `(kind, relation)` digest (rate + dismiss reasons + rubric gates) and self-corrects in the same
   turn. A frontier model generalizes from a handful of few-shot examples (e.g. "your last three
   `related_to` edges were dismissed as dumping-ground") far better than any small model trained on
   the same 0–5 samples. The "ML" is already there, for free, in the agent.
3. **Auditability (evidence-first invariant).** A deterministic rule — "`acceptance_rate < threshold`
   AND `samples ≥ minSamples` → raise the bar one notch for this `(kind, relation)`" (AB-3) — is
   traceable and reversible: you can read the cell, see the rule fire, and flip it off by config.
   This is the project's evidence-first / reversible-AI invariant (CLAUDE.md / project memory; the L1
   `rubric_verdict` fold exists for the same reason). A black-box model erodes that — you could not
   explain *why* a borderline proposal was tightened.
4. **Anti-overengineering.** A training pipeline (feature extraction, storage, retraining cadence,
   eval harness, model versioning) is massive overkill for a single-user tool whose entire signal
   fits in one small Postgres table (`proposal_signals`). The project's stated principle is to use
   mature OSS for solved problems and not build infrastructure the user did not ask for
   (CLAUDE.md scope discipline; `docs/architecture.md` design principles).

**The valuable "ML" here is therefore:** in-context learning by the LLM (Facet A) + a deterministic
aggregate gate (Facet B). Both are auditable, both degrade to a no-op on cold start, and neither adds
infra.

**Future option (only if volume grows or it goes multi-user):** embedding-cluster the free-form
`proposal_signals.dismiss_reason` text (§5 Q3) to produce a tighter, deduped reason digest (cluster
labels instead of raw recent strings). Note as future work — NOT now; at single-user scale the
"recent N, truncated" digest is sufficient and the embedding infra is not justified.

## 7. Non-goals (YAGNI)

- **No trained model / no RL.** The entire §6 rationale. The learner is the LLM (Facet A) + a
  deterministic gate (Facet B).
- **L2 NEVER loosens or unlocks L1 (B1).** A learned "user likes X" can NOT pass a proposal L1
  hard-rejects, and can NOT lower the §4.2 evidence floor. B only ever tightens borderline cases;
  `strong` evidence always passes. Acceptance §8 pins this.
- **No gate locking.** B never permanently kills a `(kind, relation)`; the bar is raised, not closed.
- **No new heavy infra.** No new table/column (AB-6 — `relation` is derived from `cooldown_key`), no
  new queue, no new cron, no event-log re-scan (AB-7).
- **No change to L1 determinism.** `validateProposalQuality` with the adaptive input omitted is
  byte-identical to today (regression, §8).
- **No inbox accept-rate ordering work.** It already exists (`inbox.ts:218` `proposalAcceptanceRateExpr`,
  ranked in `loadProposalEvents` `:274`); L2 does not touch it. (The roadmap `:880` lists it as part
  of the L2 theme, but it shipped earlier — flagged so this spec does not re-implement it.)
- **No medium-evidence downweighting / confidence-weighted writes.** L1 rejects medium; B keeps it a
  binary reject-or-pass, only making the borderline pass stricter. A weighted-write model is not in
  scope.
- **No UI.** The folded `rubric_rejected` bucket rendering is still the claude-design redraw's job
  (L1 RB-8); L2 adds no surface.

## 8. Acceptance criteria

- **Adaptive bias is provably additive (the core invariant).** A unit test asserts that for any
  proposal L1 hard-rejects with `adaptive` omitted, the SAME proposal with **any** `AdaptiveGateInput`
  still rejects with the same hard gate — L2 can NEVER pass a proposal L1 rejects, and NEVER lowers
  the §4.2 evidence floor.
- **B1 tighten-only + never-lock.** A `(kind, relation)` with low `acceptance_rate` (below threshold)
  and `samples ≥ minSamples` raises the bar (a medium/rescue case that L1 would PASS now rejects with
  `gate:'evidence_level'`). `strong` evidence on the SAME `(kind, relation)` still passes (never
  locks). Cold-start / `samples < minSamples` / `acceptance_rate ≥ threshold` = L1 default, no change.
- **L1 remains pure (regression).** `validateProposalQuality` called with NO adaptive arg behaves
  identically to today; the merged L1 test suite passes unedited.
- **Facet A digest reaches all three surfaces.** A test asserts the per-`(kind, relation)` digest —
  including `top_dismiss_reasons` (free-form) and `top_rubric_gates` (machine gate) — is present in
  the Dreaming input, the Coach input, and the Copilot run input, bounded by
  `PROPOSAL_FEEDBACK_BUDGET`. Cold-start (empty signal) = the field is `[]` and every feed is a no-op
  (Dreaming behavior matches the existing T-AR empty→no-op contract).
- **Per-`(kind, relation)` signal resolves correctly.** A test seeds `proposal_signals` rows for
  `knowledge_edge` with different relations encoded in `cooldown_key`
  (`knowledge_edge:a|b|prerequisite` vs `knowledge_edge:c|d|related_to`) and asserts
  `getProposalFeedbackDigest` returns separate cells with the right relation + rate, while a non-edge
  kind rolls up to a single `relation:null` cell (matching `getProposalAcceptanceRates`).
- **Single-source budget.** `PROPOSAL_FEEDBACK_BUDGET` in `budgets.ts` is the only place the digest
  caps are defined; no per-file fallbacks. A snapshot test locks the constant (mirrors the P5.1 /
  P5.2 budget snapshot tests).
- **No schema change.** No migration; `proposal_signals` columns unchanged. A test (or the
  `pnpm audit:schema` lint) confirms no new write path / column was introduced.
- **Adaptive reject is traceable.** A B-bump reject carries an adaptive-annotated `reason` in the
  folded `rubric_verdict` event (and the DomainTool path's `output_json` log), preserving
  evidence-first.

### Grounding contradictions / nuances flagged (per the locked design)

- **Per-`(kind, relation)` IS derivable WITHOUT a schema change — confirmed.** Edge `cooldown_key` =
  `knowledge_edge:${from}|${to}|${relation_type}` (`proposal-tools.ts:82`, `review.ts:355`), so the

  holds; no column is added. One nuance to pin at impl (§5 Q2): the `writer.test.ts:68` fixture uses
  `:` not `|` — that is a TEST artifact, not the runtime builder; the digest must split on the
  runtime `|` form (and coalesce the symmetric-relation `cooldown_key` variants `edgeCooldownKeys`
  emits into one relation cell).
- **`validateProposalQuality`'s signature cleanly takes an optional adaptive input — confirmed.** It
  is `(payload, db, ctx)` returning a stable `RubricVerdict` (`rubric-validator.ts:404`, `:63`).
  Adding an optional 4th param is additive; the pure-L1 path (param omitted) is byte-identical, so
  the regression invariant (§8) is structurally guaranteed, not just tested.

- **Inbox accept-rate ordering already shipped.** The roadmap `:880` lists "inbox 按接受率排序" under
