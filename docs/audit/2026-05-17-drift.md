## Findings

### ❌ Contradicted（优先）

#### [ADR-0011 §3] ProposeKnowledgeEdge — "mesh 不存 tree 已有边" invariant 未在 propose handler 强制

- **声明**（ADR-0011 §3，ProposeKnowledgeEdge 末尾）:
  > "Mesh 不变量：propose 时若 from→to 已是 tree parent_id 关系，agent 应拒绝（"mesh 不存 tree 已有边"）。这条 invariant **在 propose handler 前置 guard 里强制**；Zod schema 不直接表达（需要查 knowledge 表）。"
- **代码**: `src/server/knowledge/review.ts:125-145`
  — `writeEvent` 直接写入 `ProposeKnowledgeEdge` event，无任何 `knowledge.parent_id` 查询或校验。
  `app/api/knowledge/edges/proposals/[id]/route.ts:165-180`
  — 接受端只做 FK 存在性检查（not archived），同样无 tree-parent 重复检查。
- **冲突**: AI 可以提议一条 `from→to` edge 而 `to` 正好是 `from` 在 tree 中的 `parent_id`，最终被用户接受后写入 `knowledge_edge`，与 tree 骨架形成冗余双重记账——违背 ADR-0010 "tree 是骨架，mesh 是肌肉（互补不冗余）" 核心原则。
- **建议**: 在 `review.ts` 写 `ProposeKnowledgeEdge` event 前，查 `knowledge` 表确认 `from.parent_id ≠ to.id`（且 `to.parent_id ≠ from.id`）。失败时返回 tool error "tree parent edge already covers this relation"，让 agent 改提其他 relation_type。
- `gh issue create`:
  ```bash
  gh issue create \
    --title "Drift: ADR-0011 mesh-not-tree invariant missing from ProposeKnowledgeEdge handler" \
    --label "drift,bug" \
    --body "ADR-0011 §3 says: 'propose 时若 from→to 已是 tree parent_id 关系，agent 应拒绝'. src/server/knowledge/review.ts:125-145 writes ProposeKnowledgeEdge without any parent_id guard. Add guard before writeEvent call."
  ```

#### [ADR-0011 §5] RateKnowledgeEdge — `rollback` action 已定义但无路由实现

- **声明**（ADR-0011 §5，RateKnowledgeEdge schema + 行为描述）:
  > `rating: z.enum(['accept', 'dismiss', 'reverse', 'change_type', 'rollback'])`
  > "`rollback` → 仅 30d 内可用，撤销已 generate 的 edge（写 archived_at）"
- **代码**:
  - `src/core/schema/event/known.ts:294` — Zod enum 含 `rollback` ✅（schema 已定）
  - `app/api/knowledge/edges/proposals/[id]/route.ts:37` — `DecisionBody` 只有 `z.enum(['accept', 'reverse', 'change_type', 'dismiss'])`，**无 `rollback`**
  - 全代码无任何路由能写 `action='rate', subject_kind='knowledge_edge', payload.rating='rollback'`，也无 `archived_at` 置位逻辑
- **冲突**: ADR-0011 明确描述 rollback 的触发条件（30d 内）和副作用（写 knowledge_edge.archived_at），但对应的 API surface 完全缺失。Zod schema 承诺了该行为，路由层无法履约。
- **建议**: 在 edge proposals 路由（或独立 DELETE/PATCH endpoint）补 rollback 分支：30d 时效校验 + 写 `rate` event (rating='rollback') + 事务内 `UPDATE knowledge_edge SET archived_at = now()` where id=edge_id。
- `gh issue create`:
  ```bash
  gh issue create \
    --title "Drift: ADR-0011 RateKnowledgeEdge rollback action defined in Zod but has no route" \
    --label "drift,feature-gap" \
    --body "ADR-0011 §5 defines rollback rating with 30d window and archived_at side-effect. known.ts:294 has rollback in Zod enum. app/api/knowledge/edges/proposals/[id]/route.ts DecisionBody only handles 4 of 5 ratings (no rollback). Add rollback branch with 30d guard + knowledge_edge.archived_at update."
  ```

### ⚠️ Undocumented

#### [src/db/schema.ts:55] `knowledge.approval_status` — enum 含 'pending'/'rejected' 但两个状态不可达

- **代码 reality**: `src/db/schema.ts:55-59` 定义 `approval_status: text({ enum: ['pending', 'approved', 'rejected'] }).notNull().default('approved')`。
  全代码所有写入路径（`src/server/knowledge/proposals.ts:191,284`；`src/server/knowledge/seed.ts:40`）一律硬编码 `approval_status: 'approved'`。无任何代码能将节点状态改为 `'pending'` 或 `'rejected'`。
- **无对应 ADR 决策**: `docs/design/2026-05-15-data-assumptions.md §R2` 仅作诊断（"approval_status UPDATE path：要么实现 admin 流，要么字段降级为 INSERT-only"），无追认 ADR。
- **冲突**: Schema 契约暗示三种审核状态均可用，但代码实际只是 INSERT-only 的单值 flag。未来若有 agent 或 UI 依赖 `pending` 状态做流程控制，将静默失效。
- **建议**: 二选一——(A) 在 ADR 或 data-assumptions 追认 "approval_status 在 Phase 1 语境下固定为 'approved'，pending/rejected 供 Phase 2 多用户场景预留"；(B) 若 Phase 1 内无真实 pending 流程需求，将 enum 缩减为 `['approved']` 并删除死值（避免误导）。
- `gh issue create`:
  ```bash
  gh issue create \
    --title "Drift: knowledge.approval_status enum has unreachable states (pending/rejected)" \
    --label "drift,documentation" \
    --body "schema.ts:55 defines approval_status with ['pending','approved','rejected'] but all write paths hardcode 'approved'. No ADR decides what pending/rejected mean or when they'd be used. Either document the Phase 2 intent or shrink enum to ['approved'] to avoid misleading schema."
  ```
