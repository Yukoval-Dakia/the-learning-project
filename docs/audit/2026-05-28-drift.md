- **声明**: ADR-0020 §1 "三 note type 共用 `body_blocks JSONB` field" + §"Schema 变更" "DROP sections / outline_json / child_artifact_ids / knowledge_id; ADD body_blocks / knowledge_ids / attrs"
- **代码**: `src/db/schema.ts:290-313` — artifact has `body_blocks`, `knowledge_ids: jsonb[]`, `attrs` (post-ADR-0020 schema); legacy columns dropped
- **文档**: `docs/architecture.md:420-447` still describes `sections?` / `outline_json?` / `child_artifact_ids[]` / `knowledge_id?` (singular) on the Artifact entity; `docs/architecture.md:117,235` describe `NoteGenerateTask` output as "atomic artifact sections"
- **冲突**: Architecture doc is the project entrance for the data model but reads as pre-ADR-0020. New readers / agents will write code against a schema that no longer exists.
- **建议**: Rewrite `docs/architecture.md` §七 Artifact block + §五 NoteGenerateTask row to reflect `body_blocks` / `knowledge_ids` (plural) / `attrs`. Pre-YUK-88 baseline audit (F-02) already flagged this; Wave 2 P1 schema landed but doc sweep was deferred.

#### [ADR-0012 vs docs/architecture.md] Knowledge schema still lists dropped mastery fields
- **声明**: ADR-0012 §"决策" — `ALTER TABLE knowledge DROP COLUMN base_mastery / ai_delta_mastery / last_active_at`; replaced by `knowledge_mastery` derived view
- **代码**: `src/db/schema.ts:47-48` explicitly comments fields are DROPped; `knowledge_mastery` PG view at `src/db/schema.ts:673` is the canonical source
- **文档**: `docs/architecture.md:287-289` lists `Knowledge { id, name, domain, parent_id, last_active_at, base_mastery, ai_delta_mastery, ... }` as if these are columns on the table
- **冲突**: Direct contradiction with the table definition and ADR-0012.
- **建议**: In `docs/architecture.md` §七, replace the Knowledge sketch with the actual schema and add a one-line pointer to `knowledge_mastery` view for derived mastery + `last_active_at`.

### ⚠️ Documented-only

#### [ADR-0020 §4] CorrectArtifactEvent: ADR specifies hard rename to `block_id`, code retains legacy `section_id` compat shim
- **声明**: ADR-0020 §4 "correction event payload `section_id?: string` → `block_id?: string`（schema rewrite，无数据 backfill）"
- **代码**: `src/core/schema/event/known.ts:242` defines `block_id: z.string().min(1).optional()` (new field); `src/server/events/artifact-corrections.ts:33-35` retains a compat shim mapping legacy `section_id` payloads to `block_id` at read time; `artifact-corrections.test.ts` has a "legacy section_id projection" test
- **冲突**: ADR said "无 backfill, 数据无生产" implying hard cutover, but a compat shim is present. (low confidence — shim may be acceptable for old test fixtures)
- **建议**: Either (a) drop the `section_id` compat shim in `artifact-corrections.ts` if "无生产数据" still holds, or (b) add a note to ADR-0020 §4 acknowledging the shim was preserved for test fixtures and will be removed at next event-table migration.
