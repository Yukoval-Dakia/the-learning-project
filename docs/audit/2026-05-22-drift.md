#### [Plan 2026-05-18] `learning_record` + `memory_brief_note` — 一等领域表仍无 ADR-0015

- **声明**: `docs/superpowers/plans/2026-05-18-learning-data-loop.md` + 2026-05-20 drift audit 均指出 `learning_record`（user-visible activity context）和 `memory_brief_note`（Dreaming 维护的学习记忆摘要）是与 `event` / `learning_session` 并列的第三类主要持久化实体，需要 ADR-0015 追认。
- **代码**: `src/db/schema.ts:L210`（`learning_record` pgTable）、`L244`（`memory_brief_note` pgTable）；写路径在 `src/server/records/queries.ts:L74`（insert）、`L149`（update）、`L174`（update）；Migration `drizzle/0007_learning_record_loop.sql`。
- **问题**: 连续两次 drift 报告（2026-05-20、本次）指出 ADR-0015 缺失。表已在生产中，Dreaming 链路依赖 `memory_brief_note` 写路径，但设计原则（为什么不用 event、单一所有者约束）没有 ADR 保障。新成员或 Agent 读 ADR 序列时会有盲区。
- **建议**: 起草 ADR-0015，覆盖 `learning_record` 的语义边界（event 是不可变 action log，learning_record 是 user-visible activity context tag）、`memory_brief_note` 的写路径约束（Dreaming 专属写），以及与 ADR-0005 single-owner 原则的对齐方式。
- `gh issue create`:

#### [ADR-0003/0004] `ReviewIntentTask.systemPrompt` 不在 `getTaskSystemPrompt` switch 且无 `// fallback only` 标注

- **声明**: ADR-0003（2026-05-17 修订）+ `docs/superpowers/plans/2026-05-19-profile-blind-ai-tasks-closeout.md`：所有 AI task 的 system prompt 通过 `getTaskSystemPrompt(kind, profile)` 动态渲染；runner 始终调用该函数，从不直接读 `tasks[kind].systemPrompt`（除 default 分支兜底）。其余 12 个有 profile 感知需求的 task 均有 `// fallback only; runtime uses getTaskSystemPrompt(task, profile)` 注释。
- **代码**: `src/ai/task-prompts.ts:303-304` default 分支直接 `return tasks[task].systemPrompt`；`src/ai/registry.ts:L233`—`ReviewIntentTask.systemPrompt` 无 `// fallback only` 注释，也未在 switch 中出现。`VisionExtractTask`（L59）和 `VisionExtractTaskHeavy`（L73）同样不在 switch，但它们是 `invocation: 'manual_rescue_only'` 的多模态任务，语义已自注释。
- **问题**: `ReviewIntentTask` 不在 switch，无 comment，外观上与"未迁移的 task"难区分——与其他 12 个 task 的一致性模式断裂。未来新成员看到"有些 task 没有 `// fallback only`"时无法判断是遗漏还是有意为之。
- **建议**: 给 `registry.ts:L243`（ReviewIntentTask.systemPrompt）加一行注释说明 profile-neutral 原因：`// profile-neutral: generates session intent from queue stats, no subject-specific content`。
- `gh issue create`:
  ```bash
  gh issue create \
    --title "Drift: ReviewIntentTask.systemPrompt 无 profile-neutral 注释，与 12-task 一致性模式断裂" \
    --label "drift,automated" \
    --body "docs/audit/2026-05-22-drift.md §[ADR-0003/0004] 条目。\n\n\`src/ai/registry.ts:L243\` ReviewIntentTask 不在 \`getTaskSystemPrompt\` switch 且无说明注释，与其余 12 个 task 的 \`// fallback only\` 一致性模式断裂。建议加 profile-neutral 原因注释。"
  ```

#### [Plan 2026-05-22 §B] `getTaskSystemPrompt` default 分支静默 fallback — M1 exit criterion 未达成

- **声明**: `docs/superpowers/plans/2026-05-22-math-m1-drift-targets.md §B`（M1 exit criterion）明文要求：将 `task-prompts.ts:303-304` 的 `default: return tasks[task].systemPrompt` 替换为 `assertNever` 编译期检查或显式 warn throw，"current 12 tasks all in switch, default should never be hit"。
- **代码**: `src/ai/task-prompts.ts:303-304` 仍为 `default: return tasks[task].systemPrompt`，静默兜底。
- **冲突**: M1 exit criterion 明文"default 不应被命中，TypeScript 端补强"，但代码未改。此外，M1 plan 写作时计数"12 tasks 全在 switch 内"——实际上 registry 现有 **15 个** task（含 `VisionExtractTask`、`VisionExtractTaskHeavy`、`ReviewIntentTask` 均不在 switch），若直接 `assertNever` 会编译报错，需先处理这 3 个例外（确认 profile-neutral + 加注释，或加入 switch）。
- **建议**: M1 执行时先处理 3 个非 switch task（加 profile-neutral 注释或加入 switch），再将 default 分支改为 `warn` log + fallback（而非 `assertNever`，因为 VisionExtract* 是合法 fallback 使用者）。
