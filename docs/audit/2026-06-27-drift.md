#### [ADR-0046 §3] Rust-in-prod：Dockerfile 无 napi build step

- **声明**: "Dockerfile 要 build 数値核 binary"（ADR-0046 §3，2026-06-26 accepted）。数値核生产路径从今往后是 Rust，不是 TS——反转 YUK-493 滩头「JS 永远 oracle + fallback」立场。
- **代码**: `Dockerfile`（`base / deps / builder / sharpdeps / sdkdeps / sqlitedeps / runner` 六阶段，无 Rust toolchain 安装、无 `crates/calibration-native` build step、无 `.node` overlay）。`crates/calibration-native` 仓库存在但仅本地 dev/CI optional，`scripts/audit-projection.ts` 等都通过 `package.json` 的 skip-if-absent 协约跑，不进镜像。
- **缺口**: ADR-0046 §3 是最新一批 ADR（2026-06-26），接受「单实现+Dockerfile build」但实现层未开工。当前生产仍 TS-only 数值核。
- **建议**: 在 Dockerfile 增 Rust 构建阶段（`FROM rust:... AS rustbuild` + `cargo build --release --features napi`），或走 WASM-in-Node 路（跨 arch 可移植，无 rust toolchain on runner 依赖）——ADR-0046 §Open 「server napi vs WASM-in-Node」仍是 impl 决策，需落地。
- Linear issue：
  - title: `Drift: ADR-0046 ↔ Dockerfile — Rust napi build step absent`
  - labels: `drift`

#### [ADR-0046 §6] `#![forbid(unsafe_code)]` 未加入 crate

- **声明**: "napi 约定——反转后从 dev/CI 惯例升为生产正确性契约"；具体约定之一："`#![forbid(unsafe_code)]`（crate 级）——⚠️ 当前 crate 实测无 unsafe 但尚未加该 lint attribute，待 Rust 线落地"（ADR-0046 §6）。
- **代码**: `crates/calibration-native/src/lib.rs` 开头为注释（`//! YUK-493 Phase -1 — ...`），无 `#![forbid(unsafe_code)]` lint attribute。grep 确认全文件无此属性。
- **缺口**: ADR 自己标记为 pending，但因 ADR-0046 把 napi safety 约定升为「prod 正确性契约」，此 lint 应在首个 Rust-in-prod lane（A13 proper-scoring kernel）落地前加入，防后续 PR 意外引入 unsafe。
- **建议**: 在 `lib.rs` 首行加 `#![forbid(unsafe_code)]`，与 `#![deny(clippy::all)]` 并列；在 CI（`pnpm test` 或独立 `cargo clippy`）验证。
- Linear issue：
  - title: `Drift: ADR-0046 §6 ↔ crates/calibration-native — #![forbid(unsafe_code)] missing`
  - labels: `drift`

#### [ADR-0044 §5] Worker 进程未调 `registerCapabilityCopilotTools`

- **声明**: ADR-0044 §5 明确记录为「真·残留缺口」——"worker 进程靠 `bootstrap.ts` 的 `CORE_TOOLS`（41 条 superset）兜底，未调 `registerCapabilityCopilotTools`"；fix 已钉："start-worker.ts 加一行对齐 app（折进 #47 首发 lane）"。
- **代码**: `server/index.ts:50-53` 调 `registerCapabilityCopilotTools(capabilities)`；`src/server/boss/start-worker.ts` 无此调用（grep 空输出）；`scripts/worker.ts` 的 `main()` 只动态 import `startBossWorker`，同无此调用。`bootstrap.ts:68` 的 `CORE_TOOLS` 硬编码列表仍作为 worker 侧唯一来源。
- **缺口**: manifest 贡献制对 worker 进程的「工具面真相源」不生效——新增 copilotTool（如 capability manifest 增工具）需同步更新 `CORE_TOOLS` 硬列表才生效于 worker，否则 worker 里工具面滞后。ADR-0039 §代价/风险 也指出「copilotTools 贡献制实质 no-op、CORE_TOOLS bootstrap 全量先到」是让 D14 单编排者收口的阻断前置。
- **建议**: 在 `src/server/boss/start-worker.ts`（`startBossWorker` 函数入口）加 `await registerCapabilityCopilotTools(capabilities)` 调用，对齐 `server/index.ts:53` 模式。
- Linear issue：
  - title: `Drift: ADR-0044 §5 ↔ start-worker.ts — registerCapabilityCopilotTools not wired`
  - labels: `drift`

#### [ADR-0039 §4] `judge_retraction` / `archive` / `defer` 仍在 inbox 裁决面

- **声明**: "C 档（纯状态不进队列）：本就不是结构写入提议，移出裁决面。`defer → snooze 控件` / `archive → 直接软归档` / `judge_retraction → agent-notes 旁观`。这同时解决「三 kind 只能 dismiss」死占位（YUK-44）——不是补 accept applier，是它们本就不该在裁决面"（ADR-0039 §4）。
- **代码**: `src/capabilities/shell/ui/inbox-api.ts:32-33` 仍将 `archive` 与 `judge_retraction` 列入 inbox kind map（label + icon + tone）；`src/capabilities/practice/manifest.ts:357` 仍注册 `{ kind: 'judge_retraction' }` 进 proposal kinds；`inbox-api.ts:45` 注释自述 "judge_retraction 未实现 accept（400 unsupported_proposal_kind，归 YUK-44）"，即 accept applier 死路仍存在。
- **缺口**: ADR-0039 §4 明确「三 kind 本就不该在裁决面」，当前 UX 和 manifest 仍把它们当普通 proposal kind 塞进 inbox。用户看到无法 accept 的 inbox 项，只能 dismiss，与 ADR intent（snooze 控件 / 软归档 / agent-notes 旁观）不符。YUK-44 是记录的上游 blocker。
- **建议**: 按 C 档语义将三 kind 从 inbox 面移除：`defer → snooze 控件`（UI 交互层）；`archive → 直接软归档调用`（不经 inbox 裁决）；`judge_retraction → agent-notes / 只读旁观面`。同时清理 manifest 里这三 kind 的 proposal 注册，对齐 ADR-0039 §4。
- Linear issue：
  - title: `Drift: ADR-0039 §4 ↔ inbox-api / manifest — judge_retraction/archive/defer still in inbox lane`
  - labels: `drift`
