## Findings

### ❌ Contradicted（优先）

#### [ADR-0011 §落地步骤 Step 3] `writer.ts` "已经 polymorphic 写" 与现实不符

- **声明**（ADR-0011 §落地步骤 Step 3）:
  > "`src/server/events/writer.ts` 不变（已经 polymorphic 写）"
- **代码**: `src/server/events/writer.ts` 写入 `job_events` 表（Sub 0c SSE plumbing，见第 1 行 import），对 domain `event` 表零引用——该表尚未存在于 `src/db/schema.ts`
- **冲突**: ADR 断言 writer 已完成领域事件的多态写入、Step 2（Zod）落地时无需改动。实际上 `job_events` 是异步 job 的 SSE 通知管道，与 ADR-0006 v2 的 domain `event` 表是两个不同关注点。Step 1（event 表落地）后需要从头建立 domain event 写入路径，届时 "不变" 指导将直接误导实现者。
- **建议**: 修订 ADR-0011 §落地步骤 Step 3 为：
  > "Step 1 落地后，新建 `src/server/events/domain_writer.ts`（或在 Step 5 LearningSession 模块内）负责 `Event.parse(...)` + 写入 domain `event` 表；现有 `src/server/events/writer.ts` 服务 job_events SSE 管道，与 domain event 不同关注点，两者共存。"
- `gh issue create`:
  ```bash
  gh issue create \
    --title "Drift: ADR-0011 Step 3 'writer.ts 不变' misleads — domain event writer doesn't exist yet" \
    --label "documentation,drift" \
    --body "ADR-0011 §落地 Step 3 says src/server/events/writer.ts is 'already polymorphic' for domain events and needs no change. But the file only writes job_events (Sub 0c SSE plumbing). When Phase 1c.1 Step 1 (event table) lands, a domain event writer must be built from scratch. Update ADR-0011 Step 3 to: 'create src/server/events/domain_writer.ts (or within Step 5 LearningSession module) for Event.parse() + domain event INSERT; existing writer.ts stays for job_events SSE.'"
  ```

### ⚠️ Undocumented（代码做了 ADR 没追认）

#### [ADR-0011 §完整 KnownEvent union 扩展] `z.discriminatedUnion` 示例代码错误，实现用 `z.union` 未追认

- **代码**: `src/core/schema/event/known.ts:287` 使用 `z.union([...])` 而非 `z.discriminatedUnion('action', [...])`；注释说明原因——`'generate'`、`'rate'`、`'propose'` 三个 action 键各在两个分支中出现（`generate` × {artifact, knowledge_edge}；`rate` × {event, knowledge_edge}；`propose` × {knowledge, knowledge_edge}），Zod `discriminatedUnion` 要求判别键 literal 全局唯一，此处三组碰撞，无法满足
- **ADR-0011 现状**: §完整 KnownEvent union 扩展 示例写的是 `z.discriminatedUnion('action', [...])`——该示例代码技术上是错误的（会在 Zod 内部抛出），从未能编译运行
- **冲突**: ADR-0011 给出错误 Zod API，实现者选择了正确方案（`z.union`）但未向 ADR 追认，形成 ADR 文档与实现永久不一致
- **建议**: 更新 ADR-0011 §完整 KnownEvent union 扩展 示例代码：把 `z.discriminatedUnion('action', [...])` 改为 `z.union([...])` 并加注：
  > "action 键在 generate / rate / propose 三对分支上各有碰撞，`z.discriminatedUnion` 不支持，改用 `z.union`；parse 时以 subject_kind 作 secondary 区分（每个分支已用 `z.literal()` 锁住 subject_kind）"
- `gh issue create`:
  ```bash
  gh issue create \
    --title "Drift: ADR-0011 KnownEvent shows z.discriminatedUnion — technically invalid, code uses z.union" \
    --label "documentation,drift,low-priority" \
    --body "ADR-0011 §完整 KnownEvent union 扩展 shows z.discriminatedUnion('action', [...]) for KnownEvent. This is technically invalid: 'generate', 'rate', 'propose' each appear in 2 branches (action key not globally unique). Code (known.ts:287) correctly uses z.union with a comment. Update ADR-0011 example to z.union + note: 'discriminatedUnion not possible due to 3 action-key collision groups; subject_kind serves as secondary discriminator.'"
  ```
