#### [src/capabilities/agency/server/meeting/director.ts, jobs/research_meeting_agent_nightly.ts] Agent-led nested-subagent meeting lane landed with no ratifying ADR
- **声明**: no ADR — every other background-agent job in the repo uses the deterministic `runAgentTask(kind, input, {...})` single-task registry pattern.
- **代码**: `src/capabilities/agency/server/meeting/director.ts` + `director-tools.ts` implement a `query()`-based director agent with SDK-native `agents: { 'evidence-scout': AgentDefinition }` nesting (Task-tool spawning) — the first "director + nested subagent" orchestration pattern in the codebase. Registered live as a cron job in `src/capabilities/agency/manifest.ts:103` (`name: 'research_meeting_agent_nightly'`), gated by `RESEARCH_MEETING_AGENT_ENABLED` (default OFF) but present in production job infrastructure, running as a shadow lane alongside the existing deterministic `research_meeting_nightly` job.
- **冲突**: introduces a new class of AI-orchestration mechanism (multi-agent nested spawning) that departs from the repo's established single-task-registry pattern for every other background loop — the design doc (`docs/design/2026-07-06-yuk572-agent-meeting-lane-spec.md`) itself flags this as a departure. ADR-0049 (conjecture-wire) is the nearest related ADR but only scopes probe producer/consumer wiring, not this meeting/evidence-gathering mechanism. No ADR mentions "director," "evidence-scout," or nested-agent orchestration.
- **建议**: write an ADR (or amend an orchestration-pattern ADR if one is later created) ratifying nested-subagent orchestration as a sanctioned pattern, scoping when it's appropriate vs. the existing `runAgentTask` registry pattern, so future jobs don't reinvent this ad hoc.
- Linear issue: title `Drift: agent-meeting-lane nested-orchestration ↔ no ADR`, label `drift`.

- **声明**: ADR-0046 §4 "新数值 kernel 一律 Rust-first：无 TS oracle，直接 Rust 实现" and explicitly names "A13 proper-scoring kernel = 首个 Rust-first 试点（净新、边界干净、不碰 live θ̂）" — i.e. no TS oracle, straight to Rust, as the flagship proof of the ADR's reversed posture.
- **代码**: `src/server/conjectures/scoring.ts:1-49` (`scorePrediction`) is a TS placeholder — its own header comment states "ADR-0046: proper-scoring = Rust-first single-source-of-truth. This TS is a PLACEHOLDER stub — bit-exact replacement lands with the Rust calibration kernel." `crates/calibration-native/src/` contains only `lib.rs`; no brier/log-loss/proper-scoring Rust code exists yet. `docs/design/2026-06-27-a13-ts-half-design.md:6` explicitly scopes the Rust kernel as "OUT" of the current A13 half, deferring it.
- **冲突**: what shipped is the exact TS-first-with-Rust-follow-up pattern ADR-0046 says it eliminated for new kernels, on the specific example the ADR names as its flagship pilot. Low risk in practice — the stub is self-documented as temporary scaffold, log-only, never flips a label — but the letter of "A13 = first Rust-first pilot, no TS oracle" is not yet true in code.
- **建议**: either land the Rust proper-scoring kernel to close the gap, or add a status note to ADR-0046 acknowledging A13 currently ships as a TS scaffold-then-port (per §5's allowance) rather than a from-scratch Rust-first implementation, so the ADR text doesn't overstate current state.
