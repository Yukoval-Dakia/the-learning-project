#### [ADR-0029 D6] judge event capability_ref.version 硬编码未修复

- **声明**: `docs/design/2026-06-04-u0-decisions.md §D6`：「修掉 capability_ref.version 硬编码 '1.0.0'（question-contract.ts:92,239 / steps-judge.ts:10），改读 SubjectProfile.version。」ADR-0029 §决定5 同义：「`capability_ref.version` 停止硬编码改读 `SubjectProfile.version`」。
- **代码**:
  - `src/server/ai/judges/question-contract.ts:99` → `capability_ref: { id: route, version: '1.0.0' }`（硬编码）
  - `src/server/ai/judges/question-contract.ts:180` → `const capability_ref = { id: 'semantic', version: '1.0.0' }`（硬编码）
  - `src/server/ai/judges/steps-judge.ts:10` → `const CAPABILITY_REF = { id: 'steps', version: '1.0.0' }`（硬编码）
- **冲突**: ADR-0029 D6 明确要求从 SubjectProfile/capability registry 读版本；三处生产代码仍静态写死 `'1.0.0'`，judge event 的 `capability_ref.version` 无法随 profile 版本演化，历史判分的版本上下文重建目标（ADR-0029 D6 原话：「历史判分的版本上下文从此可重建」）实际未达成。
- **建议**: 在 judge 调用处注入 `SubjectProfile.version`（或从 capability registry 解析已注册的 CapabilityManifest.version），将硬编码 `'1.0.0'` 替换为运行时读取。

- **代码**: `src/server/ingestion/workflow-judge-config.ts:37-80` 新增第三个 flag `WORKFLOW_JUDGE_OBSERVE_ENABLED`（Strategy D Slice B, YUK-190）：flag OFF（默认）下仍跑 TaggingTask + WorkflowJudge 并写 audit event，但不写任何 domain 行；`autoEnrollEnabled` 的 OFF 从「硬 no-op」变为「observe mode」，只有 `WORKFLOW_JUDGE_OBSERVE_ENABLED=false` 才完全关闭。
