## Architecture Assessment

The current frame is broadly right: `Question` as source of truth, event stream as audit/provenance layer, `learning_session` as envelope, `knowledge` tree + typed mesh, pg-boss for durable background work, and proposal-only destructive AI actions are strong foundations.

The weak point is generalization. Several prompts hardcode śĖáŤ®ÄśĖá examples and behavior (`NoteGenerateTask`, `VariantGenTask`, `TeachingTurnTask`, `KnowledgeReviewTask`). That is fine for Phase 1, but not for a multi-subject framework. Add an internal `SubjectProfile` layer before the second subject: subject id, language/style policy, canonical examples policy, question kinds, judge routing, source/grounding policy, artifact templates, and prompt fragments.

Learning Intent is too narrow for ‚ÄúAI-driven‚ÄĚ: `src/server/orchestrator/learning_intent.ts` only supports case 3c, where the topic node and children already exist. Real users will say ‚ÄúśąĎśÉ≥Ś≠¶ X‚ÄĚ before the graph exists. The next version should support case 3a/3b as proposal flows: propose root/children, preview the plan, then accept materializes knowledge + hub/atomic items + notes.

The assessment engine is still thin. For math, physics, chemistry, writing, English, coding, and open-ended humanities, exact/keyword plus one attribution task is not enough. The framework needs a real judge router: deterministic checks, semantic/rubric checks, step/proof checks, code/test execution, multimodal handwriting/diagram checks, appeal/rejudge, and explicit uncertainty.

The AI layer is currently a set of useful pipelines, not yet a coherent AI planner. A genuine AI-driven product needs a unified decision queue/inbox with typed proposals, cooldowns, acceptance-rate feedback, rollback semantics, and ‚Äúwhy now‚ÄĚ evidence across review, learning items, notes, graph maintenance, variants, and coaching.

Avoid letting `event` become an unbounded junk drawer. Keep promoting stable experimental actions, add entity-specific projection/read models where UX needs them, and add retraction/rollback semantics for bad imports, wrong variants, bad notes, and user-disagreed AI judgments.
