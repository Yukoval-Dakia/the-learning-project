#### [CLAUDE.md §Layering + 8 ADRs] `src/capabilities/` module tree not captured in architecture docs

- **声明**: CLAUDE.md Architecture/Layering section declares `server/   # Server-only: ai/, ingestion/, knowledge/, review/, export/, r2.ts, http/`. Eight ADRs (0002, 0005, 0008, 0024, 0026, 0028, 0030, plus implicitly 0013) reference paths under `src/server/ingestion/*`, `src/server/review/*`, or `src/server/knowledge/*`.
- **代码**: `src/capabilities/` directory contains fully realized sub-trees: `ingestion/server/*`, `practice/server/*` (review), `knowledge/`, `agent-notes/`, `notes/`. The old flat paths (e.g. `src/server/ingestion/workflow-judge.ts`) do not exist; actual files live at `src/capabilities/ingestion/server/workflow-judge.ts`, `src/capabilities/practice/server/due-list.ts`, etc.
- **冲突**: `src/server/` no longer contains `ingestion/`, `knowledge/`, or `review/` — those have migrated to a `src/capabilities/*/server/` layout. Neither CLAUDE.md nor any ADR acknowledges this structural reorganization.
- **建议**: Update CLAUDE.md §Layering to document the `src/capabilities/<name>/{api,server,ui,jobs}/` pattern and add a note that `src/server/` retains cross-cutting modules (ai, copilot, memory, events, session, etc.). Add a one-liner to the affected ADRs noting the path migration, or create ADR-0031 to ratify the capabilities layout.

#### [Design doc 2026-06-09 §Carriers] ToolUseCard 5th state name mismatch

- **声明**: `docs/design/2026-06-09-copilot-presentation-layer.md` §Carriers lists ToolUseCard as having "5 states (running/done/empty/failed/waiting)".
- **代码**: `src/ui/primitives/ToolUseCard.tsx:L39` — `export type ToolUseStatus = 'running' | 'done' | 'empty' | 'failed' | 'awaiting-approval'`. The component's own inline comment (L13) also reads "running / done / empty / failed / waiting" — inconsistent with its own type on L39.
- **冲突**: The 5th state is implemented as `'awaiting-approval'` in the TypeScript type; the design doc and the component's own comment both say `'waiting'`. Consumer code must use `'awaiting-approval'` to match, making the design doc name stale.
- **建议**: Either rename the type to `'waiting'` (aligns with spec and inline comment) or update the design doc and L13 comment to say `'awaiting-approval'`. Prefer consistency between type and comment — update L13 in the component to match the live type.
- **Linear issue**:
  - title: `Drift: Design doc 2026-06-09 ↔ ToolUseCard status type ('waiting' vs 'awaiting-approval')`
