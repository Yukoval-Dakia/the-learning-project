### 0.3 术语 cheat sheet（详见 `CONTEXT.md`）

`event` = 单一 action log（actor × action × subject × outcome × caused_by），ADR-0006 v2。
`ActivityRef { kind, id }` = 取代 `question_id`，ADR-0014。
`LearningActivity` = 框架核心抽象；`question` 是 ActivityKind 之一。
`Capability` = judge / renderer / scheduler 三类，注册制，跨学科共享。
`SubjectProfile` = 纯数据 ~50 行/学科，声明用哪些 capability。
`Artifact` = AI 产出物，分 Note (hub/atomic/long，三态共用 body_blocks block tree，ADR-0020) 和 Tool (quiz/...)。
`Proposal` = AI 软建议，走 Inbox lifecycle（propose → accept/dismiss/rollback）。
`AI actor` = `actor_kind ∈ {user, agent, cron, system}`，AI 与用户对等。

## §1 项目定位

**一句话**：单用户、自托管（NAS）、event-driven 的 **多学科 AI 学习框架**——AI 不是聊天助手，是与用户对等的 first-class actor，所有主动行为留痕、可审计、可回滚。

### 1.1 三档承诺（A/B/C/D）

| 档 | 含义 | 状态 |
|---|---|---|
| A | 错题复习闭环 | ✅ Phase 2A Review Orchestrator |
| B | "我想学 X" 路径生成 | ✅ Phase 2B Learning Intent（仅 case 3c；3a/3b 缺位） |
| C | AI 主观能动（dreaming / maintenance / proactive proposal） | 🟡 Maintenance cron 已上；Dreaming ⬜；Global Coach ⬜ |
| D | first-class equal actor（跨 routes 的 Copilot + tool-use 可见 + event 平等 mirror） | ⬜ Copilot DomainTool Registry 未建 |

### 1.2 显式 Non-Goal（v0.3 §6 + ADR-0007 + architecture.md §六"反模式"）

- 公共 MCP server / plugin marketplace
- 外部 MCP 消费（除非真有客户端需要，且内部产品 loop 已稳）
- 多用户 / per-user auth（ADR-0007）
- 跨预算 LLM 调度执行（`maxCost / fallbackChain` 设计预留，运行时未启用）
- 多端原生 app（RN / Flutter / Tauri）
- 聊天框形态的 LLM 抽象（按 task 抽象，不按 chat() 抽象）
- 不自建 tool-calling 循环（用 Claude Agent SDK）
- 不嵌 Obsidian 当 note 框架
- 不抽通用 Tool interface（YAGNI，等第二种 tool kind 出现）

### 1.3 触发"重新评估 Non-Goal"的条件

- v0.3 §6 + ADR-0014 触发条件：profile 数量 >10 且 ~80% capability 重复 → 考虑 profile 模板机制
- ADR-0014 触发条件：capability 版本升级 fixture 维护负担 > 编写本身 → 简化版本策略
- ADR-0014 触发条件：跨科目 mapping table 膨胀且无人维护 → 重新考虑 universal base + extension 模型
- ADR-0014 触发条件：question_part FSRS 独立调度导致碎片化 → 考虑 parent-level 聚合
- ADR-0016 触发条件：3 个 uncertainty spike 全 pass → 引入 OpenAI Codex 作 alternate provider，强制 ADR-0003 revision + 设计 protocol adapter
- v0.3 §Later：内部产品 loop 稳定 + 真有外部客户端 → 重新考虑公共 MCP server

### 第 1 层 — Event 骨架 + Correction Event

**定义**：`event` 表是单一 action log；`actor_kind ∈ {user, agent, cron, system}`；append-only DAG via `caused_by_event_id`；payload 按 Zod discriminated union 守；retract / supersede 走 `correct` action 而非 mutate。

**关键 ADR**：0006 v2 / 0011 / 0014 §6 (Correction Event)。

**当前状态**：
- ✅ 11 个 KnownEvent discriminated union 已落地（`src/core/schema/event/`）
- ✅ `writeEvent()` from `src/server/events/queries.ts` 唯一写入点
- ✅ `experimental:*` 命名空间机制
- ✅ correction event + projection / UI 撤回 / 标错 / 恢复 入口（Foundation C ✅）
- ✅ `caused_by_event_id` DAG
- ✅ FSRS 投影 `material_fsrs_state` 从 event 流派生
- ✅ `event.affected_scopes` text[] 新列（ADR-0017）

**完全体目标**：所有 AI / 用户 / cron / system 动作都通过 event 写入；experimental:* 周期性 promote 到 KnownEvent + 数据迁移；correction event 是唯一撤回机制。

**Gap**：
- experimental:* promotion lifecycle 缺标准化流程（ADR-0011 提了 "3 个 tool + 2 周 stable" 但不普适）

### 第 5 层 — Artifact 多态（Note + Tool）

> **2026-05-26 update（YUK-88 ADR-0020 拍板）**：Note 三态从"hub + atomic"扩为"hub + atomic + long"；**三态共用 `body_blocks JSONB` schema**（DROP `sections[]` / `outline_json` / `child_artifact_ids[]`）；mark-wrong anchor 从 section_id 改为 block_id（ADR-0019 → ADR-0020 同日 supersede）。下方"完全体目标"与"Gap"已重写。

**定义**：Artifact 是 AI 产出物的统一抽象，分两族：
- Note（阅读型，被读）：`note_hub` / `note_atomic` / `note_long`，**三态共用 body_blocks block tree schema**；差异仅在 verifier 约束 + AI prompt + 渲染 idiom
  - `note_atomic`：单 knowledge 节点的"简介卡"，verifier 强制 5 `semantic_kind`（definition / mechanism / example / pitfall / check）至少各 1 个 block；不限节点位置（叶子 / 中层都可，mesh 视角下叶子概念已 obsolete）
  - `note_long`：跨多个 knowledge 节点 / 主题的合成笔记；无 semantic_kind 约束
  - `note_hub`：主题入口；verifier 强制 ≥ 3 个 cross_link block；含"手动区 + 自动区"双区（自动区由 nightly worker 维护，iii-curated mesh 包含判定）
- Tool（互动型，被用）：`tool_quiz`（当前唯一实例）+ `tool_visualizer` / `tool_simulator` / `tool_drill`（Phase 3 候选）

**关键性质**：

3. 不抽通用 Tool interface（YAGNI，等第二种 tool kind 出现）
4. **Artifact ↔ Knowledge = label 模型**：`artifact.knowledge_ids text[]`（plural），artifact 必须挂 ≥ 1 个 knowledge（没有就走 propose_knowledge）；atomic 数组长度 = 1（节点简介），long / hub 可 N
