# v0.3 Generalized AI Learning Framework

**Status**: planning source of truth for post-rescue architecture work.
**Date**: 2026-05-18.
**Scope**: internal product architecture only. Do not introduce external plugin, Skill, or standalone MCP ecosystems for v0.3.

## 1. Why This Exists

The current system is a strong Phase 1 implementation for classical Chinese: `Question` is the source of truth, `event` gives provenance, `learning_session` wraps activity, `knowledge` + `knowledge_edge` model the graph, pg-boss handles durable background work, and AI destructive actions are proposal-only.

The next risk is false generality. Several runtime prompts still encode wenyan assumptions directly, and AI behavior is a set of pipelines rather than a coherent learning decision system. v0.3 turns the product into a generalized, AI-driven learning framework without expanding into external plugin infrastructure.

## 2. Core Additions

### SubjectProfile

Every subject must declare its learning semantics through an internal profile instead of hardcoding subject behavior in task prompts.

Minimum profile shape:

```ts
interface SubjectProfile {
  id: string;
  displayName: string;
  languagePolicy: {
    uiLanguage: 'zh-CN' | 'en' | string;
    explanationStyle: 'exam' | 'conceptual' | 'socratic' | 'project';
  };
  questionKinds: string[];
  judgePolicy: {
    defaultByQuestionKind: Record<string, JudgeRouteKind>;
    allowAppeal: boolean;
    uncertaintyThresholds: { borderlineLow: number; borderlineHigh: number };
  };
  sourcePolicy: {
    preferredSources: Array<'user_material' | 'textbook' | 'web' | 'llm_only'>;
    citationRequiredFor: string[];
  };
  notePolicy: {
    sectionKinds: Array<'definition' | 'mechanism' | 'example' | 'pitfall' | 'check' | string>;
    examplePolicy: string;
    artifactTemplates: Record<string, unknown>;
  };
  promptFragments: {
    attribution: string;
    noteGeneration: string;
    teachingTurn: string;
    variantGeneration: string;
  };
}
```

Initial profiles:

| Subject | Profile focus |
| --- | --- |
| `wenyan` | classic-source examples, translation/reading questions, concise Chinese coaching |
| `math` | symbolic equivalence, step checking, diagrams, partial credit |
| `english` | grammar, translation, speaking/audio future path, ambiguity tolerance |
| `programming` | code execution, tests, compiler/runtime error feedback |
| `physics` | units, formulas, dimensional checks, diagrams/lab data |

Implementation rule: adding a second subject must not fork task implementations. It should add a `SubjectProfile` and subject-specific prompt fragments / judge policy.

### JudgeRouter

All answer assessment goes through one router that chooses the cheapest reliable route for the subject + question kind + answer input.

Judge routes:

| Route | Use case | Output |
| --- | --- | --- |
| `exact` / `keyword` | choice, true_false, fixed fill_blank | `event(action='judge')` |
| `semantic` | short answer, translation, reading | `event(action='judge')` with rubric evidence |
| `rubric` | essay, open humanities, project reflection | criterion scores + feedback |
| `steps` | math / physics / chemistry derivations | step evidence + final answer check |
| `unit_dimension` | physics / chemistry numeric answers | unit normalization + dimension check |
| `code_execution` | programming questions | test result + error feedback |
| `speech_audio` | language speaking tasks | transcript + pronunciation/fluency rubric |
| `diagram_handwriting` | geometry, science diagrams, handwritten work | multimodal evidence |
| `ai_flexible` | borderline / appeal / force flexible | expensive fallback with explicit uncertainty |
| `human_review` | high uncertainty or safety-critical content | pending user review |

Router output must be uniform: a judge event with verdict, score, confidence, judge route, evidence summary, and whether the result is appealable. Rejudging creates a new event; it never mutates prior judgment history.

### AI Proposal Inbox

All AI suggestions share one lifecycle, even when the underlying subject differs.

Proposal payload minimum:

```ts
interface AiProposalPayload {
  kind:
    | 'knowledge_node'
    | 'knowledge_edge'
    | 'learning_item'
    | 'note_update'
    | 'variant_question'
    | 'completion'
    | 'relearn'
    | 'archive'
    | 'judge_retraction';
  target: { subject_kind: string; subject_id: string | null };
  reason_md: string;
  evidence_refs: Array<{ kind: 'event' | 'question' | 'knowledge' | 'artifact' | 'record'; id: string }>;
  proposed_change: unknown;
  rollback_plan?: unknown;
  cooldown_key?: string;
}
```

Lifecycle:

1. Agent writes `event(action='propose', outcome='partial')`.
2. UI shows proposal with reason, evidence, and affected objects.
3. User accepts, dismisses, or requests changes.
4. Accept route performs the owner-service mutation and writes a causal `rate` / `generate` event.
5. Dismiss / accept feedback updates future ranking through acceptance-rate and cooldown signals.
6. Bad accepted proposal can be retracted through an explicit rollback/retraction event.

No AI path may directly mutate hard facts unless it is a low-risk owner-service wrapper with an existing transaction boundary.

### Learning Intent 3a / 3b / 3c

The user flow "我想学 X" must work even before the knowledge graph is prepared.

| Case | Current state | v0.3 behavior |
| --- | --- | --- |
| 3a | topic node missing | propose root node + starter children + hub/atomic outline |
| 3b | topic exists, children missing or weak | propose child graph completion + outline |
| 3c | topic and children exist | current direct outline path remains |

For 3a/3b, the system must preview a graph + learning path proposal before materialization. Accepting the proposal creates knowledge nodes/edges, LearningItems, hub/atomic artifacts, and note generation jobs in one owner-service flow. Rejecting leaves no hard-data mutation except the proposal event.

## 3. Event Layer Rules

- `event` remains the audit spine, not a junk drawer.
- Experimental actions are allowed only with `experimental:*`; promoted behavior gets a KnownEvent schema and migration plan.
- Stable UX surfaces should use projections/read models rather than ad hoc payload scanning.
- Every AI-authored user-visible artifact needs provenance: source task, model/provider, evidence refs, generation status, source tier, and verification status.
- Bad imports, wrong notes, bad variants, and wrong judgments require explicit retraction/rollback flows.

## 4. Pressure-Test Scenarios

| Scenario | Required framework support |
| --- | --- |
| 文言文 translation mistake | Source-backed examples, semantic judge, cause attribution, variant question |
| Algebra derivation | Step judge, symbolic equivalence, partial credit, handwritten rescue |
| Physics unit problem | Unit/dimension judge, formula evidence, diagram support |
| English grammar/speaking | Semantic/rubric judge, audio future path, ambiguity tolerance |
| Programming exercise | Code execution judge, tests, compiler/runtime feedback |
| History essay | Grounded sources, rubric judge, hallucination flagging |
| Hobby project learning | Project goals, weaker FSRS signal, records/artifacts over exam-only review |

## 5. Implementation Order

1. Extract wenyan assumptions into `SubjectProfile` while preserving current behavior.
2. Add `JudgeRouter` interface and route current exact/attribution behavior through it without broad new judge implementations.
3. Normalize proposal events into a reusable inbox reader; keep existing knowledge/edge proposal writers as first producers.
4. Extend Learning Intent to produce preview-only 3a/3b proposals.
5. Add subject profiles and judge routes one subject at a time, starting with math or English as the first non-wenyan pressure test.

## 6. Non-Goals

- No external plugin system in v0.3.
- No standalone MCP server as a product boundary.
- No generic tool artifact abstraction beyond current `tool_quiz` needs.
- No batch/fallback/cost-control claims until runtime enforcement exists.
- No direct AI mutation for destructive actions.

## 7. Longer-Term Roadmap

### Track A — Stabilize The Current Product

Goal: make the existing wenyan product trustworthy before broadening scope.

- Keep canonical docs and code aligned through drift audits.
- Finish current gaps: note verification, embedded checks, note editor/read UX, variant review UI, and proposal rollback.
- Move important AI side effects onto pg-boss when losing them would matter.
- Add runtime-visible AI reliability signals: task run status, degraded status, retry/failure reason, and user-facing provenance.

Exit criteria:

- A user can record, review, learn a topic, read generated notes, see AI proposals, accept/dismiss them, and recover from bad AI output without inspecting raw tables.
- No current docs describe a stack or behavior that is not true in code.

### Track B — Generalize Beyond Wenyan

Goal: add the second subject without duplicating task implementations.

- Extract wenyan prompt assumptions into `SubjectProfile`.
- Introduce one non-wenyan pressure subject, preferably `math` or `english`, because both force judge/router generality.
- Make note templates, examples, source policy, question kinds, and teaching style profile-driven.
- Keep `core/` subject-neutral; keep subject datasets and prompt fragments under `subjects/<id>/`.

Exit criteria:

- Wenyan behavior remains stable.
- The second subject uses the same Learning Intent, NoteGenerate, TeachingTurn, Review, and Proposal surfaces with only profile-level specialization.

### Track C — Build The Assessment Engine

Goal: turn "review" from one exact/LLM path into a multi-route judge system.

- Add `JudgeRouter` as the only answer-assessment entrypoint.
- Route cheap deterministic checks first: exact, keyword, unit/dimension, symbolic equivalence where available.
- Add LLM rubric/semantic routes for open answers.
- Add expensive fallback only for borderline, appeal, or force-flexible cases.
- Preserve immutable judgment history: rejudge creates a new event, never rewrites prior evidence.

Exit criteria:

- Math derivations, physics unit answers, English grammar answers, programming exercises, and wenyan translations can all produce comparable judge events.
- The UI can explain why a judge route was selected and how confident it was.

### Track D — Turn Pipelines Into An AI Decision Layer

Goal: make AI-driven behavior coherent instead of scattered task triggers.

- Normalize all AI suggestions into one Proposal Inbox lifecycle.
- Add evidence refs, reason, cooldown key, rollback plan, and accept/dismiss feedback to proposal payloads.
- Use acceptance rate and dismiss reasons to tune future proposal ranking.
- Support explicit retraction for bad notes, bad variants, wrong graph proposals, and wrong judgments.

Exit criteria:

- The user has one place to review AI decisions across graph maintenance, notes, variants, completions, relearn prompts, and archive suggestions.
- AI actions can explain "why now", "what evidence", and "how to undo".

### Track E — Expand Learning Intent Into A Planner

Goal: let "我想学 X" work even when the graph is missing.

- Implement 3a/3b preview proposals for missing topic / missing children.
- Materialize accepted graph + LearningItems + artifacts atomically through owner services.
- Add plan refinement: user can ask for fewer nodes, exam-focused path, project-focused path, or skip known prerequisites.
- Feed later review outcomes back into plan quality.

Exit criteria:

- A new subject area can start from a natural-language learning intent, not manual graph setup.
- The first generated plan is inspectable and reversible.

### Track F — Source, Grounding, And Multimodal Learning

Goal: make generated content credible and useful across real study material.

- Treat uploaded/user-captured material as first-class source evidence.
- Add source-grounded note generation and verification.
- Support diagrams, handwriting, audio, code, tables, and long passages through specialized ingestion and judge routes.
- Separate raw LearningRecord evidence from long-term MemoryItem projections.

Exit criteria:

- The system can answer "where did this note/example/judgment come from?"
- Hallucination-prone content is visibly marked until grounded or user-verified.

### Track G — Mature Operating Model

Goal: keep the tool maintainable while AI autonomy grows.

- Promote stable experimental events into KnownEvent schemas on a schedule.
- Add read models/projections for UX-heavy views instead of scanning arbitrary event payloads.
- Add budget/fallback enforcement only after usage data shows where it matters.
- Keep external MCP/plugins out until the internal product loop is stable and there is a real external-client need.

Exit criteria:

- New AI abilities enter through subject profiles, judge routes, proposal types, or owner services, not ad hoc DB writes.
- The system remains auditable, reversible, and understandable after multiple subjects and months of AI activity.
