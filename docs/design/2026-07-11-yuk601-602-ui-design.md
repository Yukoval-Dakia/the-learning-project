## 1. YUK-602 — onboarding 手填面（/welcome）

### 1.1 入口与交互

现状（`WelcomePage.tsx:178-191`）：「学科视角（可选）」chips 单选，行驱动自 `useSubjects()`。

设计：chips 行**末尾**追加一枚样式一致的虚线 chip `+ 新科目`。

- 点击 → chip 就地替换为 inline 表单：`<input>`（placeholder「科目名，如：化学」，autoFocus，maxLength 32）+ 确认按钮 + Esc 收起。**收起判定以整个表单容器为界（owner review P2）**：焦点离开容器（focus-within 消失 / `relatedTarget` 不在容器内）才收起——input 单独 blur **不**收，否则 blur 先于 click 发生时确认按钮会在点击前被卸载、提交被吞。不打断页面其余交互。
- 提交 → `POST /api/admin/subjects {displayName: 输入原文}`（trim 后空串则禁用提交；**不做任何 client 侧撞名/归一逻辑**——畸形输入由服务端写门 422 回显，幂等由服务端 200 回放兜底，UI 零去重逻辑，沿 YUK-602 单据）。
- 成功（201 或 200 回放）→ `invalidateQueries(['subjects'])` → 新 chip 经 provider 通路出现；同时 `setSubject(新 id)`（手填即选中，符合「学科视角」正在被填写的意图）；表单收起。
- 失败：422（builtin 撞名等）→ 表单下一行内联红字直出 server `error` message（如 `display name '语文' collides with builtin subject 'yuwen'`——server 文案即错误合同，UI 不改写）；400 → 「科目名无效」；网络错误 → 「网络错误，可直接重试」（API 幂等，重试安全）。
- 提交中：按钮置 loading、input 只读；**不做乐观更新**（成功路径靠 invalidate 即时性已足够，失败回滚复杂度不值）。

### 1.2 通用模式 badge

- 数据源：`useSubjects()` 已下发 `isGeneralFallback: boolean | null`（`src/ui/hooks/useSubjects.ts:18-30`，稳态 = `GET /api/subjects`，非 thin-create 响应——沿 v2 §7）。
- /welcome chips：`isGeneralFallback === true` 的 chip 文案后缀一枚小号 neutral 微标 `通用`（title/aria-label = 「AI 以通用人格教此科目——编辑该科目任一配置后自动消失」）。builtin 恒 false 无标，视觉噪音有界。**取值路径**：chips 由 `listSubjectChoices` 投影驱动（只有 `{id,label}`），badge flag 须从同一 `useSubjects()` 返回的原始 `subjectRows` 按 id join——否则 flag 被投影丢弃、badge 静默永不渲染。
- /admin/subjects 列表：独立「通用模式」列（见 §2.1），呈现同一派生值；general 行该列显示 `—`（合同：general 置 null/不适用）。
- badge 生灭全被动：fork/换绑 → 派生翻 false → 下一次 query 失效后消失；**无任何 client 侧状态机**。

### 1.3 色板

custom 科目 chips/卡片恒 neutral（`SUBJECT_TONE` 本地 map 不动，miss 即 `'neutral'`——`QuestionsPage.tsx:116` 既有行为，v3.2 §5.3 已接受的降级；复活触发器 = owner 点名视觉分科）。

### 1.4 不做什么

- 不建 copilot 建科目工具（YUK-602 单据：无 consumer）。
- 不做科目删除/改名入口（onboarding 面只有 create；控制行动作全在 admin 面）。
- 不改 `GET /api/subjects` 合同、不动 `useSubjects` 形状。

## 2. YUK-601 — admin trait 编辑面

### 2.1 列表页扩容（/admin/subjects，既有页）

数据源从现 slim 投影（`admin-subjects.ts:11-19`，只有 id/displayName/version/notation/capabilityCount）换到 §3.5 扩容后的管理枚举。列设计（沿既有 inline token 表格风格，`subjects.tsx:139-159`）：

| 列 | 呈现 |
|---|---|
| displayName | 主文本；retired 行整行降透明度 + `retired` neutral 徽标 |
| id | mono 小字（opaque id 全量可见，便于对账） |
| origin | 徽标——**`subject.origin` 权威枚举只有 `builtin` / `custom`（owner review）**；`general` 徽标不来自 origin，由 `id === 'general'` 派生（detail 页 header 同规则） |
| 通用模式 | `isGeneralFallback` true → `通用` 徽标；false → 空；general → `—` |
| version | 组合串 mono 小字（`jt:<id@effective>;…`），title 悬浮全文 |
| →  | 行链接进 `/admin/subjects/:id` |

列表页保持只读 + 一个例外：**不在列表页放任何写按钮**（写动作集中 detail 页，RL5 注释的「READ-ONLY page」语义对列表页原样保留）。

### 2.2 detail 页（/admin/subjects/$id，新 route）

布局自上而下（复用 `PageHeader`/`Card`/`Stateful`/`Badge`）：

1. **PageHeader**：displayName + origin/retired/通用模式徽标；sub 行 = opaque id + version 组合串。**保留既有 admin 横向导航习语**（runs/cost/failures/subjects/coverage link row，沿 `subjects.tsx:77-83`）+ 一枚「← subjects」返回列表链接（进 detail 后有明确回路）。右侧控制行动作区（§2.3 矩阵 rename / retire·restore / reset 三行）。**header 数据源（owner review P2）**：detail 页自行 query `GET /api/admin/subjects`（queryKey `['admin-subjects']`，与列表页共享缓存）按 `subjectId` 取本行——**直达刷新 `/admin/subjects/$id` 不依赖列表页缓存在场**；`sharedBy` tooltip 的 subjectId→displayName 映射同源取自该枚举。
2. **六绑定表（Card）**：一行一 kind（charter / judge_policy / cause_taxonomy / source_policy / render_theme / scheduling），列 = kind · 绑定 trait（traitId mono + origin + ownerSubjectId）· seedVersion · revision（`degraded` 非空时并列 `effectiveRevision` + 黄底徽标 `journal_fallback`/`code_seed`——「实际在用哪份」可见，v3.2 §3.5）· sharedBy（`共 N 科` 计数，title 列科目名）· 操作（展开编辑 / fork / 换绑 / 历史）。
3. **per-kind 编辑区**（点「展开编辑」就地展开于该行下方，同页无跳转）：
   - 表单 = 该 kind 的 strict Zod schema 逐字段（`trait-schemas.ts` 单一权威；charter 为命名节 textarea 组——含 `methodology`、`rubricGuidance` 两节，即判词 R 的编辑入口，无独立面板）。
   - 底部动作条（主按钮 = §3.1 主写面 PUT subject-scoped）——**提交前语义按所有权×共享度三分呈现（owner review P1：自有 trait 走 subject-scoped 是原地写，被借绑时会 fan-out 全部绑定者，不许静默波及）**：
     1. **非本科自有**（general 种子 / 他科种子 / 他科 fork）→ 按钮「保存（自动 fork 本科副本）」+ 常驻说明「原 trait 与其它科目不受影响」（COW 预告，不做事后惊喜）；
     2. **本科自有且独占**（sharedBy 仅本科）→ 按钮「保存（本科生效）」，原地写，无额外确认；
     3. **本科自有且被他科借绑**（sharedBy > 1）→ 原地写 = 影响全部借绑科目：按钮「保存（影响 N 科）」+ 内联波及科目清单 + 二次确认。
   - 次级按钮 **「编辑共享面（影响 N 科）」**（= §3.1 显式共享写 PUT /api/admin/traits/:traitId）：出现条件 = **`subjectId === 'general' || sharedBy.length > 1`**（owner review P2：零 custom 科目时 general 种子的 sharedBy 只有 general 自身，若只按 `>1` 判，general 编辑器会没有任何保存入口）；点击先内联展开波及科目清单 + 二次确认（合同 `:148` 的按钮语义直译）。与主按钮分工：主按钮 = 「以本科身份写」（写门自行裁 COW/原地），共享写 = 「明知影响全体地写 trait 本体」——第 3 分支两入口效果重合，确认文案保持一致。
   - **validate 预检**：表单任何编辑后、提交前，可点「预检」调 `POST …/validate`（无状态）→ issues 内联列出；预检非强制（服务端写门恒兜底），是省一次 422 往返的便利。**pending 态**：预检请求在飞时按钮 loading、提交按钮禁用；失败按普通请求错误内联呈现（不阻塞后续直接提交）。
4. **journal / rollback**（每行「历史」展开）：读 `GET /api/admin/traits/:id/journal`，倒序**纯 revision 列表**——revision / action（**全枚举**：create · edit · fork_source · rollback · **reconcile**〔migrate 系统动作，actor='migrate'，builtin 种子 semver 升级写入——v3.2 §2.2:90 / §6.3〕· reset_to_seed）/ actor / 时间；**不做 payload diff 可视化**（合同只要求 journal 作 rollback 数据源；diff 查看器留作 owner 点名后的 follow-up）。每个历史 revision 一个「回滚到此」按钮 = rollback-forward（git-revert 语义，v3.2 §3.2；**reconcile revision 同为合法回滚目标**）——按钮 title 明示「产生一条新 revision，不删除历史」。目标引用已退场 capability 的 422 直出 server message。展开区走 `Stateful` 习语：loading skeleton / error + 重试 / 空（不应出现——至少有 create 行，出现即报障）。

### 2.3 写动作 UX 矩阵

| 动作 | 入口 | 确认形态 | 失败呈现 |
|---|---|---|---|
| 保存（subject-scoped，自动 COW） | 编辑区主按钮 | 非自有 trait 时常驻 fork 说明（§2.2） | 422 issues 内联；409 见下 |
| 编辑共享面 | 编辑区次级按钮（sharedBy>1 时） | 内联波及清单 + 二次确认 | fan-out 422 回显「哪科哪条 issue」逐条列出 |
| fork（显式剥离不带编辑） | 绑定行操作 | 内联确认条「剥离后本科独立演化，不再跟随来源」 | 422 直出 |
| 换绑 | 绑定行操作 → 就地展开选择器（数据源 `GET /api/admin/traits?kind=`，每候选显示 traitId/origin/boundBy；**loading = Stateful skeleton；加载失败 = 内联错误 + 重试；除当前绑定外零候选 = empty 文案「本 kind 暂无其它可换绑 trait」**） | 选中后确认条「改绑后沿用目标 trait 的后续演化」 | kind 不匹配/target 不存在 422 直出 |
| rollback | 历史区按钮 | 确认条（rollback-forward 语义句） | 422 直出 |
| reset-to-seed（trait 级） | 编辑区「恢复出厂」（仅 seedVersion 非空时可见） | 确认条「影响绑定此 trait 的全部 N 科」 | fan-out 422 逐条 |
| rename（控制行） | PageHeader 动作区 | inline input + 确认 | 撞 norm 409 直出 |
| retire / restore | PageHeader 动作区 | 确认条（retire 说明「仍可解析历史数据，不再出现在选科/词表」）；general 行不渲染 retire | 409/422 直出 |
| reset（subject 级，只换绑回种子） | PageHeader 动作区 | 确认条「六绑定指回种子；已 fork 的副本保留为孤儿，不删除」 | 422 直出 |

**并发（CAS 409）处理——只认 `{currentRevision}` 形状，不认裸状态码**：写返回 409 **且 body 携带 `currentRevision`**（= CAS 陈旧守卫）→ 顶部内联条「配置已被其他会话更新，已刷新最新版本，请重新提交」+ 自动 refetch（v3.2 并发协议：advisory lock 已消灭真并发危险，CAS 只是陈旧 UI 提交守卫——UI 只需 refetch-and-retell，无合并 UI）。**rename/restore 的撞名 409 是另一物种**（无 currentRevision，refetch 不解决——用户必须换名）：按矩阵行内「直出」处理，内联「名称已被占用」，**不触发** refetch 重放（否则同名重放死循环 + 真错误被「已刷新」话术掩埋）。

**client 前置（owner review P1）**：现行 `src/ui/lib/api.ts` 的 `ApiError` 只保留 `status/code/message`——409 body 的 `currentRevision` 会在 client 层被丢弃，上述分流无法实现。YUK-601 须扩展 `ApiError` 携带结构化 `details`（原样保留响应 body 的附加字段）+ 配套单测；已列入 §0.3 touch 清单。

**general 特殊态**：general 的 detail 页六绑定行禁用 fork/换绑（title=「general 绑定结构性锁定（v3.2 §2.3）」）；编辑区只有「编辑共享面」路径（其种子 trait 编辑本就是全体跟随语义）+ reset-to-seed；PageHeader 无 retire。禁用是 UI 礼貌，422 写门才是红线。

detail 页数据 = `GET /api/admin/subjects`（header 行 + sharedBy 名称映射，§2.2）+ `GET /api/admin/subjects/:id/traits`（六绑定）+ 按需 journal / traits?kind=。TanStack Query，queryKey 按资源分（`['admin-subjects']` / `['admin-subject-traits', id]` / `['admin-trait-journal', traitId]` / `['admin-traits', kind]`）；一切写 onSuccess invalidate 涉及 key + `['subjects']`（badge 即时翻转到 onboarding 面）。无轮询。

**读面 UI 侧前置——`subjectRevision`（owner review P1）**：几乎所有写（subject-scoped 保存 / fork / 换绑 / 控制行动作）都要提交 `expectedSubjectRevision`，但 §3.5 两个读端点原文都不下发数字 `subject.revision`（管理枚举的 `version` 是组合串；`currentRevision` 只在 409 冲突时出现，都当不了**首次提交**的初始值——没有它 UI 连合法请求都构造不出）。YUK-601 读面实现须补：`GET /api/admin/subjects` 每行 + `GET .../:id/traits` 顶层各带 `subjectRevision`（数字 CAS 轴），并纳入写后 invalidate 与单测合同。这是 §3.5 的 UI 侧必要扩容，不改既有字段语义。
