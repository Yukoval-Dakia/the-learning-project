## owner 想解决的问题

录入是这个工具的**主要入口**（上传 / 拍照为主、手填错题次之）。它是「把外部材料变成系统里能练的东西」的关键转化点。但录入「做完之后」的体验是断的：成功后**硬跳走**到 `/mistakes`，跳走那一刻**什么都没交代**——这次录入到底产出了什么、进了哪、接下来能干嘛，全靠 owner 自己回去翻。转化完成却没有任何确认。

owner 要把录入出口从「提交完跳走」变成**有交代的着陆**：让 owner 落在一个停留的视图上，一眼明白「这次录入发生了什么 + 我下一步能去哪」，由 owner 主动选去向，而不是被系统替他决定跳到哪。

## 该呈现什么（功能层，非视觉）

录入成功后，落在一个**停留的着陆视图**（不是瞬时 toast、不自动跳走）。它要承担两件事：**交代产出** + **给去向集合**。

### 1. 出口叙事：这次录入发生了什么

让 owner 一眼明白本次录入的转化结果。功能要素：

- **产出回执**：这次产出了**几道题**（批量用 `question_ids.length`，手填为单条），以及它们进了哪些**通道**：
  - **题库**（canonical question）—— 每道导入的题产生一道 question。
  - **知识点** —— 每道题挂了 owner 选的知识点（手填 / 块编辑器都强制至少 1 个）。
  - **错题本** —— 进了 `/mistakes`（错题是标记不是独立通道，per D11）。
- **批量 vs 单条都要成立**：批量导入是「这一批 N 道都进来了」，手填是「这 1 道进来了」。两种路径共用同一个着陆视图，计数 = 0 时也不能崩（理论不可达，但要兜住）。

> 「交代到多详（一句叙事 vs 一行摘要 vs 列出每道题）」由 claude design 定；功能约束是**必须让 owner 看到产出了什么、进了哪**，不能像现在一样静默跳走。

### 2. 去向集合：owner 主动选下一步

着陆的核心是**把单一硬跳换成多出口**，由 owner 选去向（行为驱动，非死跳）。

**owner 已锁的三件套（baseline，必须在场）**：
- 去**练刚录的**（把刚转化进来的题直接拉进练习）。
- 去**看知识点**（刚录的题挂到的知识点——产出回执已把「知识点」列为产出通道之一，这条让 owner 直接去看被点亮 / 新建的那些知识点；数据依赖见「数据前置」）。
- **继续传**（继续录下一批；批量录题时不想每道都被踢走——很可能是高频默认停留）。

**handoff 追加候选（待 owner 确认是否纳入，见 owner 留白）**：
- 去看**错题本**（现状的唯一去向，保留为可选项）。
- 回**工作台 / 录入面**。

着陆视图要让「继续传」这条**留在原地、低成本回到录入流**成为一等出口——这关系到录入是「一次一道」还是「一批多道」的主用法。

## 硬功能约束

- **是着陆，不是 toast**：成功后**停留**在一个视图、由 owner 主动选下一步；**不允许自动跳走**（这是现状要废掉的反模式）。瞬时 toast + 仍自动跳的形态**不满足**本约束。
- **必须交代产出**：着陆视图必须呈现「产出了几道题 + 进了哪些通道」，不能只是一个无内容的「成功了」。
- **必须给多出口**：至少替换掉现状的单一硬跳，提供 owner 可选的去向集合（含「继续录入」）。
- **批量与手填共用同一着陆契约**：两条路径产出的数据形状不同（数组 vs 单条），但着陆视图要统一承接，计数为 0 / 1 / N 都不崩。
- **本期覆盖四条录入路径**：手填错题、拍单题、拍试卷、PDF/DOCX 导入。`AI 录入` tab（`AutoEnrolledPanel`，`RecordPage.tsx:64`）是另一条 surface，不在本期着陆范围。

## 空态 / 失败态 / 低置信态

claude design 需为这些非 happy-path 出态，不能只画成功路径：

- **空态 / 0 产出**：理论不可达（import 至少保留 1 块），但着陆视图的产出计数要能显示 0 且不崩——不需要专门的空态插画，但「N 道题」里的 N 必须能优雅地是 0。
- **失败态（着陆**不**出现）**：录入失败时根本到不了着陆（提交失败留在表单 + 一行报错 `RecordPage.tsx:301-303`；抽取失败留在 review 流 error 态）。**着陆视图只画成功路径的落点**；失败的逃生口是另一条 surface（A8 失败态 handoff），不在本面。换言之：**着陆视图永远只在录入成功后出现**。
- **低置信顺带提示（弱信号，非精确数）**：批量导入若含低置信抽取的题，着陆可顺带一句「其中 N 道是低置信抽取，建议回看」——按 ADR-0035 §24 口径，**低置信只作弱信号提示，绝不渲染看起来精确的置信度百分比**。这是可选润色，不是硬约束。

---

## 不在本面板范围

- **不处理录入失败 / 退化的逃生口**（rescue 失败态、空块、PDF 超时、DOCX 切题核对、figure 回显）—— 那些是录入 robustness / 退化叙事，是独立 handoff（A8 失败态 / `docs/design/2026-06-28-form-axis-A8-handoff.md`），与本着陆视图**相对正交、不阻塞**。
- **不动抽取 pipeline 算法**（OCR/VLM、切题、figure 门控）。
- **不做图谱侧的「录入→建节点→可达 frontier」闭环**（§2.8a，单独缺口）。
- **不新增题型 / 错因 / 知识点编辑能力**——着陆只读地交代产出，不在着陆上编辑。
- **不覆盖 `AI 录入`（AutoEnrolled）出口**——后续增量对齐。

---

## owner 留白（需 owner 拍）

1. **去向集合的具体成员与默认**：owner 已锁三件套 = 去练刚录的 / 去看知识点 / 继续传（baseline，必须在场）；handoff 追加候选 = 看错题本 / 回工作台（待 owner 确认是否纳入）。哪个是视觉上最突出的主出口？批量录题时「继续传」是否应是**默认停留**（不自动把 owner 踢去错题本）？
2. **产出回执的粒度**：着陆只报「N 道题，进了题库 / 知识点 / 错题本」这个**汇总**，还是要**逐题列出**（每道题挂了哪个知识点 / 可跳到该题）？汇总更轻、逐题更有掌控感——claude design 需要这个定调才能定形态。
3. **低置信提示是否本期纳入**：是顺带做掉（弱信号一句话），还是延后到 figure / 退化叙事那一期统一处理？

## 数据前置

- **着陆所需数据已全部就位，纯前端可做、零后端改动**：
  - 批量导入：`POST /api/ingestion/[id]/import` 已返回 `{ question_ids, mistake_ids, record_ids }`（`import.ts:552-556`），三数组等长、长度 = 题数。
  - 手填错题：`POST /api/mistakes` 已返回 `{ question_id, mistake_id, record_id }`（`RecordPage.tsx:111`，单条）。
  - 现状前端**拿到即丢弃**——着陆叙事直接消费这些计数 / id 即可。
- **去向「去练刚录的」**若要带着刚录的题 id 进练习，靠的也是上面这批 id（已在响应里），无新依赖。
- **去向「去看知识点」**（owner 锁的三件套之一）需要刚录题挂的 KC id——当前导入响应只返回 `question_ids` / `mistake_ids` / `record_ids`，**未直接返回 `knowledge_ids`**。若纳入此去向，需确认 KC id 来源（响应补 `knowledge_ids`，或由 `question_id` 反查其知识点链）——属轻量实现期接线，不阻塞视觉稿。
- **A8-infra（figure content 端点 / PDF abort，Linear YUK-522）与本着陆视图正交、不阻塞**——那是录入 robustness 的后端前置，着陆视图不依赖它，可独立设计 + 落地。
