## owner 想解决的问题

练习卡住时，owner 要的不是「点一下直接给答案」（养成依赖、毁掉独立提取的学习价值），也不是「强制苏格拉底、永远不许看答案」（Khanmigo 被诟病的反模式）。owner 要的是**一个自己掌控出手档位的滑块**：默认先给最轻的提示（H0），不够再要下一阶、再下一阶，**逐级加力逼近**；任何时刻都能**一步滑到最强档（H5 = 完整解 / 逃生口）**，不被关在苏格拉底牢笼里。

现状的提示抽屉**功能上已经能逐级给提示**，但 owner **感知不到「逐级」**——每一阶看起来都一样，是「又一条更具体的话」，没有强度梯度的体感，看不出自己在哪一档、下一档会更狠到什么程度、离「完整解」还有几步。

一句话：**把「卡住时逐级加力」从一串看不出差别的提示，变成 owner 能切身感知到强度在 H0→H5 之间爬升的 6 阶梯子。**

## 现状反模式（锚真代码）

当前散题作答面 `PfSolo.tsx` 顶部有「卡住了？解题会话」按钮（320-322 行）触发抽屉 `PfCoach`（540-645 行）。它**已是递进 hint 的雏形**，但「逐级」对 owner 不可感知：

1. **递进是「无级裸数字」，没有强度梯度的体感。** `PfCoach.nextHint()`（565-581 行）每点一次就 `solveHint(question.id, sid, hints.length)`（573 行），把累计条数 `hints.length` 当 `hint_index` 递增。后端 `buildSolveHintInput`（`solve-session.ts`:141-163）的提示词只是按 index 线性加压——index 0 发「给我一个不剧透答案的最小提示，只点一步方向」，之后每阶都发同一句「还是不会，给下一个更具体的提示（第 N 个）」。**没有「元认知 / 方向 / 概念 / 错因 / 部分示范 / 完整解」这种每阶语义分层，也没有任何强度刻度**——只有「再具体一点」的线性递增。owner 看不到自己在阶梯哪一档、这一档相对整条梯子有多强、下一档会给什么性质的帮助。

2. **没有「我在第几阶 / 共几阶」的位置感。** 抽屉里唯一的位置线索是按钮文案里的裸计数「再提示一点 · N」（635 行）和提示卡上的「提示 1 / 2 / …」（619 行）。**没有「H2 / 共 6」这种刻度**，owner 不知道自己爬到哪、还剩几阶。

3. **没有「一步到完整解」的逃生口。** 抽屉里唯一的前进动作是「再提示一点」（623-636 行），撞到上限 `MAX_HINT_INDEX=20`（`known.ts`:26）后只显示「提示用完了。回到题面把你的思路写进作答框试试。」（638 行）。**最强档（完整解 / 逃生口）在解题会话里没有独立入口**——owner 只能一阶一阶往下挪，没法直接跳到 H5。

4. **没有「看完整解 = 非独立」的语义。** 看了多少帮助之后，作答仍按完全独立处理，**没有任何「看了完整解 = 这题不是你独立做出来的」标记**。

5. **提示用量在本屏被丢弃（数据留痕断链）。** `PfCoach`（535 行实例化）数着自己的 `hints` 数组，但**不向 PfSolo 回灌**——没有回调，PfSolo 的 `commit()`（172-232 行）走 `review/submit`（`practice-api.ts`:302-319），该 body **不携带 `hints_used` / `final_hint_level`**。详见末尾「数据前置」。

> 反模式总结：现状是「无级线性加压 + 无位置刻度 + 无完整解逃生口 + 无非独立标记」。本增量要把它升级成「**可感知的 6 阶强度梯 + 位置可见 + 一步逃生口 + 非独立标记**」。

## 该呈现什么（功能层 —— 新体验维度）

claude design 要为以下这些 owner **能理解、能做**的事出视觉形态（**怎么呈现是 claude design 的活，下面只列功能语义**）：

1. **6 阶强度梯本身可感知。** owner 能切身感受到从 H0 到 H5 强度在**爬升**——这不是 6 条等价的提示，而是越往下越接近答案的**梯度**。owner 看一眼就知道「这是越来越强的一串」，而非「又一条话」。

2. **当前位置 + 总阶数可见。** owner 任何时刻看得出「我在第几阶 / 共 6 阶」。

3. **下一阶的「性质」可预知，但不剧透内容。** owner 在要下一阶之前，知道下一阶**给的是什么性质的帮助**（如「下一阶：指方向」「下一阶：点关键概念」），但**看不到具体内容**（防剧透——内容要等真的要了才出）。

4. **按需召唤，不预先铺开。** 默认只显示「当前能要的下一阶」，不一次摊开 6 阶全部内容（防剧透 + 防 owner 一眼扫到答案）。

5. **最强档（H5 / 逃生口）始终一步可达。** 与「再要下一阶」同时可用、任意阶一步可达，**始终有一个独立的「直接看完整解」入口**——owner 不必逐阶点到底，任何时刻能一步跳到 H5。这是**防强制苏格拉底**的核心保证。

6. **看 H5 的后果（非独立）可被 owner 清楚知道。** 点「直接看完整解」不是免费滑到答案——它把这题标记为**非独立完成**。owner 在看到完整解**之前**必须清楚这个后果（语义必须传达，呈现形态交 claude design）。

7. **看完 / 用完帮助后控制权回到 owner。** 帮助是叠加在作答流之上的可选层，不抢走作答主权（现状「会话不计入判分」的隔离感是对的方向）。owner 看完帮助后可回作答框自己写，或看了完整解后直接进下一题。

## 锁定的 6 阶梯（owner 已拍定，claude design 据此出形态 —— 勿重开「几阶」）

> **6 阶 H0-H5 是 owner 锁定的形态决策，不是待拍清单。** claude design 不必、也不应在「3 阶还是 6 阶」上做选择——直接按这 6 阶出强度梯的视觉形态。下表是**每阶给什么性质的帮助 + 该阶的独立性语义**（强度从上到下递增）；具体阶名标签的措辞、以及强度怎么可视化，是 claude design 的留白。

| 阶 | 性质（给什么，非具体内容） | 独立性 |
|---|---|---|
| **H0** 元认知 | 不碰题目内容，问「你卡在哪一步 / 这类题一般怎么下手」 | 独立 |
| **H1** 方向 | 指一个大方向 / 该调用哪块知识，不给具体步骤 | 独立 |
| **H2** 概念 | 点明关键概念 / 定理 / 公式，仍不代入本题 | 独立 |
| **H3** 错因 | 针对 owner 当前思路指出卡点 / 常见误区 | 独立 |
| **H4** 部分示范 | 演示第一步 / 关键一步怎么做，留剩下给 owner | **半独立** |
| **H5** 完整解 | 完整解答（= 逃生口的「直接看完整解」） | **非独立** |

> ⚠️ **锁定范围澄清（critic 抓到越权归属，已修正）**：owner 锁定的只有「**6 阶强度梯 + H5 逃生口 + hints_used 持久化 + 会话不计入判分**」。本表「**独立性**」列 + 下方硬约束 3/4 的 independence 三态机制（半独立 / 非独立标记、看 H5 后果确认门、非独立完成率指标）是 **handoff 提议、待 owner 确认**是否本屏 in-scope（或拆到 YUK-352 紧邻 follow-up）。claude design 可据此出形态，但 owner 未拍前**不当锁定项**。

**强度梯的设计意图**：H0 最轻（只逼 owner 自己想），强度逐阶递增，H5 最强（直接给答案）。claude design 要让 owner **体感到这条从轻到重的爬升**——这正是本增量的核心新维度。

## 硬功能约束（不可违背的功能语义）

1. **单调递进，不能跳级。** 逐阶只能往下一阶走（H2 → H3），不能跳到 H5——**要看完整解 = 走逃生口**（显式独立动作），不是「滑到底」。

2. **逃生口 = 任意阶一步到 H5。** 阶梯面上**始终**有独立的「直接看完整解」入口，与「再要下一阶」同时可用、任意阶一步可达。点它 = 直接跳到 H5，跳过中间所有阶。owner 永远不被关在「只能一阶一阶挪」里。

3. **H5（看完整解）是非独立动作，须先让 owner 知道后果。** 点「看完整解」会把这题标记为**非独立完成**（影响掌握度 credit）。约束：点击时须让 owner **清楚知道**「看了 = 这题记为非独立」，确认后才展示完整解。**不羞辱**（中性、诚实的后果告知，不是惩罚式警告）。

4. **独立性分级映射**（功能语义，owner 可微调命名）：
   - 没要任何帮助（hints_used=0）→ **完全独立**。
   - 要到 H0–H3 → 仍算**独立完成**，但留痕「用了第 N 阶提示」（提示依赖度信号）。
   - 看了 H4 部分示范 → **半独立**。
   - 看了 H5 完整解 → **非独立完成**。

5. **会话隔离不变。** 解题帮助本身**不计入判分**（现状「会话不计入判分」615 行是对的）——帮助内容不进判分上下文，只「用了第几阶」这个**计数**回灌作答的独立性标记。两者分离：帮助是过程，独立性标记是结果。

6. **本增量是在既有 PfCoach 抽屉内升级，不改作答主流。** 散题作答的提交、判分、评级流（PfSolo 的 `runJudge` / `commit`）不动；强度梯只活在「解题会话」这一侧。

## 空态 / 失败态 / 低置信态（功能约束）

1. **完整解未就绪 / 懒生成失败**（无既存参考解且自动生成失败）：
   - 阶梯**仍可起跑前几阶**（H0 元认知 / H1 方向不依赖完整参考解），但「看完整解」逃生口（H5）**必须显式置不可用**并说明「这道题暂时没有可展示的完整解（自动生成失败）」——**不能给一个点了没反应的死按钮，也不能假装有完整解**。owner 看得出是「暂时没有」而非「系统坏了」。

2. **某一阶 hint 生成失败 / 超时**（hint route 返回上游错误 / 解析失败 / 网络超时）：
   - 当前阶显示**可重试**的失败态（「这一阶提示没生成出来，再试一次？」），**不静默吞**、**不自动跳到下一阶**（跳级会破坏单调递进的强度梯语义）。owner 可重试本阶或直接走逃生口看完整解。

3. **滑到 H5 仍不会**（看了完整解还是不懂）：
   - H5 之后**不是死路墙**。须有「仍不懂」的去向——例如继续苏格拉底追问，或「标记为错题 + 进复习」。**约束**：owner 看完完整解仍能继续追问或把题推进复习闭环，**不能卡在「看了完整解还是不会」的死胡同**（具体接哪条由 claude design / owner 定）。

4. **逃生口被频繁使用（每题都直接看 H5）**：
   - 这是**留痕问题，不是阻拦问题**。**不得**用配额 / 冷却 / 弹窗惩罚拦截 owner 看完整解（违反 owner「控制权在我」+「护栏只告知不卡死」原则）。约束是：**每次看完整解都诚实记为非独立完成**，让「非独立完成率」成为**可观测信号**（owner 自己日后看「我有多少题是直接看答案做的」），驱动自我调节——而非系统强制限制。视觉上**中性留痕，不羞辱**。

## 不在本增量范围

- **不改判分 / 掌握度的计算**——本屏只负责「采到 hints_used / independence 信号 + 控制完整解出口 + 让强度梯可感知」，掌握度怎么吃这些信号是别处的事。
- **不做「提示依赖度 / 流畅度幻觉」的诊断展示面**——本增量只保证信号被采集 + 强度梯可感知，不画诊断面。
- **不改后端每阶 hint 生成的提示工程质量**——把现有线性提示词升级成「按 H0-H5 分阶生成对应性质帮助」的 prompt 改写是后端增量（见数据前置），**不是 claude design 的视觉活**；claude design 给的是强度梯的**视觉形态**。
- **不重画 PfSolo 整屏 / 整个 PfCoach 抽屉壳**——只为「6 阶强度梯」这个新维度出形态。
- **不重开「几阶」决策**——6 阶已锁。

## owner 留白（claude design / 后续可定）

- **强度梯怎么可视化**——刻度条 / 台阶 / 温度计 / 进度环 / 别的：claude design 的核心创作空间。本 handoff 只锁「owner 要能体感到 H0→H5 的强度爬升」，不规定用什么形态表达。
- **每阶展示给 owner 的标签措辞**——表里「元认知 / 方向 / 概念 / 错因 / 部分示范 / 完整解」是性质描述，给 owner 看的标签文案可由 claude design / owner 定。
- **「下一阶性质」预告的呈现粒度**——给一个性质标签即可，是否再加一句话提示由 owner 定。
- **H4 半独立是否单列档**——独立性映射里 H4 标「半独立」，是否在 UI 上把它和「独立 / 非独立」三态显式区分，owner 可微调。
- **independence 字段命名**（independent / semi / non_independent 或其它）—— owner 拍（数据层命名，不影响视觉）。
- **「看完整解后仍不会」接哪条去向**（追问 / 错题 / 复习）—— owner / claude design 定。

## 数据前置依赖

> **结论先行：本增量的视觉设计无硬数据前置——claude design 现在就能画。** 解题会话用的 solve / hint endpoint 已存在并在用；6 阶强度梯是**前端在既有 `hint_index` 之上的映射**（前端持 6 阶 ladder 定义，后端接口量仍是单一数字 `hint_index`）。下列两条是**实现期**的接线，不阻塞视觉稿。

**已就位（owner 所述 infra）**：
- `hints_used` / `final_hint_level` 字段已在事件 schema 层定义（`known.ts`:88-89，cap `MAX_HINT_INDEX=20`，YUK-352 Done）。
- 持久化机制存在——`solve-submit` 路径已串这两个字段（`solve-submit.ts`:27-28、49-50 → `solve-session.ts`:401-402 落 attempt payload）。
- 启动会话 / 要一阶提示 endpoint 已存在：`POST /api/questions/:id/solve`、`POST /api/questions/:id/solve/:sid/hint`（body `{ hint_index }`，`practice-api.ts`:327-338）。

**实现期接线缺口（不阻塞设计，但本增量真正落数据需要）**：
1. **本屏主流（review/submit）未透传 hint 用量。** PfSolo 的 `commit()` 走 `review/submit`（`practice-api.ts`:302-319），该 body **不携带 `hints_used` / `final_hint_level`**——只有 `solve-submit` 路径有。要让 6 阶强度梯的留痕在散题练习流真正落库，需给 `review/submit` 契约 + `submitReview` 签名接上这两个字段，并让 `PfCoach` 把帮助用量（最高阶 + 是否看了 H5）**回灌**到 PfSolo 的 `commit()`（现状 `PfCoach` 535 行无回调）。
2. **`independence` 是新留痕维度，schema 尚无此字段**——由「看了第几阶」推导的独立 / 半独立 / 非独立标记，需新增字段（命名 owner 拍）写进 attempt payload。
3. **后端 hint 生成升级为分阶语义**（可选 / 后续）——现状 `buildSolveHintInput`（`solve-session.ts`:141-163）只发线性「再具体一点」，要让 6 阶在内容上**真的给不同性质的帮助**（而非只是前端贴标签），需把每阶 prompt 改成按 H0-H5 生成对应性质帮助。属后端 prompt 工程增量。

> 上述 1+2 正是既有 YUK-352（hint 留痕）在 solo 流的未竟接线 + 本增量的 independence 扩展，建议作 YUK-352 的紧邻 follow-up 工单（后端工，独立可验收，不混进 claude design 视觉实现 lane）。**它们不是 claude design 出稿的前置**——视觉稿可先行。
