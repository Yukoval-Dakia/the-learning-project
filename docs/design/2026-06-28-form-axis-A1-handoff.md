## owner 想解决的问题

`/today` 是单人 AI 学习工具的工作台。设想中，AI 在**昨夜替 owner 干了活**——夜间 job 跑了重标定、补了题库空缺、提了知识边、留了观察笔记、备了课。owner 早上打开 `/today` 时想要的不是一屏冷数字，而是一条**叙事缕**：

- **先轻**：一眼看到「今天有几项到期、几条待裁决」——低阅读量、立刻可动作，不读叙事就能开始。
- **后叙事**：再被 AI 以第一人称口吻交班「**昨夜我替你做了什么**」——这是「代理信任」每天被**兑现一次**的地方。AI 不是黑箱：它昨夜的劳动对 owner 当下可见、可下钻、可裁决。

「先轻后叙事」是这条缕的核心节奏：动作优先于叙事，叙事建立信任。

## 现状反模式（锚真代码，A1 要修掉这些）

> 下列缺陷都锚在真实文件/行，非凭空。

### 反模式 1 — hero 的「昨夜叙事」被原设计稿想要、却被迫中性化成永远成立的空话
`src/capabilities/shell/ui/blocks/LoomHero.tsx:2-3`（文件头注释，逐字）：

> 偏差：…lead 句中性化——设计稿写死「昨晚 Dreaming agent 跑过」，真数据源（task_run 交班）M5 Copilot 收编后才可判定，**不写死假话**。

结果 `LoomHero.tsx:47` 只剩一句永远成立的空话：`这是你的工作台：复习队列、AI 的提议与改动都汇在这里。` 问候语（`LoomHero.tsx:17-27`）也只按**当前时钟小时**机械切「早上好/午安/晚上好」，与昨夜 AI 活动**零关联**。这条新缕的本质就是把这句被迫中性化的叙事**真正接通**——前提是后端能区分「真有昨夜活动」与「空夜」。

### 反模式 2 — 「今日之线」只派生「复习/裁决」两缕，交班缕被代码注释自承缺失
`src/capabilities/shell/ui/TodayPage.tsx:50-51`（`deriveThreads` 上方注释，逐字）：

> 「今日之线」派生：…真数据从 summary 聚合派生两缕（复习/裁决），**未来夜链交班缕随 M5 task_run 读模型补**。

代码实际只产出两缕：`due_count > 0` → 「复习」缕（`TodayPage.tsx:54-66`），`proposals.total > 0` → 「裁决」缕（`TodayPage.tsx:67-79`）。**「昨夜 AI 活动 → 今晨交班」缕从未实现**，注释自承「随读模型补」——那个读模型现在才立 issue（YUK-520）。

> ⚙️ **形态锚点（owner 锁定）**：这条新缕**喂 `deriveThreads` 风格卡**——即 owner 期望它与既有「今日之线」的卡片是**同一视觉家族 / 同一叙事语言**，作为「线」的第三种缕并列或延展，而非另起一套完全异质的 UI 区块。claude design 据此判断：交班缕是「今日之线」的扩张（多一类缕卡），还是其下/旁的并列叙事段——这是你的形态决策，但必须与既有缕卡协调。

### 反模式 3 — AgentNotesBoard 是扁平流，不按「昨夜」窗、不叙事、不交班
`/today` 底部挂 `AgentNotesBoard`（`TodayPage.tsx:289-295`），数据走 `/api/agents/notes?limit=20`（`TodayPage.tsx:212-218`）。它是 AI 跨上下文观察的**只读旁观流**（newest-20 扁平列表，hints not facts），**不按时间窗聚合到「昨夜」**、不叙事。它是「AI 在想什么」的长流，不是「昨夜 AI 替你做了什么」的当日交班——两者不同维度，AgentNotesBoard 保留原样，交班缕是新增维度。

### 反模式 4 — 交班缕（稳态）≠ 冷启拦截，绝不混淆
`TodayPage.tsx:236-238`：`s.kpi.goal_count === 0` → 渲染 `<ColdStart>`（首次零档案冷启 onboarding）。**那是首日零档案的冷启**。交班缕是**有历史之后**的稳态晨间叙事（`goal_count > 0`、有过往档案）。两者必须分开：交班缕的「空夜态」（下章）发生在 **`goal_count > 0` 但昨夜无 AI 产出**，**绝不能落回 ColdStart**。

## 该呈现什么（功能层，非视觉）

「先轻后叙事」两段式：

### 段 1 — 先轻：一眼今日（即时可动作）
- 今日 due 数（到期学习项）— 数据已就位（`workbench/summary` 的 `kpi.due_count`）。
- 待裁决数（AI 提议待审）— 数据已就位（`proposals.total`）。
- 这段是「轻」：低阅读量、直接 CTA（去复习 / 去裁决）。**现有 KpiRow + 复习/裁决两缕已覆盖这段数据，新缕不重造**，但叙事节奏上要让段 1 **先于、轻于**叙事段。

### 段 2 — 后叙事：昨夜 AI 替你做了什么（新维度，本 handoff 的主角）
AI 以第一人称口吻交班昨夜劳动，**有内容才叙事，无内容走空夜态**。昨夜可交班的素材类目（均为真实夜间 job 产出）：
- **重标定 / 掌握变化**：夜间重标定 job 更新了某些知识点的掌握估计 → 下钻该知识点详情。（**硬约束见下「mastery 展示约束」：绝不裸数字**。）
- **题库补缺**：为某些知识点补了练习题 → 下钻练习。
- **新提议**：提了新知识边 / 学习项（落进待裁决） → 下钻裁决收件箱。
- **备课**：夜间诱导出辨析题 conjecture（备课台「为你而备」，≤3 条） → 下钻备课台。
- **AI 观察**：昨夜留下的 agent notes。
- **录入改动**：近 24h AI 对笔记的 refine。
- **job 失败**（诚实交代）：昨夜某项没跑成。

叙事缕的功能要求：
1. **第一人称、当日窗**：是「昨夜（自上次到访以来）我替你做了 X」的交班，不是「AI 历史观察长流」。窗口口径见数据前置（缺 `last_visited_at` 锚点，首版可退化为固定「昨夜」窗）。
2. **可下钻、可裁决**：每条交班项要能跳到它的归宿（重标定→知识点详情、补题→练习、提议→裁决、备课→备课台）。交班缕是入口，不是终点。
3. **有限、不堆积**：是「昨夜这一档」的快照，不是 backlog；克制条数，避免变成无限增长的任务列表。
4. **诚实**：昨夜 job 跑挂 / 无产出时**不编造**昨夜活动——这正是反模式 1 当初被迫中性化的原因。

## 空态 / 失败态 / 低置信态（一等公民，显式功能约束）

> 这一章是这条缕的核心，不是边角。交班缕的价值高度依赖**诚实**——空夜与故障处理错了，直接摧毁代理信任。

### 空夜态（headline 状态，必须有自己的 felt 表达 — owner 显式点名）
**定义**：`goal_count > 0`（已过冷启、有档案）**但**昨夜窗内无任何可交班的 AI 产出。两种触发：
- **首日**：有档案但还没有过任何一个夜晚（job 没跑过）。
- **连续无活动**：昨夜 job 跑了但无新产出，或 owner 多日未访、无「自上次以来」增量 = 安静夜。

**功能约束**：
- 段 1（先轻）照常显示今日 due / 待裁决——即便昨夜无 AI 活动，今天该练的还在。
- 段 2（叙事）**不渲染假交班**。空夜态要有**自己的 felt 表达**：诚实地说「昨夜没有需要交班的活动」（首日可略说明 AI 夜间会做什么、何时开始），而非套通用空话或硬塞一条无意义叙事。
- **空夜态必须与「加载中 / 失败」在视觉与语义上可区分**（owner 显式要求）——安静夜是一种**合法日常稳态**，不是错误、不是数据还在路上。三态各自有面貌。
- **绝不落回 ColdStart**（那是 `goal_count===0` 的冷启路径，反模式 4）。

### 低置信 / 失信兜底（昨夜活动「有但低质」时）
- 交班缕里凡涉及 **mastery / 难度的绝对值**，一律带置信区间 / 低置信标记，**绝不给干净数字**（见下硬契约）。
- 涉及 AI 提议 / conjecture 时，措辞要区分「这是 AI 的软提议（待你裁决）」vs「这是已确认的事实」——不能让软提议看起来像既成事实。
- conjecture 交班的**内部校准概率**（confidence / predicted_p 之类）**绝不上线**到卡面，避免 owner「优化那个数字」。

### 故障态
- **digest 读模型不可用 / 报错**：段 2 走标准 error + retry 态（参照既有 `Stateful`，`TodayPage.tsx:148-154`）；段 1（今日 due）独立加载、**不被段 2 故障拖垮**——两段**独立可降级**。
- **部分素材源失败**（如重标定数据有、备课台挂）：交班缕**部分降级**呈现可得部分，缺失部分静默或标注，**绝不整缕崩**。
- **昨夜 job 实际跑挂**（job 自身 error，非读模型挂）：**如实交代「昨夜某项没跑成」**属于诚实交班的一部分，而非假装一切正常——这是与「永远报喜」的关键区别。

## mastery / 难度展示约束（硬契约）

交班缕段 2 会提及「昨夜重标定了某些知识点的掌握」。凡触及 mastery / 难度：
1. **绝对值一律带置信区间 / 低置信标记，绝不给干净数字**——不显示「掌握 78%」裸值；表达「更新了」「方向变化」「可下钻看详情」即可。
2. **来源二态可分**：硬轨真实作答校准 vs 软轨 LLM 先验回吐至少二态可视区分。冷启盲区知识点的「掌握」是先验回吐，不能让它看起来像可信校准结果。
3. 数值精度 / 精确画像归知识点详情页；交班缕只做「指过去」的入口，不在缕里渲染精确数值。

- **目标数据形状**（A1 所需的功能形状，最终 wire 由 YUK-520 定稿）：
  ```jsonc
  {
    "window": { "since": "<上次到访 or 昨日界>", "until": "<now>" },
    "has_overnight_activity": true,          // false → 空夜态（headline 状态）
    "strands": [
      { "kind": "recalibration",    "count": 4, "route": "/knowledge",        "summary": "更新了 4 个知识点的掌握估计" },
      { "kind": "question_supply",  "count": 6, "route": "/practice",         "summary": "为 3 个知识点补了 6 道题" },
      { "kind": "proposal",         "count": 3, "route": "/inbox",            "summary": "3 条新提议待裁决" },
      { "kind": "conjecture",       "count": 2, "route": "/today#prep-desk",  "summary": "备了 2 道辨析题" },
      { "kind": "job_failure",      "count": 1,                               "summary": "frontier_fill 昨夜未跑成" }
    ]
  }
  ```
  > `recalibration` strand 的「掌握更新」**不带绝对数字**（见 mastery 硬契约），只表达「更新了 / 方向」+ 可下钻。

- **窗口口径降级**：系统当前**无 `last_visited_at` 锚点**，「自上次到访以来」窗算不出。可接受降级 = 首版用固定「昨夜（昨日日界）」窗，`last_visited_at` 作后续增强。此降级口径由 YUK-520 / owner 拍板，不影响 claude design 出形态（形态对两种窗口口径通用）。
