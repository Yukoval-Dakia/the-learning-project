## 1. EXECUTIVE SUMMARY

The 8 highest-stakes findings across the whole project, ranked by (reachability today √Ч blast radius on irreplaceable state):

1. **`kc-dedup-attribution-rewrite-gap`** вАФ KC merge archives the loser but never rewrites `question.knowledge_ids`/`mastery_state`/`learning_item`, despite **four** code comments + ADR-0045 asserting it does. 2. **`pfa-plearn-formula`** вАФ owner-fixed ќ≥/ѕБ aren't "pending refit," they're a live bug: at ќ≤=0 (common cold-start case) 3 corrects flips a KC to "mastered" and drops it from the learnable-frontier pool; at ќ≤вЙИ3 an anchored prereq silently starves its whole downstream subtree.
3. **`default-softmax-selection-policy`** вАФ the ќµ-mix floor lets a selected low-weight candidate carry `1/ѕА вЙИ 2500√Ч` weight with zero trim, and at `RECALIBRATION_MIN_LABELS=12` one fluke can set the **first, irreversible** `b_calib` write.

5. **`event-sourcing-fold-sot-flip-status`** вАФ the believed backstop for already-flipped entities (`audit:projection`) is tautological (re-derives via the same reducer the writer used); `learning_item` (FSRS/mastery-adjacent, highest blast radius yet) is next in the SoT-flip queue with no independent oracle in place.

## 2. CROSS-CUTTING FINDINGS

**F1 вАФ "Documented invariant, unenforced code" recurs independently вЙ•5 times.** `kc-dedup` (ADR-0045 + 4 comments claim a rewrite that doesn't exist), `cascade-revert-reversibility-taxonomy` (tests assert a snapshot topology no production writer produces), `event-sourcing-fold-sot-flip-status` (the assumed audit backstop is self-referential), `hub-mesh-derived-from-semantic-contradiction` (nightly prompt gloss contradicts ADR-0010's own direction convention), `adr0026-flag-gated-rollout-opacity` (ADR still describes a mechanism superseded by live YUK-164/YUK-482 code). None of these were caught by review at the time they drifted вАФ worth a lightweight recurring "doc-claims-vs-code" sweep (the existing `/audit-drift` skill is the right home; consider widening its scope to include inline-comment claims, not just planning docs).

**F2 вАФ Mastery's *write* axis is hardened; its *consumption/threshold* layer is where the live bugs cluster.** `three-axis-orthogonality-invariant` and `mastery-write-invariant-cbm-floor` are enforced by static fs-walk tests вАФ genuinely solid. But every live P1 in this sweep sits one layer downstream: `learnable-frontier-gate`'s 0.7 threshold, `pfa-plearn-formula`'s ќ≥/ѕБ feeding that same threshold, and `kg-borrowing-prereq-propagation-sprawl`'s borrow-branch synthesizing `.mastery` outside the guarded write path. The invariant discipline that protects `mastery_state` writes has no analogue protecting what reads/derives from it.

**F3 вАФ Merge/absorb operations across three unrelated entity types share one unfixed shape.** KC merge (`kc-dedup-attribution-rewrite-gap`), memory MERGE (`re-embed-on-merge-cross-cutting-gap`), and (partially) note dedup all leave the survivor's own data correct but the absorbed entity's downstream references/embeddings/attributions incompletely migrated. These should be diagnosed as one reusable "merge completeness" policy, not three independent point-fixes вАФ `embed_backfill`'s existing dirty-bit pattern (YUK-393) is the template to extend, not reinvent.

**F4 вАФ A single-scalar, self-reported-confidence gate in front of an irreversible or attribution-affecting write is a recurring shape, not a one-off.** `memory-reconcile-policy` (0.6 вЖТ permanent MERGE/RETRACT_NEW), `tagging-match-or-propose` (0.55 вЖТ silent zero-event MATCH), `default-softmax-selection-policy` (ќµ-mix floor вЖТ first irreversible `b_calib` write), `induce-self-consistency` (literal-string clustering вЖТ misconception confidence), `appeal-rejudge-single-judge-direct-effect` (one LLM re-adjudicates whether one LLM was wrong, replayable at will). None of these need a heavyweight ensemble fix (that was tried and rejected project-wide, correctly) вАФ but several are missing the *cheap* structural bound (winsorize, corroborate, log, cap) that would catch the fat-tail case without adding a second judge.

**F5 вАФ The adversarial-reconciliation process itself surfaced a meta-pattern worth institutionalizing: initial diagnoses systematically overstated severity by conflating "mechanism exists/flag exists" with "live and risky."** Roughly two-thirds of the deep-dive verdicts above downgraded P0/P1вЖТP2 once actual callers, flag defaults, and gate positions were traced in code. This is the adversarial-reconciliation process working as designed, but it argues for making "trace the actual live caller graph before assigning severity" a default first step in future audits, not just an adversarial afterthought.

**F6 вАФ Cold-start is simultaneously the project's top design priority and its least-tested regime.** `pfa-plearn-formula`'s ќ≤=0 flip, `kg-borrowing`'s borrow-branch, and `tagging-match-or-propose`'s n=6 threshold all live exactly in the zero/sparse-evidence window the whole architecture is built to serve well вАФ and all three are undertested there because there's no live cohort to exercise it yet. This isn't a contradiction to resolve, but it does mean cold-start-path bugs deserve disproportionate attention relative to their current "n=1, no incident yet" framing.

**F7 вАФ Dark-shipped subsystems mostly have documented, dated unlock conditions (KEEP-as-designed) вАФ but a few don't, and those are exactly the ones flagged P1.** `bkt-soft-track-dead-sink`, `ez-diffusion-axis-provenance`, `grid-bayes-shadow-estimator-family` all have clear, named firm-up triggers. `conjecture-engine-dark-loop` and `srt-time-limit-d-quantile-gap` don't вАФ the wire-vs-retire question for the former is already on record in an owner doc and just needs an answer.
